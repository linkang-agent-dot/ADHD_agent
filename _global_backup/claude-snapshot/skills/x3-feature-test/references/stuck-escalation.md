# 卡住即升级 · 叫人工介入（判断逻辑）

> 何时停止"硬试"、转而通知真人接手。源头：2026-06-24 验 X3NEW-592 时，为补一张界面截图反复撞墙——
> 招募 UI 假死 → 杀进程重启 ×3、弹窗 whack-a-mole、桥造不出 UIData……烧了一大把请求才认清"桥结构性做不到、需真人手点"。
> **若早有此规则，撞 2~3 次就该叫人，省下大半折腾。** 本文件把"卡住判据 + 升级动作 + 通知方式"固化下来。

---

## 一、卡住判据（命中**任一**即判定"卡住"，立即进入升级，别再硬试）

机械化、可自查（直接核对话/工具调用记录），不靠"感觉"：

1. **同一 blocker 复发**：同一个报错 / 同一个失败现象，连续命中 **≥ 3 次**仍无实质进展（换了写法也不通）。
2. **单子目标空转**：为**同一个子目标**（如"打开某 UI 并截图""让某弹窗消失""让某值变成 X"）累计发出 **≥ 10~12 次**桥/工具调用，仍没把它推进一格。
3. **环境崩溃复发**：运行时环境（Unity Editor / 本地服 / 真机）**假死或崩溃 ≥ 2 次**，每次都要杀进程/重启才能继续。
4. **硬限制已确认**：已经搞清是**工具/环境能力的结构性限制**（不是写法问题），例如"桥的反射无法构造自定义对象""某能力只有真人在客户端点才能触发"。
5.（可选·墙钟）同一子步骤 **> 15~20 分钟**无进展。

> **不滥用**：只对"**无进展的重试**"计数；正常分步推进、每步都在前进，不算卡住。单点查值/改字段这类小任务别动辄升级。

---

## 二、升级动作（触发后按序做，先止损再叫人）

1. **立刻停**对该子目标的重试——继续试只会烧请求 + 可能反复搞崩环境。
2. **拆分交付**：把**不依赖该卡点**的结论/产物**先交付**（别让一个卡点拖垮整个任务）。例：核心验证已用代码证据完成 → 先出 🟢 结论，截图作为"待补"单独挂起。
3. **汇总升级单**（给人看的，四要素，别贴流水账）：
   - **卡在哪**：一句话 blocker（现象 + 已确认的根因/限制）。
   - **已尝试**：列 2~4 条试过什么、为什么不通。
   - **需人工做什么**：**具体到一步可执行的动作**（如"在游戏里手点进招募界面停住"，而不是"帮忙看看"）。
   - **影响面**：不处理会怎样 / 已能交付的部分是什么。
4. **通知真人**（见下，二选一）。
5. 通知后**挂起该子目标、转入其他可推进的工作**或收尾，等人工反馈再续——不空等、不继续硬试。

## 三、通知方式（按"有没有 dws"自动路由：有钉钉 MCP 走钉钉，没有就网页弹窗兜底）

**判定一次即可**：`dws` 是钉钉的 MCP（钉钉全产品能力）。先看本会话/本机有没有接 `dws` MCP——

### A. 有 dws（钉钉 MCP）→ 走钉钉（首选）
- 通过 **`dws` MCP** 把"升级单"发给指定人/群（值班研发、模块负责人），可异步、可留痕、能找不在场的人。
- 前提：本机已接 `dws` 且配好钉钉鉴权（鉴权只存本机，不写进 skill/仓库）。

### B. 没有 dws → 网页弹窗兜底（零依赖，永远可用）
- 没接钉钉 MCP 的使用者，**直接走网页弹窗**：把"升级单"四要素写成一个 HTML、用浏览器弹出来，引导本机用户介入。
- 实现：生成含四要素的 HTML，`Start-Process <html>`（Win）/ `webbrowser.open` 打开即可。零依赖、不需任何鉴权。

> 路由：**先探 dws 是否可用 → 可用走 A 钉钉；不可用直接 B 网页弹窗。** 不要因为没钉钉就卡住不通知——网页弹窗是永远的兜底。

---

## 四、一句话规则（贴进自查清单）

> **同一子目标撞墙 ≥3 次 / 空转 ≥10 次调用 / 环境崩 ≥2 次 / 已确认硬限制 → 立即停手，先交付不受影响的部分，再通知人工（有 dws 钉钉 MCP 就发钉钉，没有就网页弹窗兜底），把"卡在哪+已试什么+需人工做哪一步"发出去，挂起等人工，别继续烧请求。**
