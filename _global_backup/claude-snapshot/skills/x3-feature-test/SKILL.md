# X3 功能端到端测试编排

把一个 X3 功能从「代码改动」验证到「运行时真生效」。核心理念：**尽量让真实业务链路自己跑**（调游戏自己的方法、发真实请求、用 GM 构造前置状态），而不是直接塞数据 / 强开 UI——后者只能证明"渲染没崩"，证明不了"功能链路通"。

底层桥机制（client.py / eval / invoke / telnet / 截图 / 日志）**全部走 `DebugUtils` skill**，本 skill 只管编排流程 + 指路知识库。

## 知识库索引（按需读，别一次性全load）

| 文件 | 什么时候读 |
|------|-----------|
| `references/brief-templates.md` | **第 0 步起草 brief 时**：挑功能类别模板（已预填该类触发条件/验证点/坑）|
| `references/recipes.md` | **测试前先翻**：这个功能或同类是否已有验证过的配方可直接套 |
| `references/client-runtime.md` | 要读客户端运行时状态（玩家/Meta/配置/UI）、写 eval 表达式时 |
| `references/server-arch.md` | 要搞清功能在服务端"由什么触发、有哪些门槛"、连 telnet 跑服务端 C# 时 |
| `references/gm-commands.md` | 要用 GM 构造前置状态（建号/升级/加资源/开活动）、玩家域 GM 不生效时 |

## Brief 生命周期（流程骨架）
测试以一份**轻量 brief** 贯穿始终，对齐项目「可验证目标前置」——不是测完补文档，是**先出 brief 再测、边测边补、测完即配方**：
- **第 0 步起草**：从 `brief-templates.md` 挑类别模板，填「改了什么 + 成功断言」；触发条件先留空。
- **第 4 步后补全**：摸清触发链路 → 回填真实触发条件 + 复现步骤。
- **第 8 步落地**：完成态 brief = 一条配方，进 `recipes.md`（按类别分区）。
- 入参 brief（人给的 `测试brief.md` 之类）可选，有则省去推意图；没有则从 `git diff` 推 + 问用户"生效长什么样"。

## 通用流程（8 步，按规模裁剪）

### 第 0 步 · 开桥 + 起草 brief
- `DebugUtils` bootstrap → `client.py ping`，确认 Editor 开着、HTTP 桥通（21891）。读改动 `git diff`。
- **先翻 `recipes.md`**：同类功能有现成配方就直接套，省大量试错。
- **起草 brief**（从 `brief-templates.md` 挑类别模板）：填「改了什么 + 成功断言」。**先定义"成功长什么样"**——可观测断言（哪个运行时值变成什么 / 哪条日志 / 哪个 UI 显示 / 哪个 flag 置位），弱目标（"让它 work"）判断不了是否完成。触发条件这栏第 4 步后再补。
- 确认是否需要登录态玩家（读 `G.PlayerID`）；ping 通只代表 Editor 开着。

### 第 1 步 · 静态核实（轻量改动到此可能就够）
- 编译/符号/重载/事件签名对得上吗（grep + 读源码）。运行时 `recompile` 可把静态结论升级为真编译（注意会退 Play、丢登录态）。
- 改动逻辑放进完整链路推演：正常路径 + 边界 + "数据变了但没人刷新/通知"的隐藏路径。

### 第 2 步 · 运行时取证（先看现有，别急着改）
- 读当前运行时状态对照预期：配置值、玩家字段、Meta 数据、UI 状态。入口见 `client-runtime.md`。
- 看**现有日志**（Editor Console / 本地服 log），很多答案已经在里面，别一上来就加日志。
- 状态与预期矛盾 = 找到了第一个分叉点，顺着往下查。

### 第 3 步 · 配置生效核查（功能涉及配置时必做）
- ⚠️ **改了 tsv ≠ 生效**：提交进仓库的 `ProtoGen/*.bytes` 可能是 stale 的（导表没重跑）。客户端读 bytes 不读 tsv。
- 核查：运行时配置值 vs 期望值。不一致 → 走 `x3-config-export`（本地 `ExportTable.py` → 覆盖 client ProtoGen → 客户端 reimport+重进 Play；本地服 `ReloadGameServer` 热更）。详见 `recipes.md` 的配置生效闭环。

### 第 4 步 · 摸触发链路（最关键，决定怎么测）
- **读服务端代码**搞清：这个功能由**什么事件 / 什么条件**触发？有哪些**解锁门槛**？
- X3 常见模式：充值→`PayOrder`、升级→`PlayerLevelChanged`、功能解锁→`FunctionStateChange` 等事件 → `HandleNewPlayerActivityInfos` / 各 Meta handler，再过 `Check*IsUnlock`（等级区间 / 时间窗 / 前置资源 / 功能解锁）。详见 `server-arch.md`。
- 没摸清触发条件就构造测试 = 白忙（今天就因为漏看"等级≥5"门槛，新号充值跨门槛了活动还是没开）。

### 第 5 步 · 构造前置状态
- 用 GM / 新号 / 配置把玩家拉到**满足全部触发条件**的状态：等级、资源、时间窗（注册相对活动要新号）、功能解锁、已有/未有某活动。
- 玩家域 GM（升级/加资源/跳引导）走客户端 `SendGmCmd`；server-scope GM（导表/部署）走 telnet。机制 + 命令见 `gm-commands.md`。
- 一次性事件触发型功能（如"首充跨门槛"）：老号常无法复现（状态不可逆），**优先新号**。

### 第 6 步 · 走真实入口触发
- 尽量调游戏自己的方法让真实链路跑：发请求 / 购买 / 战斗 / GM 模拟真实行为。Editor 下很多 SDK 走 `#else` 假实现（如购买直发 `BuyGiftReq`），可用 eval 调客户端方法驱动。
- **能走真实入口就别强开 UI**：`WndMgr.Show`/直塞数据只能证明渲染，证明不了触发逻辑（且场景切换型 UI 直 Show 会卡死，见 DebugUtils）。强开仅用于"已确认触发链路、单纯想截一张 UI"。

### 第 7 步 · 验证 + 取证
- **区分两层**：① 代码路径跑通（运行时值 / flag / dedup key / 日志断言）② UI 真显示（截图 `ScreenCapture.CaptureScreenshot`）。两者都给才算"端到端"。
- 注意队列式 UI（`WndMgr.Push`）：推窗 ≠ 立即显示，可能排在其他弹窗后或等主场景。flag 置位证明"推了"，截图证明"显示了"。

### 第 8 步 · 清理 + 沉淀
- 透明交代改了哪些环境状态（测试号、配置、分支、GM 改动），让用户决定是否回滚。
- **新跑通的功能 → 把配方回写 `references/recipes.md`**（一个功能一节：触发条件 + 构造步骤 + 验证点）。这是本 skill 增值的关键——越用越省。
- 通用新规律 / 踩坑 → 同步回写对应 `references/*.md` 或项目 memory。

## 报告分档（对齐项目 CLAUDE.md）
- 单点验证：一句话说清验了什么、结论、证据。
- 完整端到端：给"改了什么 / 触发条件 / 验证证据表 / 待确认 / 复现配方"。触及 CSShared / 协议 / 配置结构的，点明回归面。

## 边界
- 适用：X3 客户端+服务端联动功能的运行时验证。**不适用** X2、不适用纯静态代码审查（那走 review）、不替代单元测试。
- 本 skill 不重复 DebugUtils 的桥用法、不重复 x3-config-export 的导表细节、不重复 bug-fix 的修复流程——它们是被本流程"调用"的工具。
