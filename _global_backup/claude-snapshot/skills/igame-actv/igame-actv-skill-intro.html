iGame Activity Deployment Skill

节日活动批量部署自动化工具 | 版本 1.0 | 2026-04

解决的问题

节日活动部署是高频、高风险的运营操作，传统手工方式存在以下痛点：

⏱️ 效率低下

每次节日活动需部署 40-60 条活动，手工逐条填写 iGame 表单耗时 2-3 小时，且容易疲劳出错。

🎯 参数复杂

涉及 schema 分服（3/4/5/6）、灰度服处理、跨服分组、多期活动、准备期/展示期等多维度参数组合。

⚠️ 容错率低

服务器漏配、时间错误、跨服类型填错等问题一旦提交，需要走取消流程重新提交，影响上线进度。

📊 信息分散

排期在 Google Sheet 甘特图，服务器分组在 yych-tools，活动配置在 iGame，需要反复切换核对。

核心能力

📅 甘特图解析

自动读取 Google Sheet 甘特图，识别活动名称、configId、排期（含多期）、跨服类型、负责人。

node deploy-from-gantt.js --sheet <id> --tab "页签名" --list

🖥️ 智能分服

根据活动名称自动识别 schema（S3/S4/S5/S6/S3-5/S4-6），匹配对应服务器列表和跨服分组。

Schema 6: 81服 Schema 3-5: 54服 灰度: 12服

🔄 批量部署

一键提交多条活动到 iGame API，支持 dry-run 预览、冲突检测、分批提交（超过127组自动拆分）。

node deploy-from-gantt.js --sheet <id> --tab "页签名" --deploy 1,3,5-8

✅ 预览确认

部署前生成 HTML 预览表格，展示每条活动的完整参数，人工确认后再提交，避免盲目部署。

🛡️

服务器数量校验（高风险防护）

提交前强制校验服务器总数与预览一致。历史教训：CSV 分组不含灰度服导致漏配 12 服。

// ✅ 正确：合并灰度分组 const servers = [...groups["Schema 6 分组"], groups["灰度分组"][0]]; // 93服
⏰

UTC 时间强制校验（高风险防护）

所有时间戳必须使用 Date.UTC()，禁止使用北京时间。历史教训：时间错误导致活动提前/延后开启。

// ✅ 正确：4-9 ~ 4-29 的写法 startTime: Date.UTC(2026, 3, 9, 0, 0, 0), // 4月9日 00:00 UTC endTime: Date.UTC(2026, 3, 30, 0, 0, 0), // 4月30日 00:00 UTC = 4月29日结束
🔍

重复提交检测（中风险防护）

迭代历程

从 2026 年复活节活动部署实战中持续优化，每次错误都转化为防护机制。

Phase 1 · 基础能力
甘特图解析 + iGame API 对接
问题：手工从甘特图抄写活动参数到 iGame，效率低且易出错
方案：通过 gws CLI 读取 Google Sheet，解析颜色块识别排期，调用 igame-query.js 提交
成果：deploy-from-gantt.js 实现 --list 列出活动 + --deploy 批量提交
Phase 2 · Schema 分服
自动识别 Schema + 服务器匹配
问题：不同活动需要部署到不同 schema 的服务器，手工匹配容易遗漏
方案：正则匹配活动名中的 S3/S4/S5/S6 标识，从 server-config.json 获取对应服务器列表
成果：detectSchema() + getSchemaServers() 实现自动分服
Phase 3 · 跨服分组
支持单服/跨服-全服/跨服-分组
问题：跨服活动需要按预定义分组提交，分组数据在 yych-tools
方案：fetch-server-config.js 拉取分组数据，根据 crossType 字段选择分组模式
成果：支持 acrossServer=0/1 + acrossServerRank=0/1 四种组合
Phase 4 · 错误修复
🔴 灰度服漏配事故
问题：8 条活动漏配 12 个灰度服，用户反馈"特效累充和Bingo累充为啥灰度服的活动没有"
根因：CSV 的 "Schema 6 分组" 不含灰度服（81服），直接使用未合并灰度分组
方案：非灰度活动必须合并灰度分组：[...s6Groups, grayGroup]
成果：补提交灰度服活动（ID 8107-8113），更新 SKILL.md 文档
Phase 4 · 错误修复
🔴 UTC 时间错误
问题：活动时间提前 8 小时开启
根因：使用 new Date() 而非 Date.UTC()，本地时区被带入
方案：强制使用 Date.UTC(year, month-1, day, 0, 0, 0)
成果：dateToTimestamp() 函数统一处理
Phase 4 · 错误修复
🔴 新7日活动只有1天
问题：新7日活动每期应该 7 天，实际只提交了 1 天
根因：甘特图单格表示一期，未正确扩展日期范围
方案：expandSegments() 函数根据前后期推算每期时长
成果：多期活动日期自动扩展
Phase 5 · 防护机制
DEPLOY_CHECKLIST.md 强制检查
问题：错误反复发生，缺乏系统性防护
方案：创建独立 Checklist 文件，部署前必须逐项检查
成果：8 项检查点覆盖服务器数量、灰度处理、时间参数、跨服类型、多期活动
Phase 5 · 防护机制
HTML 预览系统
问题：命令行输出不直观，难以全面核对
方案：生成 HTML 表格 + 甘特图可视化预览
成果：人工确认后再提交，避免盲目部署
