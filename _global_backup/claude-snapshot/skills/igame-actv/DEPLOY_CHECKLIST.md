# 活动部署检查清单

**每次提交活动前必须逐项检查，不可跳过。**

---

## 1. 预览核对

在提交任何活动之前，必须生成预览（HTML/表格）并与用户确认。

- [ ] 预览文件已生成并展示给用户
- [ ] 用户已确认预览内容无误

---

## 2. 服务器数量校验

**提交时打印的服务器总数必须与预览一致。**

| 检查项 | 预览值 | 实际值 | 是否一致 |
|--------|--------|--------|----------|
| 服务器总数 | ___ 服 | ___ 服 | ☐ |
| 分组数（跨服分组时） | ___ 组 | ___ 组 | ☐ |

**如果数量不符，立即停止，排查原因后再继续。**

## 3. 灰度服处理

### CSV 分组数据结构（重要！）

| 分组名 | 服务器数 | 是否含灰度服 |
|--------|---------|-------------|
| Schema 6 分组 | 81服/7组 | ❌ 不含 |
| Schema 3-5 分组 | 54服/5组 | ❌ 不含 |
| 灰度分组 | 12服/1组 | ✅ 灰度服本身 |
| 全服一组 | 147服 | ✅ 含灰度 |

### 非灰度活动（需要包含灰度服）

- [ ] S6 分组活动：Schema 6 分组 + 灰度分组 = **93服/8组**
- [ ] S6+S3-5 分组活动：Schema 6 分组 + 灰度分组 + Schema 3-5 分组 = **147服/13组**
- [ ] S5+S6 分组活动：Schema 5 分组 + Schema 6 分组 + 灰度分组 = **107服/9组**

```javascript
// ✅ 正确写法
const servers = [...groups["Schema 6 分组"], groups["灰度分组"][0]];
```

### 灰度活动（需要排除灰度服）

- [ ] 从服务器列表中过滤灰度服

```javascript
// ✅ 正确写法
const serversNoGray = servers.filter(s => !grayServers.includes(s));
```

---

## 4. 时间参数校验

- [ ] 使用 UTC 时间，不是北京时间
- [ ] startTime 和 endTime 与预览排期一致
- [ ] endTime 是结束日期的**次日** 00:00 UTC

```javascript
// 4-9 ~ 4-29 的正确写法
startTime: Date.UTC(2026, 3, 9, 0, 0, 0),   // 4月9日 00:00 UTC
endTime: Date.UTC(2026, 3, 30, 0, 0, 0),    // 4月30日 00:00 UTC = 4月29日结束
```

---

## 5. 跨服类型校验

- [ ] 核对甘特图的"跨服"列
- [ ] acrossServer 参数正确（0=单服, 1=跨服）
- [ ] acrossServerRank 参数正确（0=无跨服排名, 1=有跨服排名）

| 跨服类型 | acrossServer | acrossServerRank | servers 格式 |
|---------|--------------|------------------|--------------|
| 单服 | 0 | 0 | `[allServers]` |
| 跨服-全服 | 1 | 0 | `[allServers]` |
| 跨服-分组 | 1 | 1 | `[group1, group2, ...]` |

---

## 6. 多期活动校验

- [ ] 检查甘特图是否有 `|` 分隔的多期标记
- [ ] 每期单独提交，日期各自独立
- [ ] 多期活动的期数与预览一致

---

## 7. 特殊参数校验

- [ ] previewTime（准备期，分钟）：挖矿等活动需要 1440（24小时）
- [ ] endShowTime（展示期，分钟）：节日押镖需要 30（30分钟）

---

## 8. 提交后验证

- [ ] 检查返回 `success: true`
- [ ] 记录返回的活动 id
- [ ] 如需下线活动，按当前状态选接口：
  - **部署申请状态**（未上线，审核中/已审未到时间）→ `activity/operation/recall`（撤回，单 id 一次一个，退回后重新审批）
  - **上线中状态**（活动已经在跑）→ `activity/operation/cancel`（批量 ids+reason，作废后需重新部署）
  - 都不要用 `del` 接口（对待审核活动无效）
