# 竞品记录录入

将用户提供的礼包/活动信息，按模版格式写入 Obsidian 仓库中对应游戏的独立笔记文件。

## Obsidian 仓库

- **路径：** `C:\ADHD_agent`

## 已注册游戏

| 游戏 | 文件夹 | 缩写 |
|------|--------|------|
| Last Asylum: Plague | `竞品调研/Last Asylum Plague/` | LAP |
| Kingshot | `竞品调研/Kingshot/` | KS |

新增游戏时：在 `竞品调研/` 下建文件夹，创建以下目录结构，并在上表注册：
```
竞品调研/{游戏名}/
├── 活动/              ← 每个活动一个独立 .md 笔记
├── 礼包/              ← 每个礼包一个独立 .md 笔记
├── 日报/              ← 监控日报
├── attachments/       ← 截图和抽帧图片
├── 活动记录.md         ← 活动索引页
├── 礼包记录.md         ← 礼包索引页
└── 竞品监测链接大全.md  ← 数据源链接汇总
```

## 录入流程

### 1. 判断类型
用户输入中包含价格、物品内容、限购 → **礼包**
用户输入中包含活动规则、起止时间、奖励层级 → **活动**
不确定 → 问用户

### 2. 确认游戏
从用户消息中识别游戏名，匹配上表。未匹配到 → 问用户。

### 3. 读取模版
- 礼包 → 读 `references/礼包模版.md`
- 活动 → 读 `references/活动模版.md`

### 4. 解析信息
从用户的口述/语音文字/截图/录屏视频中提取字段，填入模版。
- 截图：用 image 工具识别内容，提取物品名称、数量、价格等
- 录屏视频：用 youtube-analyzer skill 的本地文件模式分析视频内容（`python3 scripts/analyze_youtube.py "<视频路径>" "<提取prompt>" --model gemini-2.5-pro`），prompt 要求逐画面识别所有UI文字、物品名称、数量、价格、活动规则等，**并要求输出每个画面的精确秒数和页面名称**

### 录屏抽帧规范（重要）

**核心原则：用 Gemini 返回的时间戳驱动抽帧，绝对不要自己猜帧号。**

1. **Gemini 分析时要求输出时间戳**，格式如：
   ```
   0:00 - 寻医之道/累计登录
   0:09 - 寻医之道/教堂升级
   0:34 - 寻医通行证/主页
   1:02 - 寻医启示
   ```

2. **按秒数抽帧，用 `-ss` 参数**，不要用帧号（`eq(n,...)`）：
   ```bash
   ffmpeg -y -ss 34 -i video.mp4 -frames:v 1 -q:v 2 output.jpg
   ```

3. **文件命名带秒数和活动标识**，方便对应：
   ```
   LAP_2026-03-23_bp_34s.jpg      # bp=battle pass
   LAP_2026-03-23_seekdoc_09s.jpg  # seekdoc=寻医之道
   ```

4. **按活动分组引用截图**，不要按文件序号盲排。每个活动的截图只引用属于该活动的帧。

#### 踩过的坑
- ❌ 用固定帧号间隔（`eq(n,0)+eq(n,120)+...`）→ 帧全挤在前半段，后面的活动页面抽不到
- ❌ 按文件序号猜活动归属 → 截图和活动对不上
- ✅ 用 Gemini 给的秒数 + `-ss` 精确抽帧 → 一一对应
- 口述/语音：直接从文字中提取
- 缺失字段标注 `待补充`，不要编造
- **截图/视频识别不清的内容：** 标注 `看不清`，绝对不要猜测或编造。宁可整个表格都标"看不清"，也不要填入不确定的内容。录入完成后提醒用户补充看不清的部分

### 5. 写入独立笔记文件

**每个活动/礼包创建一个独立的 .md 文件**，不再追加到单个大文件中。

- 活动 → 写入 `{游戏文件夹}/活动/{活动名称}.md`
- 礼包 → 写入 `{游戏文件夹}/礼包/{礼包名称}.md`

文件必须包含完整的 frontmatter（tags + aliases + game + 记录日期等），参见模版。

**关键要求：**
- frontmatter 的 `tags` 中必须包含活动类型/商业化定位等标签（不只是正文里写标签文字）
- frontmatter 的 `aliases` 必须包含活动/礼包的常用简称，方便 `[[]]` 双向链接引用
- 正文中关联的活动/礼包用 `[[活动名]]` / `[[礼包名]]` 链接到独立笔记（不再用 `[[#标题]]` 锚点）

### 6. 更新索引页

写入独立笔记后，还需要更新对应的索引页：

- 活动 → 在 `{游戏文件夹}/活动记录.md` 的表格中追加一行
- 礼包 → 在 `{游戏文件夹}/礼包记录.md` 的表格中追加一行

索引页格式：

活动索引行：`| [[活动名称]] | 活动类型 | 记录日期 | 状态 |`
礼包索引行：`| [[礼包名称]] | 礼包类型 | 价格 | 记录日期 | 商业化定位 |`

### 7. 确认
告知用户已录入，简要列出录入的关键信息。如有缺失字段，提醒补充。

## 截图/图片存储

收到用户发来的截图时，必须第一时间将图片复制到 Obsidian 仓库的附件目录，用相对路径引用，确保 Obsidian 能直接显示。

### 流程
1. 图片到达后，复制到对应游戏文件夹下的 `attachments/` 目录
   - 路径示例：`C:\ADHD_agent\竞品调研\Last Asylum Plague\attachments\`
   - 目录不存在时自动创建
2. 文件命名规范：`{游戏缩写}_{YYYYMMDD}_{活动或礼包名称}_{序号}.jpg`
   - 活动截图示例：`LAP_20260323_寻医之道_01.jpg`、`LAP_20260323_寻医通行证_03.jpg`
   - 礼包截图示例：`LAP_20260323_新手礼包_01.jpg`
   - 用中文活动/礼包名直接命名，比缩写直观，Obsidian 完全支持
   - 日期必须带上，同一个活动可能跨版本更新
3. 独立笔记中用 Obsidian 格式引用：`![[attachments/LAP_20260323_寻医之道_01.jpg]]`
4. 按活动/礼包分组引用截图，不要按文件序号盲排

### 注意
- 钉钉发来的图片是临时文件，不及时复制会丢失
- 视频抽帧的图片在 `/tmp/` 下，也需要及时复制
- 不要引用 `/tmp/` 或其他外部绝对路径，Obsidian 无法显示

## 标签体系

录入时必须按标签规范打标签。标签同时出现在两个位置：
1. **frontmatter 的 `tags` 字段** — 让 Obsidian 的标签面板和搜索能识别
2. **正文的对应字段** — 人类可读

核心要求：
- **活动记录**必须打：活动类型标签 + 商业化定位标签
- **礼包记录**必须打：商业化定位标签 + 核心付费内容标签 + 目标人群
- 关联的活动/礼包之间用 `[[活动名]]` / `[[礼包名]]` 双向链接（链接到独立笔记文件）

录入时如果能判断出设计目的、受众人群、复用性，就顺手填上。判断不了的标 `待补充`，留给 analyzer 后续分析。

## 注意事项
- 只记录，不分析。深度分析是 competitor-analyzer 的事
- 物品名称保持游戏内原文（英文游戏用英文）
- 价格统一标注币种（$USD / ¥CNY）
- 时间统一用 YYYY-MM-DD 格式，活动精确到 HH:MM
- 一次可以录入多个礼包或活动，每个都创建独立笔记文件
