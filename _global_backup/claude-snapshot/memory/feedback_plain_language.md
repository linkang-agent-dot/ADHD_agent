写计划、笔记、规划类文档时，避免用"schema / 外键链 / 转义 / 连通性 / 反向验证"这类术语和 "V1-A/V2-B" 类代号**独立表达含义**。代号可以作为锚点存在（方便索引），但解释要用人话。

**坏例子**：
> V1-A · 变更计划 JSON schema 设计（字段：project/tables/rows_to_add/lc_keys）

**好例子**：
> **V1-A　先定数据格式**
> 搞清楚一份"改动清单"长什么样 — 要改哪些表、哪些行、ID 怎么对照。
> 输出一份 JSON 模板。

**Why:** 用户明确反馈"有些内容看不懂，有点不讲人话"。术语密集的文档在碎片化时间回来看时认知负担很重，会放大"启动阻力"，违背 ADHD 友好的初衷。

**How to apply:**
1. **标题用功能动词**：不写"schema 设计"写"先定数据格式"
2. **每个任务一句人话解释要干嘛**：不是罗列产出物
3. **术语出现时配翻译**：如必须写"dryRun"，旁边注"（空跑预览）"
4. **代号 + 短标题并列**：`V2-B　特殊字符测试 ★` 而不是只有 `V2-B`
5. **风险描述要具体**：不写"外键链风险"，写"几张表连着写，写到一半某张失败了，前面写进去的要清掉"
6. **数据表 → MD 文档时，列名要重命名为大白话**，不要直接抄源表表头。
   - 真实案例：X2 养成线深度手册抄了源表「坑深/单英雄」做列名，跟 §1.5「总坑深」撞名，含义其实是"全程只买此礼包养满单英雄需付费"——用户当场看不懂。改为「养满单英雄需付费」后才清楚。
   - 原因：源表表头通常是数值/策划自用简写，放到运营/制作人读的手册里会引起歧义。
7. 本条规则**只适用于人读的规划/笔记类**文档；代码注释、schema 定义、API 文档保持术语精确
