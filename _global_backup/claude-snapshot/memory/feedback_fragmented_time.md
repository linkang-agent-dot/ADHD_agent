用户的时间是高度碎片化的，每天都有多个任务在并行跑（CS 工单 + 开发任务 + BI 分析 + 外部打断）。不能按"连续 4 小时专注做一件事"来设计计划。

**Why:** 用户明确说过"我的时间是特别碎片的，每天肯定都是多个任务在跑"。按连续专注时间做计划会失真，一旦中途被打断，整个 D1/D2/D3 的串行链条就会塌方。

**How to apply:**
1. **任务拆分粒度要小** — 单个子任务控制在 30-60 分钟可交付的体量，不做"D1 上午"这种大块切法。改成"任务 A1 / A2 / A3"这类可独立推进的颗粒
2. **避免串行依赖链** — 不要做 A→B→C→D 这种必须按顺序才能跑的计划，中断一次就全停。尽量让任务之间弱耦合，单独捡起任一个都能推进
3. **状态要外化** — 每个任务的当前进度、下一步、blocker 写在 TaskList 或 Markdown 里，碎片时间回来能 5 分钟内续上
4. **同时跑多个任务是常态** — 规划时默认"并行 3-4 个任务"，不是"专注完成 1 个再做下一个"
5. **给每个碎片时间段提供"可抓取任务清单"** — 15 分钟能做什么、30 分钟能做什么、1 小时能做什么，分层备好
