**每次往 GSheet 配置表写入新行前，必须完成以下检查，不能跳过。**

**Why:** 占星节（2026-04）和拓荒节（2026-04）配置先后暴露：INSERT_ROWS 插到表头、误写线上 tab、只写了1行其余丢失、末尾空列被截断等问题。

**How to apply:**

## 写入前（必做）

1. **确认目标页签名**：QA tab 和线上 tab 名字不同（如 `activity_calendar_QA` vs `activity_calendar`），写之前显式打印 tab 名，确认是 QA
2. **读表头行**：确认列数、列名、frozenRowCount（表头占几行）
3. **读参考行并逐列全比**：不能只看"明显的几列"，必须逐列比对。占星节集卡册 1108 漏了 FXDisplayKey 就是因为只比了前4个 DK 字段
4. **确认 SheetID 对应项目**：X2/P2 同编号表（如 1111）SheetID 不同，写之前先核对。占星节曾误写进 P2 的 1111
5. **确认写入位置**：找到最后一条数据行的行号，新行应在其后。注意：新行应该插到同类 ID 段之后，不是无脑追加到表末尾（占星节 1111 的 11114017-19 曾被追加到 row 2188 而非 row 644 之后）

## 写入方式（不限方法，但写后必验）

用什么方式写都行（insertDimension+update、append、直接 update），关键是**写入后的验证必须过**。不同方式有不同风险：

| 方式 | 风险 | 写后重点验证 |
|------|------|-------------|
| insertDimension + update | 行号算错 → 覆盖已有行 | 前后行 ID 连续性 |
| append INSERT_ROWS | 可能插到表头后而非末尾 | 行位置是否在数据区末尾 |
| append OVERWRITE | 可能覆盖已有数据 | 旧行是否还在 |
| 直接 values.update | range 算错 → 写串位置 | 行列位置 |

**原则：写入方式可以选，但写后验证不能省。**

**额外风险：**
- **A 列空值导致列偏移**：X2 部分表 A 列可能为空，append 会把数据从 B 列开始写，整行右移一列。关键写入用明确 range 的 values.update 更安全
- **中文页签名编码**：PowerShell 传中文 tab 名可能乱码，脚本建议用 Python + Unicode escape
- **2135 空行断裂**：2135 中间有空行再接旧数据时，fwcli 在空 ID 行会解析失败。新增行必须保证 B 列 ID 连续，不能用 `len(values)+1` 盲定位
- **2013 列数到 AE**：A_INT_country_use_type 在 AE 列，写/查到 AD 会漏最后一列，必须显式补 0

## 写入后（必做）

1. **验证行数**：预期写 N 行，实际读回来确认 N 行都在（拓荒节2116曾只写了1行，其余9行丢失）
2. **验证行顺序**：读写入行前后各3行的 ID，确认 IntId 递增、位置正确
3. **验证列完整性**：读回写入行，确认列数 = 表头列数，中间无空洞、末尾空列有占位
4. **抽查 JSON 字段**：至少抽1行核对关键 JSON 字段内容
5. **搜索旧 ID 残留**：如果是换皮/替换场景，在相关表全表搜索旧 ID（如旧 recharge_actv），确认无残留指向旧活动
