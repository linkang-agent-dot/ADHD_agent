# 策划案设计质量验收清单（type=design-merit）

> 跟 `design-doc`（查"全 / 对 / 一致"）**互补**：design-doc 查文档完不完整、正不正确；design-merit 查**设计本身好不好**。
> **核心原则**：好不好 = 对它**自己声明的目的**实现到不到位、对它**自己要解决的问题**解决得彻不彻底。**不套万能清单，顺实际目的长。** 质量在概念层判不动，靠**把颗粒度拆细到可查叶子**。
> 这是**按需触发的"定稿前设计压测"**，不挂 Stop hook 自动阻断（设计赌注不该被机器自动拦）；唯一能 block 的是 §1 客观项。

## 0 · 工作姿态（checker 先读）
- **默认手段不对路 + 执行不到位**，你的任务是证伪它，不是夸它。
- **不预设分类框**：读案子声明的**实际目的**，顺着它长拆解树。别拿一张固定清单去套。
- 判断永远是同一个递归动作：**这层的目的是啥 → 它的表现/数值到不到位 → 够不够达成这层目的**，每层重复。
- **不输出"这设计好/坏"的总评**（判不动，输出了也是噪音）；输出**一棵拆解树**，每片叶子是"有没有 / 够不够" + 证据。

## 1 · 硬 gate（客观·可 block —— 唯一能机判 pass/fail 的部分）
缺任一 → **fail**（没有目的，"解决得好不好"无从判起）：
| rule | level | 怎么查 |
|---|---|---|
| `目的已声明` | block | 案子里有没有显式写「目的」（想达成什么）。每个活动模块也应能追到它服务的目的。缺=fail，列出哪些模块挂不上目的 |
| `要解决的问题已声明` | block | 有没有写「针对什么现状/痛点」。缺=fail |
| `成功有可衡量指标` | block | 有没有「成功长什么样」+ 指标；且口径对（节日 ARPU=流水/**当日总付费人数**，不是人次 ARPPU 自欺——见 `feedback_x3_festival_arpu_denominator`）。缺/口径错=fail |
| `批判已逐条回应` | block | 本 pass 抛出的每条批判，定稿文档须带一行作者回应（采纳+怎么改 / 不采纳+为什么）。有未回应的=fail。**注意：这里 block 的是"有没有回应"，不是"设计好不好"** |

## 2 · 递归拆解法（主体 —— 不预设分类，顺目的长）
**骨架：目的 → 模块 → 表现 & 数值**，每一层自带「局部目的」：
- **模块** = 实现目的的手段。
- **表现**【自带表现目的，如"让玩家在这模块里感知到爽/期待/紧迫/稀有"】→ 拆到 动画 / 特效 / 反馈 / 音效 / 节奏 → 答"有没有·够不够达成那个感知目的"。
- **数值**【自带数值目的，如"撑付费深度 / 控进度节奏 / 立价值锚"】→ 拆到 数值 / 曲线 / 档位 / 概率 → 答"有没有·够不够达成那个数值目的"。

**触底规则**：拆到叶子能**指向一段实际表现 / 一个实际数值**、可直接答"有没有·够不够"就停。再拆不改变结论 = 停。
**颗粒度预算**：细颗粒**只花在【主目的】的模块上**；辅目的浅拆；无关目的只做 §3 副作用检查（一个案子不可能解决全部问题，别越界乱判）。
**落地基底可扩展**：默认触底到 **表现 & 数值**；但允许某些目的触底到 **节奏排期 / 文案情感 / 规则结构**（留存→节奏曲线、情感→外显文案）。**别硬套成只有两类。**

> 拆法范例见 `design-merit-qbank.md`（自下而上攒的子问题库，复用别重拆）。

## 3 · 三个必带的横向维度（最容易漏，单独拎出来跑）
| 维度 | 怎么查 + 为什么 |
|---|---|
| `因果假设(手段对不对路)` | 每个模块在拆表现/数值**之前**先问：**"就算执行满分，这手段凭什么能搬动这个目的？"** 执行到位 ≠ 手段对路（例：靠排行榜拉深度→排行榜驱动竞争不是深度）。把**最脆弱的那条假设**点出来，要求作者论证或给历史佐证。 |
| `基线参照` | 每个"够不够"**必须对参照系**：历史同形式活动 / 上期同类 / 竞品（调 `reference_festival_knowledge_graph` + `reference_x3_festival_performance`）。**禁绝对判**（"$99 够不够"必须 vs 某基线）。 |
| `副作用` | 为主目的有没有**伤到别的盘子**（挖深度挤掉小R低门槛广度、加付费点伤体验、回收过猛伤留存）。列出受损目的 + 受损点。 |

## 4 · 输出形态
- 一棵**拆解树**：节点挂「局部目的」，叶子 = 有没有/够不够 + **证据**（指向 prefab 段 / 配置行 / 具体数值 / 引用的历史数据）。
- 不相关目的标 N/A 并说明。

## 5 · 主观项（验收员判不了 → 标 level=human，不计 pass/fail）
判断没被消灭，是被**挪到最省力的两件事**上，由人定：
- **每层「局部目的」写清楚没**（目的模糊则下面全拆不准）。
- **拆得对不对**（有没有问中要害子问题、§3 因果假设那条脆不脆弱）。

## 6 · 子问题库生长 + 剪枝（自下而上，不预设）
- 拆过的好树存进 `design-merit-qbank.md`，下次同类目的**复用别重拆**；被坑一次 → 补一片叶子。
- 剪枝同其它清单：某叶子连续多轮没命中且非关键 → 评审删。

## 7 · 闭环到复盘（长期自校准）
- 设计期 §1 声明的「目的 + 成功指标」= 上线后**回归的标尺**（接 `reference_x3_festival_monitor` / 复盘）。
- 回归结果回灌 qbank：**哪些拆解/假设预测准、哪些跑偏** → 下次拆法更狠。

## fail-closed
读不到案子 / 无目的声明 / 该给的证据拿不到 → 标 `passed:false` + 「需人工确认」，blocking_failures 带 cannot_verify，不放行。
