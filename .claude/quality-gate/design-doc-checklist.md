# 策划案验收清单（type=design-doc）

> task-checker 跑 design-doc 验收时对照的**单一来源**。
> 标准真源 = memory `workflow_x3_festival_design_doc.md` §六/§八 + `feedback_gsheet_design_doc_format.md`(排版)；**本文件 = 它们的「可机检子集」**——完整知识看 memory，这里只放能机检的客观项，别写全(防两份标准漂移)。
> 改标准就改这里，验收员运行时读这里，不另存副本。
> **回流**：验收/评审发现清单没覆盖的新坑 → 当场补成规则并记『清单进化日志』(见末段)，清单越用越全。

## 类别（按问题性质给规则/日志打标，看哪类高发；**不拆文件**——拆文件的利弊评估见结论：单人维护下跨类归属难+破契约+拖累回流，故用打标代替）
- **排版**：写入写错(插表头/列错位/页签名) / 配色区块 / 列阶梯·横铺schema·拆页签 / 不好看
- **逻辑**：缺件(本地化/配置/机制规则) / 改A没弄清连带B(删除链·依赖链) / 跨处一致(命名·三层·摘要↔详细)
- **交互**：缺状态(状态机) / 表现缺动态(客户端表现·动效·音效)
> 跨切面规则(如 删改一致性)允许多标。task-checker 输出 JSON 时**按类别分组** checks（报告更清晰，读取契约不变）。

## 删改老模块原则（删除链 · 2026-06-03 大富翁案沉淀）
删除/取消一个老系统 = 清一条**依赖链**，不能只删主条：
`主条(升级) → 衍生表现(变色/升级特效/环形经验条) → 关联字段(Event.Level) → 关联配置表(ActvVoyageLevel) → 界面元素(等级UI/左右切级) → 其它处的效果引用(神秘岛「经验+2」、岛屿表「可升级=是」)`。
**中途改过决策**(如「弱化→改直接删」)时，旧决策的残留内容(如③等级变色)必须跟着删——这是最常见的悬空来源。落地后靠上表 `删改一致性` 规则兜底扫。
