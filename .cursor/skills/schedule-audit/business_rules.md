# 排期审核 - 业务规则

## 活动别名映射

以下活动在排期表和上线表中使用不同名称，属于同一活动：

| 排期表叫法 | 上线表叫法 | 说明 |
|-----------|-----------|------|
| 挖矿 | 合成小游戏 | 同一活动，上线表按 schema 拆分为多条 |

> Agent 在匹配时应将这些别名视为同一活动。

## 服务器数量标准

### 参考服务器集合

| 集合名称 | 服务器数 | 说明 |
|----------|---------|------|
| 全服标准集 | 130 | 不含灰度服务器的标准全服 |
| 全服含灰度集 | 142 | 含 12 个灰度/额外服务器 |
| Schema6 标准集 | 77 | S6 分组 |
| Schema3-5 标准集 | 53 | S3-5 分组 |

> 确认方法：取一个已知全服活动（如签到）的服务器列表作为参考基准。

### "不灰"活动规则

**核心规则：标注"不灰"的活动，正式上线时应包含灰度服务器。**

- "不灰" = 该活动不做灰度测试，直接全量发布（含灰度服务器）
- 预期服务器数 = **142**（全服含灰度）
- 如果"不灰"活动只有 130 服务器 → **可能有问题**（缺少灰度服务器）
- 如果"不灰"活动有 142 服务器 → **正常**

### 普通活动规则

- 没有"不灰"标注的普通活动
- 全服预期 = **130** 服务器
- Schema6 预期 = **77** 服务器
- Schema3-5 预期 = **53** 服务器

### Schema 拆分规则

- 许多活动按 schema6 / schema3-5 拆分为两条
- S6 + S3-5 服务器集合**不应有重叠**
- S6(77) + S3-5(53) = 130（全服标准集）

## 时间规则

### 排期标记日 = 部署日

排期审核表中标注的日期是**部署操作日**，活动于次日 00:00 正式开启。

- 排期标记 02.11 → 实际上线 02.12
- 时间比对时需 +1 天偏移

### 多期活动

部分活动有多期标记（如聚宝盆 5 期、挖矿 3 期），每期单独比对时间。

## 跨服类型映射

| 排期表跨服字段 | 上线表"是否跨服"预期 | 上线表"是否跨服排名"预期 |
|--------------|-------------------|---------------------|
| 单服 | 单服 | 否 |
| 跨服-全服 | 跨服 | 是 |
| 跨服-分组 | 跨服 | 视情况 |

## 活动命名检查

上线表活动名称可能沿用历史节日模板，常见问题：
- 名称中包含错误的节日名（如情人节活动写了"圣诞节"）
- 年份错误（如 2026 年活动写了"2025"）
- 应提取当前节日关键词，检查上线表中是否有名称包含其他节日或错误年份

## 报告格式

最终报告结构：

```
# {节日名}活动 - 排期审核 vs 正式上线 总结报告
> 审核日期 / 数据来源

## 一、总览（表格：活动数/日期范围/服务器概况）
## 二、活动条目匹配（已匹配/排期缺失/上线缺失）
## 三、上线时间对比（有标记逐期比对 + 无标记列表）
## 四、服务器配置对比（跨服类型/服务器数量/Schema分割）
## 五、活动命名问题
## 六、待确认事项清单（🔴高/🟡中/🟢低）
```
