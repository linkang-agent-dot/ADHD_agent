## ⚠️ 本地化写入目标：永远写整理版页签

卡册本地化的最终版本必须写入「**整理版**」页签（`复活节-整理版`，sheetId=512019199），
而不是原始页签（`卡册key（复活节）+本地化`）。

整理版内需同步更新两个区域（缺一不可）：
- **【卡包本地化】R95起** — Col B（中文）+ Col C（英文）
- **【卡包描述修改版】R122起** — Col B（修改后中文）+ Col D（英文翻译），Col A（原文）不改

## 换皮验证通用原则

> 这几条原则不只适用于卡册，适用于所有节日配置换皮。

### 原则1：结构验证 ≠ 内容验证，两个都要做
"卡 ID 对了"、"group_id 列表对了"是**结构验证**。
但配置里每个引用外部资源的字段（rewards、drop JSON、goods_id、category_param 等），
里面的 **id 值指向的是哪个节日**，这是**内容验证**，必须单独做，不能默认沿用就是对的。

### 原则2：换皮时，先列"所有含 id 引用的字段"，再逐一核查
不要一边改一边想着"这个应该没问题"，要先建一张清单：
- 哪些字段存了外键 id？
- 这些 id 是节日专属还是通用？
- 通用 → 跳过；节日专属 → 必须换

没有清单就开始改，就会出现"改了主体、漏了奖励"的情况。

### 原则3："节日专属"快速判断法
在 1111 表拿到道具后，看两个字段：
- `S_STR_comment`：注释里通常直接写节日名（"科技节"、"复活节"、"圣诞"）
- `A_MAP_lc_name.txt`：LC key 里通常含节日缩写（`tech_fest`、`easter`、`christmas`）

只要命中关键词就是节日专属，必须换；`item_buff`/`item_speedup`/`item_rss` 类通常是通用的。

### 原则4：验证完才能说"没问题"
说"没问题"之前，要能指出自己检查了哪些字段、覆盖了哪些表。
没有显式跑过的字段，不能默认为正确。

## PowerShell + gws JSON 引号
Windows PowerShell 无法正确传递嵌套 JSON 给 `gws` CLI。
用 Python `subprocess.run()` 包装，彻底回避。

## 大批量 gws 写入静默失败
单次 `values.update` 超过 ~50 行含 JSON 内容的数据时，可能无报错但返回空。
拆成 10-20 行一批，失败时逐行重试。

## 资源 ID 不一定连续分配
科技节卡片 ID 和主题 ID 连续（81卡 + 9主题 = 90连号），但复活节的主题 ID 跳了一段。
永远从实际数据读取 ID，不要用 `max(卡片ID) + 1` 推算。

## 复活节比科技节多一个空列
复活节页签在 Col E 插了"AI出图"空列，导致后续所有列索引 +1。
每次处理新节日必须先分析实际列结构，不能直接复用上一节的列号。

## 简体 vs 繁体
表头可能写"中文名稱"（繁体风格），但实际内容用简体。
除非策划明确要求繁体，一律用简体中文。如需繁体用 `opencc-python-reimplemented` 的 S2T 转换。

## 行共享区域容易漏改
原始页签的多个逻辑区域共享同一批行号（仅用不同列区分），非常容易漏改某个区域。
建议先做完整的列分析，列出每个区域的确切行列范围，再逐区域操作。

## 已被占用的 ID
`111110008`、`111110104`、`111110105` 是科技节卡册奖励道具，分配新道具 ID 时要跳过这段。
注意：`111110008`（永久资源增产特权）是通用道具，6001/6002 rewards 里可以继续用；
但 `111110104`（科技节头像框）和 `111110105`（科技节自选宝箱）是**科技节专属**，
Easter 换皮时必须替换（`111110104` → `111111014` 复活节静态头像框；`111110105` → `111110264` 复活节自选宝箱）。

## 6002 rewards 字段不能可靠判断5★
原以为 6002 rewards 含 `11112924` → ✅5★、含 `11112936` → ❌5★。
但复活节验证发现：极速飞车(60021313) rewards 里没有 `11112936`，实际 6003 最高只有 4★；矿洞寻宝(60021314) rewards 也没有这两个特殊道具，实际 6003 有 5★ 金色传说。
**唯一可信的判断方法**：查 6003 表 col[2] A_INT_star，直接读卡的实际星级。

## 1111 表搜索需注意 token 有效期
gws token 约30–60分钟过期，大批量查询中途可能失效。
症状：query 返回 "Sheet1" 而非 "item" 页签，且结果为0行。
处理：运行 `gws auth login` 重新授权，然后重试。

## 节日 ID 段附近可能被其他类型道具占用
复活节卡包道具段（111110301–309）之后，111110310–320 已被城市皮肤、行军表情等占用。
发现道具"缺失"时，不要默认它在连续段，要用 idrange 实际查。

## Book 里有低星组，不需要专属主题卡包
每个节日 book 包含 8 个 group，其中 3 个是低星组（1★–2★ 或 1★–3★），
这些组通过全册保底卡包（白色/绿色保底）自然覆盖，**不做专属主题卡包**。
换皮时不要误以为"少了3个主题卡包"。

## ⚠️ 6002 rewards 道具 ID 必须逐个对照 1111 表验证（容易漏且曾误判）
遍历 6002 group 时，光看 card_id 和 display_order 是不够的——**A_ARR_rewards 里的每个 `id` 都要在 1111 表查一遍**，
确认它不是上一个节日的专属道具（如皮肤、专属宝箱、专属头像框）。
复活节换皮时遗漏了这一步，导致以下道具仍残留在 6002/6001 rewards 里：
- `111110105` 2026科技节自选宝箱 → 全部8组 + 6001 book completion 均含
- `111110104` 2026科技节头像框-集卡册-静态 → 6001 book completion 含
- `11112936` 科技节-大富翁皮肤 → 60021312 金蛋工坊 含
- `11112924` 弹珠皮肤-科技节 → 60021315 彩蛋大亨 含
**验证步骤**：对 rewards 里每个 id 执行 `python gsheet_query.py row 1111_P2 {id}`，
检查 `S_STR_comment` 和 `A_MAP_lc_name` 是否含节日名称关键词。

## ⚠️ 卡包本地化换皮：Key 前缀改了，但描述正文里的游戏名没改（2026复活节踩坑）

【现象】  
本地化页签里，`easter_fest_pack_core_decipher_name` 的 Key 前缀已正确改成 `easter_fest_`，
但中文名仍是"寻找破译器卡包"、描述仍提到"寻找破译器"游戏 —— 这是科技节子游戏名，复活节根本没有这个玩法。

【原因】  
换皮时只做了"Key 前缀的批量替换"（`tech_fest_pack_` → `easter_fest_pack_`），
Key 后半段的主题词（`_core_decipher_`、`_underground_` 等）以及 Col K/L 的中英文描述正文均未更新。

【影响范围】  
4 个主题卡包的 name + desc，含中英文共 8 行 × 2 列 = 16 个单元格。

【科技节 → 复活节游戏名对照（区域2，Col K 描述里改这里）】

【正确格式】
- **名称**：`XXXX卡包`
- **描述**：`打开即可获得两张3星以上"XXXX"游戏卡片。`
- **英文描述**：`Open to receive 2 "XXXX" game cards of 3 stars or above.`

【科技节 → 复活节游戏名对照（区域2，Col K 描述里改这里）】

| LC Key 主题词 | 正确中文名 | 正确描述 |
|---|---|---|
| `core_decipher` | 矿洞寻宝卡包 | 打开即可获得两张3星以上"矿洞寻宝"游戏卡片。|
| `underground` | 极速飞车卡包 | 打开即可获得两张3星以上"极速飞车"游戏卡片。|
| `lucky_marble` | 彩蛋大亨卡包 | 打开即可获得两张3星以上"彩蛋大亨"游戏卡片。|
| `coin_pusher` | 异族探秘卡包 | 打开即可获得两张3星以上"异族探秘"游戏卡片。|

（`roaming` / `lab_rampage` 对应科技节趣味骰子/天外来客，复活节无专属卡包，不需要对应处理。）

【检查方法】  
换皮完成后，回读本地化区域2（Col J-L，R2-R25），
对每行主题卡包的 name/desc，检查 Col K 中文描述里是否含当前节日的游戏名，而非上一节日名称。

```bash
# 快速回读区域2校验（用 gws）
python -c "
import subprocess, json, os
GWS_CMD = os.path.join(os.environ.get('APPDATA',''), 'npm', 'gws.cmd')
os.environ.setdefault('GOOGLE_WORKSPACE_PROJECT_ID', 'calm-repeater-489707-n1')
params = {
    'spreadsheetId': '1d7AXDs3Jc4cq0XhyKpsWicqJjHcGa6mZQAr7oH12lsw',
    'range': \"'卡册key（复活节）+本地化'!J2:L13\"
}
r = subprocess.run([GWS_CMD, 'sheets', 'spreadsheets', 'values', 'get',
                    '--params', json.dumps(params, ensure_ascii=False)],
                   capture_output=True, encoding='utf-8', errors='replace')
for row in json.loads(r.stdout).get('values', []):
    print(row)
"
```

## 6004 集卡商店是换皮必改项，非常容易漏
6004 表里的 rows 60040026–60040030 是通用的"节日保底卡包"商店条目，
`goods_id` 字段指向具体节日的 1111 道具 ID。每次节日换皮必须同步更新这5行。
科技节 → 复活节换皮时，这5行未被列入初始变更范围，差点遗漏。
下次换皮时直接 `python gsheet_query.py tail card_6004 10` 先检查。
