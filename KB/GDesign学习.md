## 【已落地】checker（独立验收员）下沉到各流程

GDesign 最值得学的点：把"判定是否合格"从执行者手里拿走，交给一个**无写权限、无对话历史、按客观清单跑 block/warn 的独立验收 agent**（执行/验收物理分权）。

我们已自建一套 `quality-gate`（位置 `C:\ADHD_agent\.claude\quality-gate\`）：
- 通用验收员 `task-checker`（按任务 type 选清单）+ 收工 Stop hook 守门（产物没验收就拦住，fail-closed）。
- 已覆盖：策划案 / 配置·换皮 / i18n（加新类型 = 只写一份 `<type>-checklist.md`）。
- 客观项机器判（页签齐/占位残留/分支正确/引用ID/Reward seq/MailID/key命名…），主观项（颗粒度/数值合理性/文案）只列出逼人看。

## 【待反馈①】前置 gate（"问全才开干"）对迭代式策划流程不适配

- **现状**：L1_game_ux 阶段一是"苏格拉底问全 → 确认 → 才生成"，隐含假设"需求开工前能定全"。
- **问题**：实际策划流程常是**迭代涌现式**——信息少就先开干，边讨论边补、案子边长，最后成型才走 checker。强制"问全才开干"会**堵住靠讨论收敛这个核心工作方式**。
- **建议**：gate 不该只放开头；关键 gate 挪到 **"讨论 → 生成正式案"的交界**——生成前做一次"收敛确认"（把讨论里确定的内容汇总成清单 → 对一遍 → 再生成）。
- **为什么 checker 补不了这缝**：后置 checker 只看最终产物、看不到讨论过程，所以查不出"**讨论里定了、但生成时漏掉**"——而这正是长讨论流程最容易丢东西的地方。

## 【待反馈②｜更根本】工作流该分型，颗粒度下沉到"子功能"（不是 feature 级线性管线）

- **老工作流**（GDesign 现在）= 先把全部策划案内容定全 → 再做 UX = **换了个工具的瀑布流**，没真正脱离老流程。
- **实际工作流**：单位是"一个功能下的各个子功能"，各跑各的迭代 loop：
  - **交互模块**：先跑出**最简交互单元** → 看反馈 → AI 不断调到匹配（**原型先行**，因为体验脱离实物判断不了）。
  - **数值模块**：**直接跑当前模块内容** → 边调边改（**内容先行**，因为数值能直接评判）。
- **分型依据**：这个子功能的好坏**能不能脱离实物直接判断**——能 → 内容先行；不能 → 原型先行。
- **gate / checker 随之改**：
  - 没有一个大的前置 gate；gate = **每个 loop 的收敛点**（这个交互单元调匹配了吗 / 这个数值模块调到位了吗），轻量、贴着 loop。
  - checker **下沉到子功能/模块级** + 总装时再加一道（各模块拼成完整案子时防漏、防冲突）。
- **一句话**：DesignDeck 整条**线性、feature 级、定全才开干**的管线，对迭代式设计是**错的颗粒度**——该按子功能拆、按类型分流、各自迭代。
