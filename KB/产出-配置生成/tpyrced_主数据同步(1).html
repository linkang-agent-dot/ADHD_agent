🛡️ 抗风险措施

风险场景 等级 应对措施
QA 表有重复 ID 低 写入前自动去重，同一 iap_id 只保留第一条
数据量超出读取范围 低 读取范围已扩展到 10000 行，覆盖当前数据量
API 调用失败 低 分批写入（每批 5 条），单批失败不影响其他批次
误覆盖已有数据 低 只做增量追加，不修改/删除已有数据
分类不准确 中 未匹配的标记为"未分类"，后续可人工补充规则
命令行过长 低 使用临时 bash 脚本文件执行，避免参数长度限制

🔄 迭代历程

V1.0 - 基础同步

问题：需要手动同步礼包数据
方案：实现基于 temp_type 的自动分类和增量同步

V1.1 - 修复命令行过长

问题：ENAMETOOLONG 错误，gws 命令参数过长
方案：改用临时 bash 脚本文件 + 分批写入（每批 5 条）

V1.2 - 分类规则重构

问题：基于 temp_type 分类不准确，实际应按礼包名称分类
方案：重写为基于名称关键词匹配的 NAME_RULES 规则引擎（70+ 条规则）

V1.3 - 去重机制

问题：QA 表存在 24 条重复 iap_id，导致主数据表出现重复
方案：写入前对 QA 数据按 iap_id 去重，同一 ID 只保留第一条

V1.4 - 扩展读取范围

问题：部分礼包（如"掉落转付费"）未同步，因超出 5000 行读取范围
方案：读取范围从 A8:Z5000 扩展到 A8:Z10000
