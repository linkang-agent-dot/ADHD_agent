# 模式 07 · 假设必标注：推断和实证不许长得一样

## 通用形
写进知识库的内容分两种：**实证**（验证过的事实）和**推断**（合理但未实锤的解释）。推断必须**行内显式标注**（「未验证」「推测」），验证后回写删标。**混在一起的后果：下一个读者把假设当铁律用。**

## 识别信号
- 结论句里有"可能/大概/应该是/多半"——写的时候你自己知道是猜的
- 根因分析只做到"最合理解释"没做到"复现确认"
- 数据结论的样本量小到换一批就可能翻

## 动作
- 写入时：`〔结论〕（⚠️ 推测未验证：依据是〔什么〕，验证方法是〔什么〕）`。
- 下次任务触碰到同一对象时，**先做那个验证**，然后删标或改写。
- 不需要为此建置信度分级制度（本体系知识写入即高置信，分级无区分度）——只需要这一个二元标记：**验过 / 没验过**。

## 案例（脱敏）
原体系 memory 里存在极少数推断型条目（如"某配置项的名字可能是历史复用残留"——合理解释但没实锤）。它们混在几百条实证知识里，冷启动的 agent 无法分辨"这是定论还是假设"。2026-07 裁决：不引入三级置信度（那是为"观察人"的场景设计的），只保留"推断必标注"这一个习惯。

## 反模式
- ❌ 为了让报告"完整"，把猜的部分写得和查实的部分一样笃定
- ❌ 标注了"待验证"但永远不回来验证（写入时就要附验证方法，降低回验成本）
