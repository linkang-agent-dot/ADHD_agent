# 模式 01 · 标识符禁硬抄，现场解析

## 通用形
当多个系统/项目的**标识符空间重叠**（同一个 id 在两边都存在但指向不同东西）时：**禁止从文档/历史记录硬抄标识符，必须每次用官方解析工具现场解析。**

## 识别信号
- 两个项目由同一套系统克隆/演化而来（配置表、数据库、服务集群）
- 同一个 id/编号在两边都"能用"——查询不报错，只是返回的是**另一边的数据**
- 团队有"老项目的对照文档"，新项目的人习惯从里面抄

## 动作
- 内核铁律：〔系统X〕的标识符一律用〔解析命令〕现解，禁止引用〔旧文档〕里的值。
- 把解析步骤**焊进查询工具**：工具内部先 resolve 再查，绕开工具才可能违规。
- 在路由表里显式写交叉警告："⚠️ A 域的 id 文档是 A 专用，B 域别用"。

## 案例（脱敏）
原体系两个游戏项目共用一套 Google Sheet 配置表体系，子表 id 空间重叠。AI 从 A 项目的配置库文档硬抄 id 去查 B 项目的表——**查询成功、返回了 A 项目的数据、全程无报错**。数据静默错误比报错恶劣一个量级：它会被当成正确结果用进后续决策。
教训落地为三层：铁律（必现解禁硬抄）+ 文档头警告 + 解析工具作为唯一入口。

## 反模式
- ❌ "上次就是这个 id，直接用"——上次对不代表这次对，重叠空间里"能查到"不等于"查对了"
- ❌ 只在文档里写警告不改工具——文档警告防君子不防赶时间的人
