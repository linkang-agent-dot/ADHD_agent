# 模式 03 · 真源切换后，旧生成链路必须物理断开

## 通用形
当数据的**真源**从 A 迁到 B（xlsx→tsv、数据库→缓存、旧系统→新系统），**从 A 生成 B 的旧链路就变成了污染源**——任何人再跑一次旧生成，就会用过时的 A 覆盖已在 B 上做过的新修改。切换不算完成，直到旧链路被物理断开或加锁。

## 识别信号
- "以后直接改 B，不用再从 A 生成了"——这句话说出口的那天，风险诞生
- A 还留在仓库里且看起来仍然"可用"
- 生成脚本还在原位、无警告、任何人/AI 都能跑

## 动作
- 切换时**同步处理旧链路**，任选其一：删除生成脚本 / 脚本加拦截（检测目标已被直改则拒绝运行）/ 把 A 移出工作区。
- 在内核铁律 + 真源相关的 memory 里**双写**："真源已是 B，禁止对已改过的对象重跑 A→B 生成"。
- 有"两边同改"过渡期的（翻译等特殊流程仍走 A），把**哪些对象走哪条链**写成显式清单。

## 案例（脱敏）
原体系导表流水线从"改 xlsx → 生成 tsv"迁移为"直接改 tsv"。迁移后 xlsx 和生成脚本原地保留。风险：对已经 tsv-直改过的表再跑一次生成，就会**用旧 xlsx 静默覆盖回旧值**——改动无声消失，且 git diff 看起来像一次正常的批量更新。
教训固化为：内核铁律（禁对已改表重跑生成）+ 迁移专项 memory + 例外清单（翻译流程仍走旧链的表显式列出）。

## 反模式
- ❌ "留着旧链路以防万一"却不加锁（万一 = 一定）
- ❌ 只在 memory 里记、不在生成脚本处设防（下一个不读 memory 的 agent 照样跑）
