# 模式 08 · 唯一入口：一个案子只有一个"从这里进"

## 通用形
每个项目/案子指定**一份**入口文档，所有关键产物（方案、决策记录、复刻清单、数据结论、目录说明）从它出发可达。**知识不怕多，怕的是不知道从哪进**——没有唯一入口的沉淀等于没沉淀。

## 识别信号
- 一个案子的产物散在 ≥3 个位置（聊天记录、临时目录、知识库、表格）
- 被问"那个项目的方案在哪"时需要回忆/搜索而不是给一个链接
- 交接时发的是"一堆文件"而不是"一个文档"

## 动作
- 案子启动或第一次产出时就建入口文档；此后**每个新产物在入口挂一行指针**（产物什么状态也标上：FINAL/草稿/废弃）。
- 索引层（MEMORY.md 的 project 条目）只指向入口，不直接指向散件。
- 案子收口时，接管化三件套挂进入口的"文档指针"段——入口从"进行中看板"变成"交接总文档"。
- 入口文档自身要**自包含可读**：新人只读它+它指向的 FINAL 件就能接手，不需要读废稿和过程稿。

## 案例（脱敏）
原体系每个在途项目的 memory 条目都强制写"唯一入口=〔路径〕"，交接验收标准是"换人先读入口总文档"。反面教训在先：早期项目产物齐全但散落，接手 agent 把试错版当上线版用——此后"产物目录必须标 FINAL/废弃"和"唯一入口"一起成为收口标配。

## 反模式
- ❌ 用"目录结构很清晰"代替入口文档（清晰是你觉得，接手人面对的是 40 个文件）
- ❌ 入口文档建了但不维护（指针断链的入口比没有更误导）
