# 模式 02 · 合并三铁律（预检驱动 / 禁批量选边 / 双向审计）

## 通用形
凡是依赖**自定义合并驱动/脚本**的仓库（大型数据文件、生成物、非纯文本），合并操作必须：**① 开工先验驱动在位；② 禁止批量选边；③ 合完跑双向丢失审计。**

## 识别信号
- 仓库里有 `.gitattributes` 指定 merge driver，或团队有专用合并脚本
- 被合并的文件是"行级数据"（配置表、翻译表、id 表）——一行丢失不会引起任何语法错误
- 发版流程要求两个长期分支互合

## 动作
1. **预检**：合并前验证 `git config --get merge.<driver>.driver` 非空；空 = 先装钩子再开工。**驱动缺失不报错，只是静默按默认策略合——丢行不留痕。**
2. **禁批量选边**：禁止 `checkout --ours/--theirs`、禁止"保留XX侧"这类一刀切——数据文件的冲突必须逐行/用专用工具合。
3. **双向审计**：合完对比两个方向（我有你无 / 你有我无）的行差异，确认每条差异都"有意为之"；再跑一次本地构建/导出 exit 0 才允许 commit。
4. 发版方向的合并必须是**真 merge commit**，禁止线性怼上去（丢掉合并语义 = 下次合并地狱）。

## 案例（脱敏）
原体系配置仓两次事故：a) 新环境没装 merge driver，合并"成功"但静默丢了几十行配置，上线后玩家侧功能缺失才发现；b) 双节日分支互合时图省事批量选边，把对方分支独有的整块配置抹掉。
教训固化为：官方合并 skill（预检+审计打包成流程）+ 三铁律进内核。

## 反模式
- ❌ "合并没报冲突 = 合并没问题"（数据文件最危险的合并恰恰是不报冲突的）
- ❌ 审计只查单向（只看"我丢没丢"，不看"我把对方的丢没丢"）
