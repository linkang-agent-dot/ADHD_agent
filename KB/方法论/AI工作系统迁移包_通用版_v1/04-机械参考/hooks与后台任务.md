# 机械参考：hook 三位点 + 验收门 + 后台任务

> 原则回顾（换皮清单·皮5）：**自觉为主，hook 兜底**。hook 是给"被违反过的规则"的物理保险，不是给所有规则的。脚本逻辑是你的业务（皮），这里给结构（骨）。

## 一、三个 hook 位点（Claude Code `settings.json`）

| 位点 | 时机 | 用途 | 原体系实例（脱敏） |
|---|---|---|---|
| `SessionStart` | 会话开始 | 把"该带进上下文的状态"注入（旗标/待办/告警） | 开会话检查标志文件，有未处理事项就提醒 |
| `PreToolUse` | 每次写/执行工具调用前 | **物理拦截**危险操作——不靠 AI 记得，靠拦 | 配置仓隔离闸：对特定路径的写操作校验分支/环境，不符直接 block |
| `Stop` | AI 回合结束 | 收工质检门：有未过验收的产物 → 拦住收工 | quality-gate：检查待验收标记，未过 checklist 的产物拦停并提示跑 checker |

配置片段：

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Edit|Write|Bash",
      "hooks": [{ "type": "command", "command": "python <你的闸门脚本>.py", "timeout": 10 }]
    }],
    "Stop": [{
      "matcher": "",
      "hooks": [{ "type": "command", "command": "python <你的质检门脚本>.py", "timeout": 10 }]
    }]
  }
}
```

脚本约定：从 stdin 读 JSON（含工具名/参数），**exit 0 放行**；PreToolUse 需要拦截时按宿主文档返回 block 输出。写闸门只拦"静默且不可逆"的操作——拦太宽，用户会把闸门整个关掉。

## 二、验收门的构成（配合《判执分离》）

```
.claude/quality-gate/
├── <产物类型>-checklist.md   # 每类产物一份，按四原则写
└── pending_verify_gate.py    # Stop hook：扫描待验收标记，未过则拦
```

流程：执行 agent 产出时打"待验收"标记 → 独立 checker（沙箱只读 agent）按对应 checklist 跑检查 → 通过才摘标记 → Stop hook 确认无残留标记才放行收工。

## 三、后台自动任务（系统的"心跳"，可选进阶）

用操作系统计划任务 + 无头 CLI（`claude -p "<任务prompt>"`）跑周期工作。原体系在跑的类型：
- **巡检类**：工作日每小时扫 bug 单，变化写摘要文件（SessionStart hook 顺手提醒）
- **日报类**：每日拉数据出监控报告
- **周报类**：每周五出 token 用量 + **本周新增知识归纳清单**（用户批量把关沉淀质量的机制，见《收工自检四步》）

无头运行两个坑（原体系实测）：a) 脚本里跑无头 claude 必须**关掉 Stop hook**（否则质检门把自动任务拦死）；b) 输出落文件，别依赖 stdout。

## 四、上线顺序建议

1. 第 1 个月：**零 hook**——先靠内核自觉跑，观察哪条规则会被忘。
2. 有铁律被违反 → 上第一个 PreToolUse 闸门（只拦那一条）。
3. 有验收清单后 → 上 Stop 质检门。
4. 日常任务出现固定周期 → 再考虑后台任务。
**机械是长出来的，不是装出来的**——一开始全装上，你会在没有事故史的情况下维护一堆拦截逻辑。
