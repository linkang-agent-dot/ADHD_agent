# memory 事实文件 · 格式规范

## 一文件一事实

每个 md 文件只承载**一个主题**的知识。文件名 = 类型前缀 + 主题 slug：
`reference_xxx.md / feedback_xxx.md / project_xxx.md / workflow_xxx.md / user_xxx.md`

## frontmatter

```markdown
---
name: <短横线slug>
description: <一行摘要——AI 判断相关性靠它，写触发场景不写空话>
metadata:
  type: user | feedback | project | reference
---
```

## 正文要求（按类型）

### feedback（最重要的类型——行为纠偏）
```markdown
<发生了什么，一段>

**Why:** <为什么这是对的做法/为什么那次错了——没有 why 的结论无法迁移>

**How to apply:** <下次遇到什么场景、具体怎么做——写到 AI 能直接执行>
```

### reference（链路/口径/系统知识）
- 写"**现在还成立的事实**"，不写探索日记。
- 坑用 ⚠️ 前置标出；推断未实锤的结论行内标「未验证」（见模式07）。

### project（在途案子）
- 必含一行：`唯一入口 = <路径>`。
- 记状态和关键决策指针，不记流水账（流水账进案子自己的目录）。
- 结案后：有复用价值的升 reference，没有的从索引摘除。
- **半活案子**（单次交付完成但业务线还会继续，如客户待跟进）：先只在 MEMORY.md「在途案子」段留一行（状态+下一步）；同一案子**第二次**产生任务时再建 project 文件。

### user（用户画像）
- 只记用户亲口说过/行为中明确表现的。
- **一句话级偏好**直接写在 MEMORY.md 的 User Preferences 段行内即可；**成块画像**（身份+业务域+沟通偏好）才建 `user_` 文件、索引挂指针。

### workflow（流程规范）
- 写成"步骤清单 + 每步防什么"，标准同 feedback 的 How：AI 不需要口述补充就能照跑。

## 双链
正文里想到相关主题就写 `[[对方的name]]`——链接一个还不存在的名字也可以，它标记"值得写"，不是错误。

## 写入纪律
1. **当场写**（收工自检①触发），≤5 分钟一条。
2. **写前查重**：更新旧文件 > 新建重复文件（两个真源会分歧）。
3. 写完在 MEMORY.md 加一行索引——**没有索引的 memory 等于不存在**。
4. 发现某条错了/过期了：当场改或删，不留"历史版本"。
