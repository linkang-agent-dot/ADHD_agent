# 三段防线：操作前 / 操作中 / 操作后

> 框架是骨；每段里的具体条目是皮（从你的翻车史长出来）。骨的价值：翻车后你知道**教训该挂在哪一段**。

## ⛔ 操作前（防"没看清就动手"）

| 通用条目 | 防什么 |
|---|---|
| 改共享数据前，先追**完整引用链至少两层** | 只看一层 = 改了 A 不知道 B 在用，下游静默坏 |
| 写入真源前，先确认**分支/环境/目标对象** | 写对了内容、写错了地方——最冤的事故 |
| 删除/清理前，先出「保留/删除」清单等确认 | AI 对"什么重要"的判断不可信 |
| 大改动先给方案+影响范围等拍板 | 边想边改一堆文件 = 回退成本爆炸 |

## 🔒 操作中（防"用错真源/走错路径"）

这一段几乎全是皮——每条都长这样：**"在〔系统X〕里做〔操作Y〕时，禁止〔省事做法〕，必须〔安全做法〕，因为〔那次事故〕"**。
通用模式先借用 `02-模式库/`：标识符现场解析、合并三铁律、真源切换断旧链。

判断一条规则该不该进这段：**违反它的后果是否"静默"**——会报错的错误不需要铁律（错误自己会喊），静默的错误才需要（数据错了没人知道）。

## ✅ 操作后（防"做完了但没做对"）

| 通用条目 | 防什么 |
|---|---|
| **端到端反查**：所有真源层各自验证一遍（源头系统+本地缓存+推送目标） | 只验自己改的那层 = 层间不同步照样上线事故 |
| 有自动构建/校验的，改完**必须触发并等结果**，把成功/失败如实报告 | "我改完了"≠"它生效了" |
| 交付物按受众裁剪后再发（见信息有效度） | 把过程当结果倾倒给用户 |

## 防线的升级路径

```
第一次翻车 → feedback 事实文件（带Why+How）
     ↓ 同类第二次
CLAUDE.md 铁律段（每会话常驻）
     ↓ 铁律被违反（AI或人忘了守）
hook 物理闸门（不靠自觉，靠拦截）→ 见 04-机械参考
```

**不要跳级**：一次性的坑写成常驻铁律 = 内核肥胖；所有规则都上 hook = 维护地狱。让事件的重复次数决定规则的强度。
