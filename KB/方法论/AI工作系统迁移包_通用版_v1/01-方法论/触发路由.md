# 触发路由：关键词 → 先读知识 / 操作词 → 先调工具

## 解决什么问题
AI 的默认行为是"凭通用知识直接干"。在多业务域环境里这是事故源：**拿错背景知识干错活**——尤其当两个业务域长得像的时候（原体系事故：项目A和项目B共用一套配置表系统、id 空间重叠，AI 拿 A 的知识查 B 的表，**静默返回错误数据不报错**）。

路由 = 在内核（CLAUDE.md）里写死：**出现什么词，动手前必须先读什么文件 / 先调什么工具**。

## 两类路由

### 1. 关键词 → 知识库（防拿错知识）
```
### 〔业务域〕
触发词：〔该域的项目名/黑话/系统名〕
先读：memory/reference_〔域〕.md（1-2 个文件，别堆）
专属工具：〔该域固化的 skill〕
```
- 每个域指向的"先读"文件要**极少而准**（1-2 个入口文件，入口里再双链下钻）——路由指 10 个文件等于没指。
- 两个域有共享系统但真源不同时，在路由里**显式写警告**（"⚠️ X 域别用 Y 域的 id 表"）。

### 2. 操作词 → 强制工具（防漏调）
只把**最容易漏调**的写进内核强制表（原体系只放了 3 个：改BUG→调试流程、配置检查→检查工具、上线→部署工具）。其余工具靠自身触发文档自动命中。
判据：这个工具被漏调过 ≥2 次 → 进强制表。

## 什么时候加一行路由
- **首次唤醒**：按用户自报的主业务域先立**空骨架**——"先读"栏指向还不存在的文件时写 `〔待建：预期文件名〕`，该域第一次真干活时建文件、删标。
- 骨架之外的新域：**第三次**出现 → 建域路由（第一二次先把 reference 文件写好）。
- 发现 AI 在某域用了错误/过时的背景知识 → 当场补路由（这是翻车驱动的插队通道）。

## 维护
- 路由表也**有进有出**：域死了（项目结束）就移到归档段，别让内核常驻死域。
- 路由的触发词要包含**你实际会说的口语**（黑话、缩写），不只是正式名称——路由是给真实对话用的。
