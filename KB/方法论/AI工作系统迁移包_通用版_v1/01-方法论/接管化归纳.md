# 接管化归纳（本体系的灵魂）

## 一句话
沉淀的验收标准不是"记下来了"，而是：**一个新 AI 冷启动（无对话历史），能否在「修问题 / 接管模块 / 复刻项目」任一入口，立刻拿到全部 keypoint 并开始干活。** 不能 → 当场归纳，不许收工。

## 为什么（原体系的真实教训）
原主人积累了 150+ 条记忆后发现一个病：沉淀极厚，但全停在两个高度——
- **太碎**：一条条孤立事实（reference）；
- **太具体**：项目日记式的**时间线流水账**（v0.1→v0.2→v0.3…）。

流水账是编年史，给旁观者看，不给接手人看——**理由埋在时间线里 = 做完就没了**。缺的是"中间高度"：按**接管入口**组织的可复用结构。

## 怎么做

### 深度随任务大小伸缩
- **小任务** = 回写一行 memory / 给改动补一句"为什么这么改、解决了什么"。
- **一个案子/项目** = 三件套，并挂进该案子的**唯一入口**文档：

### 三件套
1. **`_决策记录_按模块.md`** — 每条四栏：`决策 / 当时卡哪 / 试过什么失败了 / 解决了什么`。**显式标 ⚠️ 被否决的方案**（防接手人重蹈覆辙）。**按模块排，不按时间排。**
2. **`_复刻清单.md`**（原体系叫"换皮清单"）— 骨 vs 皮：哪些结构可直接复用（copy）、哪些参数必须替换（swap），做成 copy-and-swap 对照表 + checklist。注意：它是**给下一轮的遗产**，不是建造期工具——不会被任何干活流程自动触发，只能靠收口仪式逼出来。
3. **产物目录 `_目录说明.md`** — 标 `FINAL / 废弃 / 留档`。试错版和上线版混在一起时，接手人必拿错废稿。非破坏式：写索引，不删不改原文件。

### 唯一入口
三件套必须挂进案子的唯一入口文档（一个案子只有一个"从这里进"），否则归纳了也没人找得到。

## 铁律
- **禁止只留时间线流水账。** 流水账可留作深挖版，但不能是唯一沉淀。
- **默认动作，不问用户。** 归纳不是选项，判完判据直接做，一句话带过即可；用户把关靠周期性归纳清单（每周把新增知识列成"知识|模块|来源"给用户过目），不逐次打扰。

## 与"项目 playbook"的关系
三件套是案例级；同类案子做过 2-3 个后，跨案子再抽一层 = 该类项目的 playbook。
