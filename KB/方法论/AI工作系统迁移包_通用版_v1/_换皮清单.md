# ★ 换皮清单 · 骨 vs 皮

> 这套系统 = 骨（可迁移的结构与方法，随包带走）+ 皮（你的业务内容，必须自己长）。
> 本清单回答一个问题：**"要把这套东西用好，我需要长出什么内容？"**
> ——这是全包最重要的文件。方法论文章可以慢慢看，这张表决定你能不能用起来。

---

## 一、骨（随包带走，开箱即用，不用改）

| 部件 | 位置 | 作用 |
|---|---|---|
| 内核行为规范 | `CLAUDE.md` 上半部 | 信息有效度 / 三段防线 / 收工自检四步——通用，直接生效 |
| 方法论 8 篇 | `01-方法论/` | 接管化归纳、判执分离、薄索引、工具固化等——换任何行业成立 |
| 模式库 8 条 | `02-模式库/` | 带真实案例的通用防坑模式，遇到对应场景照做 |
| 记忆目录骨架 | `03-记忆骨架/` | MEMORY.md 薄索引模板 + 事实文件格式规范 |
| 机械结构 | `04-机械参考/` | 三个 hook 位点怎么用、验收门怎么搭的结构图纸 |

## 二、皮（必须你自己长——共 7 项）

> 每项四栏：**为什么带不走 / 包里给了什么 / 怎么长 / 什么事件触发去长**。
> 总原则见第三节：**皮是事件逼出来的，不是填表填出来的。**

### 皮1 · 触发路由表（关键词 → 先读哪份知识）

- **为什么带不走**：路由的价值在"防止拿错知识干错活"（原主人的事故：两个项目的配置表 id 空间重叠，拿错真源会静默返回错误数据不报错）——你的业务域、你的关键词、你的真源，只有你有。
- **包里给了**：`CLAUDE.md` 的路由表占位段 + 填写格式。
- **怎么长**：首次唤醒时按用户自报的主业务域先立**空骨架**（"先读"标〔待建：预期文件名〕）；骨架之外的新域，第一次干活时建 reference 文件、**第三次出现**时补进路由表。
- **触发事件**：发现 AI 在某个域里用了错误/过时的背景知识 → 当场补路由。

### 皮2 · 铁律（防线的业务参数）

- **为什么带不走**：铁律 = 真实事故的疤。包里的三段防线是框架（操作前追引用链/确认目标、操作中守真源、操作后反查），但"具体查什么命令、哪个系统禁止硬抄什么"来自你的翻车史。
- **包里给了**：`01-方法论/三段防线.md` 框架 + `02-模式库/` 8 个已验证模式（可先当现成铁律借用）。
- **怎么长**：翻车 → 当场写一条 feedback 事实文件（带 Why + How）→ **同类坑第二次出现**时升级进 `CLAUDE.md` 铁律段。**例外**：用户明示「以后必须/禁止……」= 当场确认了候选铁律，直接入内核，不等第二次。
- **触发事件**：任何一次"AI 做了不可逆/错误操作"或"差点做了"。

### 皮3 · memory 事实层（原主人有 199 条，你从 0 开始）

- **为什么带不走**：全部是业务实证——表结构、链路、口径、坑。抽象掉具体性就没有价值了。
- **包里给了**：`03-记忆骨架/`——薄索引模板、四类型格式规范（user/feedback/project/reference）、一条示例。
- **怎么长**：每次收工自检①（见 `01-方法论/收工自检四步.md`）："这次有新规律/踩坑/新链路吗？"有 → 当场写一条，索引加一行。
- **触发事件**：每次任务收工（默认动作，不问、不攒）。

### 皮4 · 验收清单（quality-gate 的 checklist）

- **为什么带不走**：每条 check 防的是一个具体事故（"某字段跳号 → 服务端静默吞奖励"），事故是你的。
- **包里给了**：`01-方法论/判执分离_独立验收.md` 里的**清单设计四原则**（默认有罪 preamble / 强制留证据 / 内联防什么事故 / 开放项+剪枝）——照着写你自己的。
- **怎么长**：某类产出（配置/文档/翻译/数值）第一次出严重质量问题 → 为这类产出建一份 checklist，之后该类产出收工必过。
- **触发事件**：同类产出第二次出问题（第一次可以只写 feedback，第二次说明需要制度化）。

### 皮5 · hook 脚本（物理闸门）

- **为什么带不走**：闸门逻辑全是业务判断（"这个路径的写操作必须先确认分支"）。
- **包里给了**：`04-机械参考/`——三个位点（SessionStart 注入提醒 / PreToolUse 拦截误操作 / Stop 收工质检门）的作用、settings 配置片段、脚本骨架。
- **怎么长**：某条铁律**靠 AI 自觉守了两次都守住了 → 不用写**；一旦发现 AI 忘守（或你自己都会忘）→ 把它变成 hook 物理拦截。
- **触发事件**：铁律被违反的那一次。原则：**自觉为主，hook 兜底；hook 是给"会被忘的规则"的，不是给所有规则的。**

### 皮6 · 工具层（skill / 固化脚本）

- **为什么带不走**：工具封装的是你的系统调用、你的鉴权、你的流程。
- **包里给了**：`01-方法论/工具固化.md`——固化时机判据与反模式（禁止留一堆一次性脚本）。
- **怎么长**：**同一个操作第二次手工做**时固化成脚本；一个流程跑通三次后封装成 skill（含文档，让 AI 能自动触发）。
- **触发事件**：你意识到"这个我上次做过"的那一刻。

### 皮7 · 归档路径与唯一入口（KB 目录约定）

- **为什么带不走**：目录结构跟你的产出类型走。
- **包里给了**：`01-方法论/接管化归纳.md` 里的三件套模板与"唯一入口"要求。
- **怎么长**：第一个真实项目收口时定下：产出归到哪、案子的唯一入口文档放哪。之后所有项目沿用。
- **触发事件**：第一个跨越 3 天以上的项目结束时。

## 三、长皮的总原则

1. **皮是事件逼出来的**。三个事件源对应三类皮：
   - **翻车** → feedback / 铁律 / hook（皮2、皮5）
   - **重复** → 路由 / 工具（皮1、皮6）
   - **收工** → memory / 归档 / 验收清单（皮3、皮4、皮7）
2. **当场做，不攒批**。"回头统一整理"= 永远不整理。所有皮的写入动作都应该 ≤5 分钟，做不到说明格式太重。
3. **有进有出**。路由表、铁律、清单都会老化——某条规则半年没被触发过，评估降级或删除。膨胀的清单和没有清单一样没人守。

## 四、验收：怎么知道皮长好了

用这套系统自己的**接管化判据**自测（详见 `01-方法论/接管化归纳.md`）：

> 假设明天换一个全新的 AI 会话（冷启动、无对话历史），只靠你的 CLAUDE.md + memory + KB——
> 它能否在「修一个线上问题 / 接管一个模块 / 复刻一个项目」任一入口，立刻拿到全部关键信息并开始干活？

- 能 → 皮长好了。
- 不能 → 缺哪块补哪块，本清单第二节就是补口清单。
