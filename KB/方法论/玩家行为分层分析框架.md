# 玩家行为分层分析框架

> 从 X2 大富翁（74服，2026-04-29~05-06）案例提炼。
> 原始报告：`产出-数据分析/X2大富翁行为分析_74服_20260429.md`

## 框架概述

**适用场景**：某个活动参与度低、转化差，需要定位根因时。

**核心思路**：从粗到细，逐层下钻，避免过早归因。

```
第1层：reason_id 大类行为排名
    → 定位：该活动在玩家整体行为优先级中排第几？
    ↓
第2层：reason_sub_id 子行为细拆
    → 定位：高频大类里，玩家具体在做什么子任务？
    ↓
第3层：ods_user_event counter 级真实操作事件
    → 定位：客户端最底层行为，发现隐藏高频行为
    ↓
第4层：活跃度分群对比
    → 定位：核心玩家 vs 全体，差异在哪里？
```

---

## 第1层：reason_id 大类行为排名

**目的**：看目标活动是否被玩家"优先做"。

**数据源**：`dl_user_asset_reason_detail_h`

```sql
SELECT reason_id,
       SUM(add_times) AS gain_times,
       SUM(use_times) AS cost_times,
       COUNT(DISTINCT user_id) AS participants
FROM iceberg.v1089.dl_user_asset_reason_detail_h
WHERE server_id = '{server_id}'
  AND partition_date BETWEEN '{start}' AND '{end}'
  AND user_id IN ({user_ids})
GROUP BY reason_id
ORDER BY gain_times DESC
```

**解读方向**：
- 目标活动排名靠后 → 玩家把时间花在别处，活动本身不够吸引
- 目标活动排名靠前 → 不是参与度问题，是转化/深度问题

**大富翁案例**：大富翁（`monopoly_gacha`）排第11位，属于边缘行为。
玩家最爱确定性回报（PVE、活动任务），随机性玩法天然处于劣势。

---

## 第2层：reason_sub_id 子行为细拆

**目的**：拆解高频大类，找出玩家真正渴望的资源/任务类型。

**数据源**：`ods_user_asset` LEFT JOIN `dim_asset`

```sql
SELECT a.reason_sub_id,
       COUNT(DISTINCT a.user_id) AS participants,
       COUNT(*) AS total_actions,
       SUM(a.change_count) AS total_gain,
       ARRAY_AGG(DISTINCT b.asset_name) AS reward_names
FROM iceberg.v1089.ods_user_asset a
LEFT JOIN iceberg.v1089.dim_asset b ON a.asset_id = b.asset_id
WHERE a.server_id = '{server_id}'
  AND a.partition_date BETWEEN '{start}' AND '{end}'
  AND a.reason_id = '{target_reason}'
  AND a.change_type = '1'
  AND a.user_id IN ({user_ids})
GROUP BY a.reason_sub_id
ORDER BY total_actions DESC
LIMIT 30
```

**解读方向**：
- 参与人数呈漏斗结构（全量→进阶→深度→高难）
- 频次最高的任务 = 玩家最渴望的资源类型
- 如果目标活动的奖励和这些资源不对齐 → 是吸引力问题

**大富翁案例**：加速道具是第一渴望资源（~3,200次），远超资源类任务。
大富翁格子奖励与此弱关联，导致玩家缺乏驱动力。

## 第3层：ods_user_event counter 级真实事件

**目的**：突破资产维度的盲区，发现资产表看不到的高频行为。

**数据源**：`ods_user_event`

```sql
SELECT event_name,
       COUNT(*) AS cnt,
       COUNT(DISTINCT user_id) AS users,
       ROUND(COUNT(*) * 1.0 / COUNT(DISTINCT user_id) / {days}, 1) AS avg_per_user_per_day
FROM iceberg.v1089.ods_user_event
WHERE server_id = '{server_id}'
  AND partition_date BETWEEN '{start}' AND '{end}'
  AND user_id IN ({user_ids})
GROUP BY event_name
ORDER BY cnt DESC
LIMIT 40
```

**扩展**：对高频事件做 JSON 参数拆解（如宝箱ID、关卡cfgid、战力等）：
```sql
SELECT json_extract_scalar(event_paramter, '$.cfgid') AS cfgid,
       COUNT(*) AS cnt,
       AVG(CAST(json_extract_scalar(event_paramter, '$.playerpower') AS BIGINT)) AS avg_power
FROM iceberg.v1089.ods_user_event
WHERE event_name = 'adventure'
  AND ...
GROUP BY 1 ORDER BY cnt DESC
```

**解读方向**：
- 发现隐藏高频行为（推币机、冒险等）
- 对比目标活动的操作频次差距
- 找"上瘾点"：某个区间内频次异常高的行为

**大富翁案例**：推币机（~63,000次，62%参与率）远高于大富翁。
对比维度：即时反馈 vs 慢反馈、低门槛 vs 骰子稀缺。

## 第4层：活跃度分群对比

**目的**：排除低活跃玩家的噪音，看核心玩家的行为是否有差异。

**数据源**：`ods_user_login` + `ods_user_event`

**Step 1：筛出活跃玩家**
```sql
SELECT user_id, COUNT(DISTINCT partition_date) AS login_days
FROM iceberg.v1089.ods_user_login
WHERE server_id = '{server_id}'
  AND partition_date BETWEEN '{start}' AND '{end}'
  AND user_id IN ({user_ids})
GROUP BY user_id
HAVING COUNT(DISTINCT partition_date) >= {threshold}  -- 通常 7天/8天
ORDER BY login_days DESC
```

**Step 2：重跑第3层 SQL，替换 user_id 为活跃玩家子集**

**解读方向**：
- 活跃玩家行为 vs 全体的差异幅度
- 若差异均匀（全面高15~20%）→ 活跃玩家只是"更勤快"，不是结构性差异
- 若某类行为差异特别大 → 活跃玩家有特定偏好，可作为设计参考

**大富翁案例**：活跃37人 vs 全体47人，所有行为均匀高15~20%，属全面活跃。
说明问题不是某类玩家的偏好差异，而是活动本身设计问题。

## 补充：对标分析框架（找竞品行为）

当目标活动参与度明显低于某个同期活动时，做1v1对标：

| 对比维度 | 目标活动 | 竞品活动 |
|---------|---------|---------|
| 进入门槛 | ? | ? |
| 确定性 | 高/低 | 高/低 |
| 反馈速度 | 即时/慢 | 即时/慢 |
| 正反馈循环 | 有/无 | 有/无 |
| 操作频次（人均/天）| ? | ? |

**大富翁 vs 推币机案例**：推币机在每个维度都优于大富翁，解释了参与度差距。

## 涉及表汇总

| 表名 | 用途 | 粒度 |
|------|------|------|
| `dl_user_asset_reason_detail_h` | 按reason_id聚合的资产变动汇总 | 小时级 |
| `ods_user_asset` | 资产变动明细（含reason_sub_id） | 事件级 |
| `ods_user_event` | 客户端行为事件（counter级） | 事件级 |
| `ods_user_login` | 登录记录 | 日级 |
| `dim_asset` | 道具维度表（asset_id→名称）| 维度 |
| `dim_reason` | 原因维度表（reason_id→名称）| 维度 |

所有表均在 `iceberg.v1089`（X2），X3用 `iceberg.v1090`。

## 优化方向输出模板

分析结束后，对照以下维度输出建议：

1. **降低门槛**：自动产出资源 vs 手动获取
2. **加速反馈**：即时奖励弹出 vs 等到完成再给
3. **对接核心需求**：奖励是否是玩家第1~3渴望资源
4. **制造"差一点感"**：显示下一个大奖是什么
5. **嵌入主线循环**：活动 → 主线产出 → 回流活动，形成闭环
6. **频次机制**：可反复触发（参考推币机）vs 骰子限量
