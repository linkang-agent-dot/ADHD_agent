# 交互原型素材化工作流（让原型 ≈ 真游戏 8-9 成）

> 目标：交互原型（活原型+实时说明一体 HTML）做到**跟上线游戏 8-9 成相似**，让 design-merit 的「表现」支能对着真图评、phase3 mockup 少返工。
> ⚠️ **工作流与产物全在 KB，零 `.designdeck` 依赖**。`.designdeck` 仅作上游参考（其 E:\ 路径已失效，别引）。真素材源 = `C:\x3-project\client\Assets\Res\UI\Spirits`（6034 张）。

## ★★正确流水线：保真靠「效果图」整体渲染，不靠 HTML 堆素材（2026-06-05 实证）
```
①线框/结构 → ②效果图(AI整体渲染) → ③交互原型(拿②当静态底,HTML只叠会动的件) → ④拆分
                ★视觉保真在这步                  ↑只负责"动":角色/骰子/进度条/弹窗
```
- **关键认知**：交互原型用 CSS 手拼离散 PNG **有天花板**（拼不出有机路径/景深/叠压/统一光照），保真上不去；真正的"像真游戏"是**②效果图一张整体渲出来的**。别让 HTML 既当画家又当程序。
- ②效果图做法：**Morphix「线框转效果图/主题转换」prompt × 真游戏截图当布局+风格参照**，过 GRFal `generate_image`。实证：深海大富翁棋盘 1 张渲到 ~9 成（`KB\产出-交互原型\X3_2026深海节\03_效果图\board_mockup_v1.png`，参照真截图 `复用参考图\航海大富翁复用\Snipaste_15-43-39`）。
- ⚠️**③交互原型的正解(2026-06-05 实测纠错，推翻"效果图当底")**：
  - **错法**：把②效果图(整张烤好的图)当底、上面飘"会动的件"。实测更差——照片把玩法元素(格子/角色/进度)全冻死，浮层坐标只能瞎猜、对不齐，成了"好看的图+乱飘的鬼火"，**比纯 CSS 活原型还差**。
  - **正解**：**活 DOM + 真单体美术素材(PNG)，位置由代码构造摆放 → 天生对齐**。格子=真海岛 PNG、罗盘/角色/宝箱=真 PNG，全是活 DOM，角色走到的就是我摆的坐标(零猜)。三轴(交互/布局/素材)全"活"，没有一块是烤死的照片。范例 `大富翁_..._真菱形版.html`。
  - **②效果图的真正用途**：(a) design-merit「表现」支的**视觉目标/参照**(渲出来 vs 真截图比像不像)；(b) 只取**纯背景层**(海水/珊瑚等不含玩法元素的部分)当静态底——玩法元素绝不烤进去。
  - **元教训**：没经实测的"看起来对"的结论别当定论写死(我连续两轮把"效果图当底"当正解，被实测推翻)。
- gdesign 工作流本就编排了这步(`02_structure→03_mockups→04_split`)，别跳过 03(03 效果图=目标/参照，不是交互原型的底)。

### GRFal 跑效果图操作坑（call_grfal.py）
`generate_image` 不在脚本 async 白名单、同步路径被服务端拒(`long_running`) → 必须：`--submit-only` 拿 task_id → `--check-task <id>` 轮询(gpt-image 带参照约 300-500s) → 结果是 `/api/output/...` 相对路径，用 `curl -H "Cookie: $GRFAL_COOKIE" https://grfal.tap4fun.com<路径>` 下载。params 文件用 C:\ 绝对路径(bash /tmp 与 Win-python /tmp 不同)。

## ★★★最硬的教训：自洽 > 真（2026-06-05 连续踩坑后定论）
**整体硬塞一堆「近似/不对应」的真素材，比干净自洽的占位(emoji/CSS)还差** —— 一堆不同源、比例光照不统一的真 PNG 凑一起 = "直接生成的 collage 感"，读起来不如风格统一、组件清楚的 v1。
- **第一版(自洽占位)往往就是交互原型该有的样子**：它的活是讲清交互，不是当美术成品。真实视觉交给②效果图(静态目标图)。
- **往真走的唯一正确姿势 = 组件级、逐个换、精确对应、自洽为闸**：
  1. 一次只换一个组件，换成"这玩法**确切该用的那张**真资产"(不是手边近似的)；
  2. 每换一个**验整体自洽**(比例/光照/风格搭不搭)；
  3. 搭→留；不搭→先把素材处理到搭(裁剪/统一尺寸/调色)，否则**保留干净占位**——**自洽优先于真**；
  4. 拿②效果图当"每个组件该长啥样"的对照标尺，逐个逼近。
- **别再做的事**：一把梭把整屏换成"真素材活DOM"(会变 collage)、或把效果图当交互底叠浮层(对不齐)。这俩 2026-06-05 都实测翻车。

## ★保真度 = 三根轴，别把"换皮"当"保真"（2026-06-05 踩坑沉淀）
| 轴 | 是什么 | 升级动作 |
|---|---|---|
| **①交互/逻辑** | 状态机、动线、动画时序 | 原型版通常已有 |
| **②布局/构成（="质"）** | 屏幕几何：元素**真实位置、比例、路径形状、层叠** | **对照真游戏截图/prefab 重建布局** |
| **③素材/皮** | 填充纹理（emoji vs 真 PNG） | 换真素材 |

**最易犯的错：只升轴③ = "换皮不换质"。** 在简化线框骨架上贴真图，出来只是"漂亮的线框"，布局还是简化的（如把环形赛道拍成 4 列网格）。**真保真必须先换轴②**——拿真游戏那一屏的实际布局当参照，重建构成，再填素材。
- 实例：大富翁原型把"中央罗盘+外圈环形赛道"做成了 4 列网格；对照真截图 `复用参考图\航海大富翁复用\Snipaste_*` 重建成环道后才算换质（`大富翁_..._真布局版.html`；`pos%N` 循环逻辑天然适配环形，零改）。
- **动手前先问**：有没有真游戏截图/prefab？没有先去 KB `复用参考图\` 找；找不到再按真美术底图(board_bg)的构图近似重建。

## 核心病根
交互原型默认用 **emoji + CSS 渐变**糊视觉（🧜‍♀️角色 / 🦪道具 / CSS 方块当地块）、且**布局是线框级简化**（网格 vs 真实环道/路径），看着"代码味"不是"游戏味" → 保真度只 2-3 成。

## 三步法
1. **真素材优先**：能在 6034 库里找到对应件的，**一律引真 PNG**，禁 CSS 模拟核心控件（button / frame / 地块 / 角色 / 宝箱 / 关闭键）。
2. **缺的生成补齐**：库里没有的，用 **Morphix prompt × gdesign token**（见 [[reference_morphix_reskin_prompts]]）按 X3 风格生成 → 丢进原型本地 `_assets/` → 引用。
3. **KB 自包含**：素材**拷进**原型同目录 `_assets/`（从 C:\x3-project 拷，不引绝对路径、不引 .designdeck），tokens 内联或本地 css。原型可独立打开渲染。
