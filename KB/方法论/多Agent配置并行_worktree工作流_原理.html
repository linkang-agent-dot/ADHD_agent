多 Agent 同机改配置：git worktree 工作流

一台机器、多个窗口、同一个配置仓 —— 怎么不撞车
2026-06-24 · 配套 memory workflow_x3_multiagent_worktree · skill x3-config-export 步骤0

1问题：共享一个工作目录

多个 Claude 窗口在同一台机器跑，改的是同一个仓 C:\x3\gdconfig。git 的铁律是：一个工作目录同一时刻只能停在一个分支上。

撞车 窗口 A 想在 dev_festival 改，窗口 B 一句 git checkout dev，整个目录的文件被换成 dev 的版本 —— A 正在编辑的内容瞬间被掀翻。今天上午实测：主仓在 dev↔dev_festival 之间来回切了 5 次以上。

土办法（各自备份到文件夹、手动合）治标不治本：状态只存在人脑里，会忘记合并。

2核心原理：git worktree

worktree 让同一个仓库长出多个独立工作目录，每个停在自己的分支，共享同一份 .git（不重复 clone、不占多份历史）。改动互不可见。

.git （唯一，共享对象库）
↑ 三个目录共用 ↑
gdconfig/
dev_festival
主目录（你自己）
gdconfig-bpfix/
feature/bpfix
窗口 A 隔离区
gdconfig-wt-hud/
deepsea-hud
窗口 B 隔离区

各停各的分支 · 文件物理隔离 · 谁切分支都不影响别人

# 给一个窗口开隔离区
git worktree add ../gdconfig-bpfix -b feature/bpfix dev_festival
# 干完拆掉
git worktree remove ../gdconfig-bpfix --force
git worktree list   # 谁还没收尾，一查就知道（取代"人脑记忆"）

3判断：什么时候才开 worktree

不是每次都开（开 worktree 有 ~2 分钟后台重建 xlsx 的成本）。判据很简单 —— 看主仓当前是否被别人占着没收尾：

cd C:\x3\gdconfig ; git status -sb
干净 · 无 [ahead]

没人占 → 直接在主仓干。第一个进来的人留主仓。

有改动 / [ahead N]

别人在改没收尾 → 开 worktree 隔离。后到的人自动分流。

这道判断写在 skill x3-config-export 的「步骤0」，改配置触发 skill 时第一件事就是它。

4兜底：PreToolUse Hook

skill 步骤0 靠 LLM 自觉；为防漏判，加一道硬兜底——一个 hook 在「动手碰主仓配置表那一刻」自动拦截。

触发面：Edit/Write 主仓 .tsv/.xlsx，或 Bash 跑 tsv_edit.py 没指向 worktree。
判断：主仓干净 → 放行 + 落 session 标记（本会话认领主仓，之后自己改不再打扰）；主仓脏/ahead → exit 2 拦下，提示开 worktree。
放行例外：改 worktree 里的、跑别的命令、合并冲突解决中（MERGE_HEAD 存在） —— 一律放行，不误伤。
fail-open：hook 自己出任何错都放行，绝不卡住正常工作。

关键认知 hook 在会话启动那一刻从 settings.json 加载。上线前就开着的老窗口装不上 —— 要兜底生效，得重启或新开窗口。

5合并回：本地直合（不走 MR）

dev_festival 是工作分支（不受保护）、gdconfig 仓也不受保护 → 本地 merge + 直接 push，不开 MR（MR 只为进受保护的 dev）。

方向铁律 把 dev_festival 合进 feature（在 worktree 里解冲突），再让 dev_festival 干净 ff 上来。 冲突解决全程在隔离区，主仓只做 ff，不产生冲突编辑。

前置（一次性）：确认 merge driver 注册 —— git config --get merge.tsv3way.driver 非空。头号坑没注册就 merge = tsv 炸 <<<< 标记 + xlsx 二进制选边静默丢改动。
串行化：同一刻只许一个窗口往 dev_festival 整合（整合=改共享分支=临界区）。并行编辑随意，整合要排队。
worktree 内合：git merge --no-commit --no-ff origin/dev_festival，按审计 KB 解冲突。
验收四连：sync_xlsx_tsv.py --check=0 → merge_tsv_audit.py 双向 lost=0 → ExportTable.py exit 0（前三过≠能导表，这才是硬门）→ commit。
ff 回去 + 推：主仓 git merge --ff-only feature/<名> → git push origin dev_festival → jolt_verify.py dev_festival skip_check=true。ff 失败=整合期被别人推进了→重合再来。
收尾：git worktree remove + git branch -D。

6四个真实踩坑

坑后果 & 解法
merge driver 没注册不随 clone，在本机 .git/config。没注册 merge 静默丢改动 → 先 install_hooks.py。
审计看了本地分支本地 dev_festival 常滞后 origin（这套流程 push 到远端、别窗口不 pull）。判"合回没合回"必须看 origin/dev_festival，看本地会误判"工作丢失"。
tsv_edit.py 默认写主仓默认 --repo C:\x3\gdconfig。worktree 里干活每条命令必带 --repo <worktree>，漏了静默写回主仓，worktree 白开。
建 worktree 后秒拆失败post-checkout 钩子后台重建 xlsx(~2min)锁住新目录。真干活不受影响，只"建完立刻拆"会撞 Device busy。

7实战验证（2026-06-24）

首个完整成功案例 · bpfix-monopoly-bp

真人在另一个窗口独立跑通全链路：开 worktree(15:19) → 提交(15:28) → 把 dev_festival 合进 feature(15:29) → push origin → 拆 worktree+删分支。merge driver 已注册、合并方向完全正确、work 安全落在 origin/dev_festival。

同一时段 gdconfig-bpfix 与 gdconfig-wt-hud 两个 worktree 并行，互不干扰 —— 满血多 agent 场景。从工作流搭好到被真实采用，几个小时。

8推广到客户端仓（代码 / 资源）

同样多窗口改 x3-project（C:\x3-project\，代码+资源）也撞共享 checkout —— 隔离概念照搬，但合并回和成本跟配置仓反着来，照搬必翻车。

合并回看目标分支（你=Developer：非保护分支全权，保护 dev 只能开 MR 不能点合并）

· 合进 dev_festival（非保护，日常）：自助、不用 MR，push feature → 整合 → push dev_festival。
· dev_festival → dev（保护，发版）：才需 MR，你开单、大哥(Maintainer)合，title/desc 纯 ASCII。

本地禁跨分叉 merge 不论目标，x3-project 分叉分支本地 merge 会炸二进制 ProtoGen 冲突 + LFS pointer + git clean 误删合法 Unity 目录。本地只做干净 ff，跨分叉合并交给远端/MR/大哥。大文件先 git lfs push。

gdconfig（配置）x3-project（代码/资源）
dev 保护否，可直推是，必 MR
worktree 成本极低（~2080文件）高（多GB+LFS）
worktree 用法随建随拆长寿 / 按feature / sparse-checkout
合并回本地 merge+ff+push dev_festival进 dev_festival 自助(无MR)；进 dev 才 MR(你开·大哥合)；本地禁跨分叉 merge
串行化本地整合需排队MR 服务端天然排队

只改代码不碰资源时用 git worktree add --no-checkout + git sparse-checkout set server/ 只铺子树，省几 GB。提交规则不变：X3NEW- 前缀 · 内嵌 gdconfig 子模块须干净 · 大文件先 lfs push。
