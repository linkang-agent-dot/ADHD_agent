🦸 X3 英雄皮肤生产工作流

英雄皮肤从「概念稿 → Spine 资产 → 展示视频」的全链路打法。重头戏是用 AI 视频替代 Spine 做皮肤展示（idle 循环 + 入场片），含模型选型、透明化流水、演出/prompt 心法、踩坑。是 《X3 美术出图工作流总览》 的皮肤专项深挖。
概念稿 gpt 展示视频 seedance SBS 透明 1080×1920 更新 2026-06-24
为什么有这条线：英雄皮肤核心是 Spine 2D 骨骼动画立绘，AI 做不了 Spine。但皮肤的展示（详情/预览/抽卡/获得/礼包里那段循环立绘）可以用视频替代——AI 量产视频比美术逐个绑 Spine 便宜得多。于是分工：老皮肤 Spine 不变，新皮肤展示走视频。

1 皮肤的三种资产 + AI 落地度

资产是什么路径AI 落地度
① 换装概念稿给美术对齐方向的静态原画归档 KB✅ AI 端到端出概念
② Spine 立绘资产骨骼动画 .skel.bytes+atlas+SkeletonData+prefabRes\Spine\Prefabs_Download\Role_Spine_<id>_Skin<n>.prefab❌ 美术在 Spine 软件做
③ 展示视频替代 Spine 展示的 SBS 透明循环视频client\Assets\Res\Video\VideoRes\✅ AI 量产（本文重点）

GRFal ✗ Spine 实测 0 命中：sprite_sheet=逐帧序列帧(和 Spine 骨骼两套体系不兼容)、generate_motion=3D 人形(非 2D Spine)、generate_video=MP4(非可换肤循环立绘)。AI 能帮的只有上游：出静态全身原画 + image_layered 辅助拆件。

2 线 A：换装概念稿（gpt）

  1. 选基础英雄 → 找客户端立绘 Role/FullLength/Role_F_{id}.png。
  2. 本人立绘当 reference，prompt =「保身份(脸/发/身形/画风)，只换造型(服饰/道具/姿势)」→ 出 R1。
  3. 后续每轮用上一版选中图当 ref，逐轮只改一处，版本名带语义后缀。
  4. 用户给的「质量好但带违规元素」的图：以它当质量/姿势锚 ref，prompt 逐条列要改的设计点（红→白金 / 国徽→奖杯 / 去脸颊国旗）——既得质量又合规，FINAL 常这么来。

坑：基底选错会白跑，动手前确认 ref 是哪一版；IP 合规用原创风格化不精确复刻商标。这张概念稿同时是展示视频的起点图（见 §3 心法第 1 条）。

3 线 C：皮肤展示视频（替代 Spine，重头戏）

3.1 显示框架与格式真相（先读，决定怎么出）

  • 显示链：HeroSkin.DKVideo → EffectDisplay.VideoDisplayKey → VideoRoot → RawImage，ScaleMode=ScaleToFit（保比例、加黑边、不裁切）。
  • ★ 统一 1080×1920(9:16) 原生直出，不做后期裁切（2026-06-15 客户端拍板）。早期容器是 1024×1536(2:3) 导致 letterbox + 补偿裁切把彩球切出框，已作废。
  • SBS 透明视频：左半=彩色画面，右半=alpha 烧成的白模 on 黑底，并排。Unity shader 从右半重建 alpha（mp4 存不了原生 alpha，SBS 是标准 trick）。格式锚 Res/Video/VideoRes/AllianceCard_icon_1_a.mp4。
  • 不透明 showcase 视频（节日/礼包/解锁展示，非角色透明）：普通满幅 mp4，格式锚 egypt_sphinx.mp4（608×1080/h264/yuv420p/24fps）。分辨率无统一值，按展示框走。
  • 时长基准 4-8s（idle 循环）；入场片方案 ~10s 单条直出。

3.2 模型选型（实测推翻早期结论）

模型结论说明
seedance角色立绘 idle 首选2D 脸稳(嘴不张/不 warp) + 运动 3.2（真 Spine 量级）。代价：fflf 首尾差 1.79，无缝循环需后续治
kling仅用于需严格无缝循环fflf 首尾锁得紧(0.6)，但给 2D 立绘脸必张嘴/warp 且压不动，是模型级硬伤
vidu需 ≤3s 时备选fflf 能更短(1-16s)，默认带音频
veo3 / sora排除渲染慢(>10min)、过杀
happyhorse排除强制音频不可关

3.3 透明化流水（白底主稿 → SBS 游戏格式）

generate_video
(seedance / kling fflf，白底)→ video_remove_background
(.webm 原生 alpha)→ export_sbs_video
(左彩右白模)→ 抽帧验收
  • ★ export_sbs_video 的 quality = CRF（越低越清晰），不是 0-100 百分比！ q12 = 甜点(7MB 肉眼无损，正式用这个)；传 95 会糊成马赛克。15s 片用 q19(~12MB)。
  • ★ export_sbs_video 禁并发——同账号两个 export 会互相覆盖服务端输出路径，下到字节级相同的废文件。永远串行。
  • 无缝循环杀招：kling ref_types="首帧图像,尾帧图像" 同一张主稿传两次 → 首尾强制一致（实测像素差 0.64）。
  • 抠像最大敌人=杂背景/撞色。主稿纯白底+边缘干净已规避大半；银白发丝吃边的保险=背景换纯绿/品红再抠。
  • 验 SBS：左半 BGR std 高(彩色) / 右半三通道差≈0(灰阶白模)；验透明：RGBA alpha 0-255 都有。
  • ★ 最终压缩走 GRFal compress_video，别用本地 ffmpeg 手调参数（2026-06-24 大哥反馈：compress_video 后端参数已调优、效果明显更好；本地 ffmpeg 手调 crf/preset 参数不对、效果差很多）。本地 ffmpeg 只留给 compress_video 覆盖不到的几何操作（SBS 双半 crop+hstack / 裁切 / 缩放 / 去音轨）。

3.4 演出脚本 = 入场表演 + 静态循环 结合（一条直出，循环播）

★ 方案演进（重要）：早期想做纯静态 idle 循环（站着微呼吸+wink），但站桩撩不起来、纯静态太单薄。最终收敛成 一条 ~10s 直出片：前半段「入场表演」+ 落定 + 后半段「静态 idle 循环」，客户端循环播整条（循环点=待机完→再次入场，叙事自洽）。下面分两段看，编舞自由度和「别动太多」的约束在两段是相反的。

看的人：纠结冲不冲榜的付费玩家 + 已拥有反复欣赏的鲸鱼。视频=转化面/炫耀面。钩眼三板斧：①眼神(扫镜头+了然微笑，最狠) ②招牌 beat 挑一个做透 ③身段曲线持续撩。

① 入场表演段（前半，可以动、要有戏）

叙事：拿彩球入场跳舞 → 踩球/颠球(挑起→膝颠→脚背两触) → 兜进左腋 → 落定到主稿姿势 → 表情用整整 1s 连续渐变到 idle 终帧的脸。

  • 这段是转化的钩子，有大动作/技巧/表演张力，不受「别动太多」约束。
  • 表演段措辞从「动态感」起笔；动作写成可数节拍才会被执行。
  • 颠球=最高崩险(球物理+脚接触)，崩了减一触。
② 静态 idle 循环段（后半，别动太多）

落定后接 2-3s 待机循环，要克制自然，动太大循环就假。

  • 必须有：胸随呼吸微微起伏（核心卖点，克制不浮夸）。
  • beat：能做出 wink 就够了，不要转球/抛接/撩发/大幅身段（循环杀手）。
  • 道具(足球)：完全静止（转动破坏循环）。
  • 节奏：基准帧→吸气峰值(胸微起)→中段一次 wink→呼气回落→精确回到基准帧。

表情连续性：先定 idle 循环的脸（如高冷），入场表情弧线倒推——入场全程笑容收弱、落定段连续渐变到循环终帧脸，禁突切。fflf 首=入场版图 / 尾=idle 循环基准帧，保证整条循环无缝交棒。

3.5 ★ 皮肤视频撩人 prompt 心法（跨 12 版总结）

  1. 撩不撩的根在起点图：fflf 锁主稿 → 主稿端正则视频再调也端正。撩人主稿(有张力姿态+眼神)是地基。
  2. 眼神给不给镜头 = 死活线：不给=证件照/糖人；给(暖撩或冷扫都行)=活。
  3. 写「一条贯穿全身的动作线」不写零件清单：重心移→胯→腰→胸→肩→头→眼神，强调整体协调；否则各部位独立动=糖人。
  4. 情绪要细颗粒可导演：别用「性感/高冷」糙词，拆成眼睑/嘴角/节奏/重心的具体微行为。
  5. 压制词(minimal/never/almost still)会叠加压死动作，克制用。idle 段措辞永远从「动态感」起笔，别从「安静」起笔。
  6. prompt 长度 = 注意力预算：350 词长 prompt 里可数节拍照样被丢；瘦身到 200 词、把最重要那段标 THIS IS THE MOST IMPORTANT PART，token 砸给最易被丢的那段。
  7. 动作要可数才会被执行：氛围形容词模型常丢弃，写成「左…右…左…右，约 2s 一周期、共 3 周期、NEVER frozen」。idle 运动量尺子：<3.5=木头，4-7=正常活（帧间平均像素差）。
  8. 氪佬视角验收（色不色/吸引力/购买欲）比技术指标(联动/幅度)更接近真实成败。

3.6 关键迭代教训（少走 12 版弯路）

现象根因 / 解
糖人病（人是死的、各动各的）动作写成零件清单 → 改成「一条贯穿全身的重心线」
太木（运动掉到 0.1）minimal/never/almost still 压制词叠加压死 → 克制用，从动态感起笔
kling 给 2D 脸嘴必张/warp模型级硬伤，措辞绕不过 → 换 seedance
锁嘴锁成呆脸锁「嘴形状」不锁「位置」——嘴形不变但可随头平移；脸部活气来自头微侧倾+眨眼，不是嘴
大动作甩出框缩小角色救不了（是动作够到框外）→ 收敛编舞(彩球只在胸/胯前抖) + 首尾参考帧人物整体缩 90% 让安全边距
时序不准（写 7s 实跑 9.5s）seedance 不遵守秒数 → 压时长=砍动作内容，数字只是参考

3.7 单条直出的工程铁律（§3.4 演出落地时必守）

  • ★ 用户铁律：皮肤视频全部一条直出，不拼接。拼接方案(SegA+SegB+渐变缝)技术终缝 0.0 但缝内光影漂移肉眼仍突兀，被毙两次。入场段+idle 段必须写进同一镜的 Phase 时序里一次出。
  • 程序零改动：单条循环片直接替换原循环 mp4、走现有播放链路（不需要「入场播一次→切循环」的程序支持）。这是选单条直出而非两段切换的关键原因。
  • 时长：seedance 单镜 10s 七阶段时序实测扛得住；不够先探 submit 是否接受更长 duration（15s 可提交），别想当然按 10s 上限切段。
  • seedance 不遵守 prompt 里的秒数 → 压时长=砍动作内容，数字只作参考；验时序=抽目标秒数处的帧看演到哪。

4 调用 & 工具坑

坑解
认证走 ~/.config/grfal-api/token_store.json 的 Bearer(长效到 2036)，什么都不用设；脚本开头 os.environ.pop('GRFAL_COOKIE', None) 防过期 cookie 顶掉好 token→401
generate_video 是 long_running--submit-only 拿 task_id → --check-task 轮询(间隔 20s，video 5-12min)，禁 --sync
cookie/token 失效症状轮询整场空状态(check-task 返 success:false)，轮询必须对 success:false fail-fast；后台跑别加 | tail -N(截掉 task_id 无法续查)
cv2.imwrite 写不了中文路径(静默 False)cv2.imencode('.png',fr)[1].tofile(path)（读 VideoCapture 中文路径反而没事）
aspect_ratio auto 会抽风竖版显式传 '9:16'，auto 可能随机出横屏
最终压缩走 GRFal compress_video（参数已调优，效果优于本地手调 ffmpeg）；本地 ffmpeg.exe 只用于几何操作(crop/缩放/去音轨)，cv2 写不了可控码率不拿来编码
GBK 控制台 print 中文崩脚本开头 sys.stdout=io.TextIOWrapper(...,errors='replace')

验收 HTML 工具：_loop播放.html(切版对比循环，注意只能引 GRFal 原片 H.264) / _透明验证.html(webm 叠棋盘格/绿/深蓝三背景看发丝) / 单文件分享=mp4 base64 内嵌进 HTML。
