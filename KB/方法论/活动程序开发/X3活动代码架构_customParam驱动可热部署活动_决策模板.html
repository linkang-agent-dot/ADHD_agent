customParam 驱动「可热部署活动」

把活动的可变运营参数从「配表 + 热更」搬到「iGame 后台部署参数」——逆向自大哥的世界杯竞猜重构（X3NEW-1432）

范例：世界杯竞猜 ActvType=64 commit f6e2d08 作者 zhangli 76 个活动 · 一次切
活动的可变参数（哪两队对阵）通过 iGame 后台 customParam(JSON) 部署时传入，服务端读它建专属数据结构 —— 换对阵 = 改后台重部署。
✓ 零代码改动✓ 零服务端热更✓ 零客户端打包✓ 运营自助换

1为什么要重构

老套路：换对阵要「改 ActvPack 配表 + 服务端热更」。客户端早已零打包，但运营换对阵还卡在改表+热更，不对称。

🔴 重构前 · 痛点

  • 换对阵要 改配表 + 服务端热更，和客户端零打包不对称
  • 竞猜两队硬塞进"进度礼包"的 day=0 / day=1 hack，语义模糊
  • 76 场比赛 = 76 行 ActvPack 各维护一行，配置冗余
  • 带着"最终奖判定"死代码（本玩法根本不会触发）

🟢 重构后 · 结果

  • 换对阵 = iGame 后台填两个数字重新部署，运营自助
  • 专属 proto WorldCupGuessData{packA,packB}，语义清晰
  • 旧 76 行配表绕开不读，参数挪到部署参数
  • 新结构不含最终奖，运营用 BI 筛名单统一邮件发奖

4九条代码决策

这才是模板的精华 —— 每条都带「为什么」和「被否决方案」，加同类活动时逐条照判。

D1

专属 Condition 入口，不动通用流程

新玩法走自己的 Condition，通用进度礼包链路物理保留。

收益：回滚 = rollback MR，旧链路立即可用
D2

customParam payload 取最薄

只传数据本质（两个 packId）。

❌ 传国家 code / matchupId（刻死命名 / 回到改配表）
D3

新建独立 DTO，不复用现成的

建 WorldCupGuessCustomParam，字段干净。

❌ 塞进 KvK 的 DTO（带无关 mapServerID）
D4

proto 新 tag 顺序追加

先查当前最大 tag 再追加（竞猜误判 36 空闲，实占 → 改 62）。

❌ 复用 / 改旧 tag = 全链路兼容灾难
D5

字段名贴数据本质

叫 packA/packB 不叫 teamA/teamB。服务端只认 packId，业务语义留 UI。

D6

反序列化 + 校验放 Condition 层

Condition 自己 Deserialize + CPack 校验。

❌ 放共享 deploy handler（它不该懂每种格式）
D7

原始 JSON 持久化到 entity Data

写 server entity，不放 proto（客户端不需看原始 JSON）→ OnLogin/OnReload 可复读。

D8

复用底层原语，挂专属结构

gift 创建复用 CreateActivityPack，数据挂专属 WorldCupGuessData。

复用机制 + 专属语义
D9

配套 GM 命令本地复现

GMDeployWorldCupGuess 走同一条 CreateNewServerActivity，无需 iGame 即可端到端验收。

6陷阱 & 回滚

proto tag 别信"空闲"假设
必须先查最大 tag 再追加。竞猜踩过 36→实占→改 62。
客户端版本必须同步上线
新 tag 旧客户端忽略 → UI 拿不到数据显示空。VersionControl 版本号联动。
iGame customParam 错填
填错 packId 靠 CPack 校验在 deploy 时回滚，运营收失败回执。保留校验。
旧配表保留不读有误用风险
76 行 ActvPack 还在但已绕开，建议加备注"已被 customParam 替代，勿改"。
🔄 回滚成本极低
rollback MR → 旧 progressPackData 链路（tag 35）完整保留立即可用；已部署活动需 iGame 重新 deploy 一次重建数据。

7复盘：我们当时为什么没做到

这套设计不是大哥才想得到——第一版（含本 AI）把一条约束抬成了死墙，于是绕过去而不是穿过去。

🧱 当时自设的硬墙

"新增 proto 字段 = 走客户端打包（proto 是客户端 bundle、热更只服务端）= 不可接受"

→ 在"绝不能动客户端"前提下，复用 progressPackData 的 day=0/day=1 hack 是局部最优。不是 proto 技术卡住——proto 改动是机械活，生成工具自动出 250 行。

🚪 大哥怎么穿过去的

没把"新增 proto"当死墙，识别出它只是一次性成本。

→ 付一次客户端打包上线专属 proto，换来永久干净架构 + 未来对阵切换照样零打包。一次成本买永久收益。

1
分清"永久红线"还是"一次性成本"
遇到"绝不能动 X"，先问：动 X 是每次都要付，还是付一次就翻篇？一次成本常比永久 hack 债划算。
2
警惕红线范围蔓延
合理目标"换对阵时零打包"被无意识扩成绝对禁令"任何时候不准动客户端"，然后为守自抬的禁令接受 hack。定期回看红线有没有被悄悄放大。
3
hack ≠ 错，长期当正解才错
没发版窗口时复用 progressPackData 是合理的"先上线"动作（第一版客户端改动还是大哥重构的前置台阶）。但要留 TODO：有窗口了做对。
