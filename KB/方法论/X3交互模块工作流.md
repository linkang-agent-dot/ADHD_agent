# X3 交互模块工作流

> 「交互模块」= 一个功能下**有交互的子功能**（商店/弹窗/排行/结算…，好坏需脱离实物才看得出的那类）。
> 触发「做交互原型 / 交互模块 / 做个界面 / 界面交互 / 交互方案」先读本条。
> **范本**：`范本_交互模块_限时商店一体原型.html`（同目录，点开即懂目标形态）。

---

## 一、交付物 = 一份「活原型 + 实时说明」一体 HTML
**不是两份文件对照，是同一页**：左边能点的原型，右边实时跟着你的操作讲解——**操作即说明**。
它是策划案「界面交互」那段的**可交互真源**：先用它跟人对齐交互，确认后再把内容落成策划案文字。

> 为什么一体：分两份（原型 + 说明文档）要左右脑补对照，体验断裂；合一后点哪讲哪，确认效率最高。这是和用户反复打磨后定的最终形态。

## 二、怎么产（顺序：**原型先行**，不是先写说明）
1. **可点原型先行**：先搭能点的最简交互原型。X3 配色 CSS 近似即可（**验交互、不验美术**，真实 PNG 落地再换）；跑通主动线 + 各状态 + 分支。
2. **验原型**：自己点一遍，**靠点暴露状态模型/分支问题**（例：「售罄 vs 已购买该不该合并」就是点出来的 → 选 A 纯个人限购 → 合并为唯一终态）。待定项让用户拍板。
3. **说明 1:1 对应**：原型确认后，**照原型行为倒推**交互说明（状态机 / 操作动线 / 边界异常 / 🟡待定），不多写不少写。标 **【已演示】**（原型点得到）vs **【规则·待补】**（设计约定、原型没做动作）。
4. **合成一体 HTML**：原型 + 实时说明合到一页，每个交互事件联动右侧说明。
5. **确认 → 落策划案**：确认后把「包装 + 全元素表 + 交互说明」填进策划案「界面交互模块」。

## 三、一体 HTML 结构规范（照范本）
- **左：手机原型**（1080×1920 比例、X3 配色、真能点）。
- **右：实时说明**，四块 +
  1. **当前焦点**：大字「你刚做了啥 + 对应哪条规则」（标 `操作动线`/`状态机`/`边界` 来源）
  2. **状态机条**：高亮当前状态，修饰（折扣/限购剩余）亮灭
  3. **操作动线**：节点串，高亮当前走的路径（拦截类标红）
  4. **交互轨迹**：每步留一行
  - 折叠区：边界/异常/空态（规则·未演示）+ 🟡待定
- **联动规则**：原型每个操作都过一个统一入口 `ev(type,arg)` → 同时刷新 焦点 + 轨迹 + 状态高亮 + 动线高亮。（实现照范本）

## 四、知识库调用（产原型/说明时）
| 时机 | 必读 | 按需 |
|---|---|---|
| 搭原型 / 写说明 | 交互模式库（挑现成范式，别重造）、X3 设计系统（配色/状态色） | 节日活动形式图谱（节日玩法时） |
| 落策划案 | reference_x3_config（每元素读哪个配置）、reference_x3_i18n（Text key 命名） | 美术链路（voyage/pack 等，涉及时） |
> 规则：**先模式后原创**（模式库有就照搬）；**真源对齐**（配置对 tsv、文案按 i18n）。

## 五、沉淀 / 迭代 / 反馈（跑完一个模块）
- **沉淀**（各回各家）：新交互范式 → 交互模式库；没拦住的坑 → design-doc-checklist；好样本 → 本范本库；新配置/美术链路 → `reference_x3_*`。
- **迭代 = 验收飞轮**：被坑 → 回填 checklist 一条规则 → 下次同类自动被拦（如「删除链」那条）。
- **反馈两条道**：对内（自己能解决）→ memory/checklist/模式库，**当场做**；对外（要 GDesign 改产品）→ `KB/GDesign学习.md`【待反馈】暂存。
- **元规则**：发现新规律 **当场更新 KB，不攒、不等提醒**。
- **确认卡口**：加知识/模式/范本=当场自动；**改 checklist 的 block 规则=先报用户确认**。

---
**一句话**：交互模块 = 先搭能点的原型 → 点出问题、定待定 → 照原型 1:1 写说明 → 合成「点哪讲哪」的一体 HTML → 确认后落策划案。范本在同目录。
