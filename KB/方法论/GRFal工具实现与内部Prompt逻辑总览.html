★ 一句话结论

GRFal = 公司内网自建的 AI 媒体能力网关。它自己不训练模型，而是把请求转发给一堆外部商业模型后端（FAL / Azure / 阿里百炼 / 即梦豆包 / ElevenLabs / MiniMax / Tripo / Hunyuan…），再做本地后处理。 Prompt 不在客户端，全在服务端工具内部——分三类：① 工具自带的写死 prompt；② LLM 驱动的中间步骤；③ creative_workflow 多步管线。Morphix 那套换皮 prompt 只是应用层，底层还是调 GRFal 的 3 个工具。

A 客户端层（你机器上的部分）

入口脚本 ~/.claude/skills/grfal-api/scripts/call_grfal.py，约 1150 行，纯 urllib 写的 HTTP 客户端，不含任何 AI / prompt 逻辑。

干的四件事

  • 调用转发：本地文件→base64 → POST /api/call（图片同步）或 /api/async/submit+轮询（视频/3D/训练）
  • token 管理：device flow 浏览器授权、自动续期，缓存在 ~/.config/grfal-api/token_store.json
  • 故障回退：域名挂了自动切内网 IP 172.20.90.45:6018
  • URL 补全：服务端返回相对路径 /api/output/… 时按来源补全 host

D Prompt 设计模式（可复用方法论）

从 GRFal 内置 prompt + Morphix 8 段 build*Prompt 提炼出的通用规律，做任何「图像换皮/编辑」prompt 时直接套。

模式 1 · 职责切分（双参考图）

多张参考图各司其职，显式声明每张图只贡献什么：图#1 = 布局/内容/语义来源（只留 WHAT & WHERE）；图#2 = 视觉/design system 来源（HOW IT LOOKS）。换头同理：图1=底+光照，图2=头。

模式 2 · ERASE 铁律（防串味）

凡"只要布局不要视觉"的场景，必须逐项点名要擦掉的视觉特征（背景色/边框/按钮形/字体/调色/进度条颜色…），否则模型会把图#1 的视觉漏出来。光说"换风格"不够，要说"ERASE 哪些"。

模式 3 · 先文本化再批量（图标换皮）

多张要套同一风格时：先 describe_media 把风格榨成 ~80 词英文文本，再逐张 generate_image 并发套用。避免每张都丢多图参考造成风格漂移。

模式 4 · 灰底而非白底（拆图/抠图）

出待抠图的雪碧图用 #B8B8B8 灰底，不要白底——抗锯齿边缘与灰融合，去底后不留白边光晕。

模式 5 · 语义翻译（线框→实渲）

把图里的中文标注/占位框翻译成实际渲染内容（"篮子、紫宝石、900" → 画出装满紫宝石的篮子+数量 900），并擦掉线框线/×标记/草图感。

模式 6 · 放飞分级（节日换皮 vs 严格换皮）

节日主题转换是唯一鼓励放飞的：可重排布局+加氛围元素+角色重塑+背景光影戏剧化，但文字逐字保留、区域语义角色不变。普通换皮则严格保布局。做节日换皮抄主题转换那段。

模式 7 · 参考图能说明的别在 prompt 里说

配方原则：stylePrompt 只写参考图传达不出的信息（量级/分布/数量约束），不描述具体颜色/场景——那些交给参考图。过度描述反而和参考图打架。
踩坑黑名单（来自 Morphix templates.js 注释）：X3 的 "painterly"/"笔触" 永久 BAN（涂抹感罪魁）；"L 形角包"单独点名会让所有装饰退化成 L 角包 → 要列举多种金色装饰让模型按 reference 灵活挑。

E Morphix —— GRFal 应用层换皮 prompt 库（全文内嵌）

Morphix（demo.tap4fun.com/morphix-demo_0e8f）是个独立换皮网页，本质是 GRFal 的一层「前处理」：没有任何独有能力，底层只调 GRFal 的 3 个工具（generate_image / describe_media / remove_background），调用契约跟 call_grfal.py 完全一致。它的全部价值 = 精调的 prompt + 多步编排。8 段 build*Prompt 全文已整段内嵌在本页签，直接复用别重写。

A · 换皮（核心）

调用

reference_images = [原图, 风格图]
prompt = buildReskinPrompt(用户附加)
model = gpt(默认) / gemini / seedream
num_images = 1~4 · aspect_ratio = 由原图自动推

精髓

图#1 = LAYOUT & CONTENT，只抽布局/文字/语义/控件位置。
图#1 一切视觉特征必须全 ERASE（背景/边框/按钮形/字体/调色，连进度条开关颜色都不留）。
图#2 = 完整 design system，整套照搬。
buildReskinPrompt 全文
I am providing TWO reference images. This is a complete visual style transfer task.

Reference image #1 = LAYOUT & CONTENT SOURCE.
From image #1, extract ONLY these things:
- The structural arrangement of the screen (where each element sits, the overall composition)
- The text labels and what they say (titles, section names, button text, value labels)
- What each icon/avatar represents semantically (an arrow, a coin, a monster portrait, etc.)
- The positions of every control (sliders, checkboxes, toggles, buttons)
DISCARD EVERYTHING ELSE about image #1: its background color, panel color, border style, ornaments, button shapes, button colors, slider colors, checkbox style, toggle style, frame decorations, font choice, font color, color palette, icon rendering style — every visual aspect of image #1 must be ERASED. Only the WHAT and WHERE survives, never the HOW IT LOOKS.

Reference image #2 = COMPLETE VISUAL STYLE SOURCE.
From image #2, copy EVERY visual aspect onto the output:
- Background and panel color/texture
- Border style, frame decorations, corner ornaments, divider style
- Section heading typography (font weight, color, size, decorative ornaments around the heading)
- Button rendering: shape, corner radius, gradient/solid, depth/shadow, outline, inner highlights
- Slider track and knob style / Checkbox style / Toggle/switch style
- Icon rendering style (line weight, fill, color)
- Sub-panel rendering (inner shadow, highlight rim, separator style)
- Color palette as a whole (every color must come from image #2 or be derived from its palette)
- Font family/weight feel
Treat image #2 as the design system.

Goal: The output should look 100% like a design made by the artist of image #2, redrawn from scratch in their style, but containing the layout and content of image #1.

Output: clean mobile UI mockup, vertical portrait, sharp font and icon rendering, high resolution.

B · 线框转效果图

最值得抄：把线框里的中文标注翻译成实际渲染内容（"篮子、紫宝石、900" → 装满紫宝石的篮子+数量 900），擦掉所有线框线/占位框/×标记/草图感。
buildWireframePrompt 全文
This is a wireframe-to-mockup conversion task. Two reference images are provided.

Reference image #1 = WIREFRAME (low-fidelity sketch with text annotations).
It contains:
- Placeholder boxes (often with × cross marks or empty rectangles) representing where finished UI elements will go
- Text annotations (mostly Chinese) describing what each placeholder should contain
- Layout structure showing positions and proportions of all elements

Your job for image #1: extract ONLY the layout structure and the meaning of each annotation. Replace EVERY placeholder box, every × marked rectangle, and every wireframe element with a fully-rendered illustration or UI component matching what its annotation describes. Discard the wireframe lines, placeholder boxes, and the rough sketch quality entirely. Translate Chinese annotations into actual rendered content (e.g. "篮子、紫色宝石、900" → render a beautiful basket overflowing with purple gems with a "900" quantity label).

Reference image #2 = COMPLETE VISUAL STYLE TARGET.
Apply EVERY visual aspect from image #2 to the output: overall art style, color palette, panel/frame design, button rendering, icon style, typography feel, background atmosphere, any characters/mascots.

Goal: a polished, production-ready high-fidelity UI mockup that (1) matches the wireframe layout exactly, (2) replaces every placeholder/× box with a rendered illustration matching its annotation, (3) uses image #2's complete visual language.
The output must NOT look like a wireframe.

Output: clean game UI mockup, sharp rendering, high resolution.

C · 图标换皮（两步流水，最多 10 个）

设计要点：先文本化风格再批量套用，避免每张丢多图参考造成混乱；prompt 强制透明背景(PNG alpha)+居中+无文字，透明由 prompt 驱动不二次抠图。
Step1 风格提取 prompt（describe_media）全文
Describe the VISUAL ART STYLE of this image in detail. Focus ONLY on style, NOT on what objects are shown. Cover: rendering technique (3D/2D/hand-drawn/painted/pixel art/etc.), exact materials and textures (crystal/glass/metal/clay/paper/etc.), color palette and saturation, lighting style and reflections, glossiness or translucency level, edge/outline treatment, level of detail and simplification, mood. Output a single English paragraph of about 80 words, written as a reusable art-style instruction. Do not mention any specific subjects.
Step2 buildIconReskinPromptSingle 全文（generate_image · Promise.all 并发）
Redraw this icon in a NEW visual style described below. Keep the icon's subject identity (its silhouette and what it depicts must remain clearly recognizable), but COMPLETELY REPLACE all its visual treatment — colors, materials, surface finish, rendering technique, lighting, edges, level of detail — with the target style.

TARGET STYLE TO APPLY:
[styleDescription]

CRITICAL: Discard the icon's original art style entirely. The output must look fundamentally different from the input in terms of materials and rendering, while still depicting the same subject. Imagine the target style's artist drew this subject from scratch.

Background: fully transparent (PNG alpha channel). Center the icon. No text, no labels, no extra decorations.

D · 横竖转换（prompt 工程最重）

轴向翻转(最核心)：竖排奖励阶梯/进度条 → 横排，每个档位从纵向堆叠重排成横向并列。禁止"只拉宽不翻轴"。
滚动列表智慧："只画放得下的，其余隐含可滚动"——12 档竖阶梯转横屏只显示 5-7 档。
buildFlipPrompt 全文（竖→横分支；横→竖为方向词镜像）
GAME UI LAYOUT REORIENTATION — redesign a game UI for a new aspect ratio, FOLLOWING the conventions of professional game UI for the target orientation.

Reference image #1 = a game UI screen currently in PORTRAIT (vertical, tall).
Your job: redesign the SAME UI for a LANDSCAPE (horizontal, wide) canvas. This is NOT a stretch, NOT a rotation, NOT a crop. It is a full layout REDESIGN where you act as a senior game UI designer.

LANDSCAPE GAME UI LAYOUT CONVENTIONS (non-negotiable):
• TOP-LEFT: back/close button (←). NEVER at the bottom in landscape.
• TOP-RIGHT: player resources/currencies in a horizontal row.
• TOP CENTER: screen title with subtitle below.
• BOTTOM CENTER: primary action buttons / tab switchers, horizontal, equal spacing, centered.
• MAIN CENTER: the primary content takes the largest area.
• LEFT/RIGHT SIDE: vertical sidebar tabs/nav.
• Decorative banner strips → full-width header bar; MUST NOT float mid-screen.
• 3-column portrait grids → 4-6 columns in landscape.
• Character art → left or right side, NOT centered.

SCROLLABLE LISTS / CAROUSELS / TAB STRIPS — DO NOT cram every item into view.
• Show only what comfortably fits; the rest is implicit (off-screen, scrollable).
• Keep the SAME visual density per item, NOT compressed to fit all N items.

AXIAL / LINEAR COMPONENTS — FLIP THEIR AXIS TO MATCH THE NEW SCREEN'S LONG SIDE.
(progress bar, reward ladder, season pass track, level path, timeline, segmented rail)
• PORTRAIT→LANDSCAPE: a VERTICAL reward ladder must become a HORIZONTAL ladder spanning most width. Each rung rotates from rows-stacked into columns-side-by-side. The active-glow anchor still marks progress along the horizontal axis.
• The rotation is SEMANTIC, not literal — do NOT rotate pixels 90°. Re-lay-out each rung as a proper target-orientation cell.
• A 12-tier vertical ladder need not show all 12 — 5-7 visible tiers with implied scroll is correct.

INTERNAL STEPS: inventory every element incl. axial components → reposition each by convention → reflow grids → flip axis of every linear component → re-design decorative strips → move back button → center action buttons → rebalance whitespace.

STRICT: preserve every structural element; preserve text verbatim; preserve semantic roles.
FORBIDDEN: stretch/squeeze/distort; rotate 90° shortcut; crop/drop elements; keep a vertical ladder vertical when flipping to landscape (the #1 sign of a lazy stretch); back button in non-conventional position; float banners mid-screen; change any wording.

[有风格参考图] Reference image #2 = OPTIONAL VISUAL STYLE ANCHOR. Output = image #1 content + image #2 style + new orientation.
[无] Match image #1's original visual style — only LAYOUT and POSITIONS change.

Output: a polished, professionally laid-out game UI mockup in LANDSCAPE.

buildSplitPrompt 全文（节选核心段）
This is a UI ART ASSET extraction task. The output is a sprite sheet for game artists / developers.

WHAT TO EXTRACT — only GRAPHIC ART ASSETS:
Buttons (no label text) / Icons / Decorative artwork / Panel-frame-background plates (no content inside) / Character portraits / Slider tracks (bar only) / Slider knobs / Checkbox squares / Toggle backgrounds & knobs / Progress fills & tracks / Header bars (no text) / Small badges (shape only, no number).

WHAT TO EXCLUDE: Text strings (NEVER extract text as a sprite) / Number-only elements / Composite widgets (decompose them).

DECOMPOSE COMPOUND WIDGETS into atomic parts:
- A slider with ± buttons → 4 elements: minus button, plus button, empty track, knob.
- A button with icon+text → button plate alone + icon alone.
- A reward tile → frame/background alone + icon/illustration alone (no badge/number).

STRICT DEDUPLICATION: each unique element appears ONCE; rotational/mirror copies → keep one; identical reward tiles → keep one. KEEP separate meaningful variants (checked vs unchecked; different color frames; n/pressed/disabled states).

Layout: tidy rows by category (decorations → panels → buttons → icons → character → sliders/knobs/toggles); ≥20px padding; preserve each element's exact original style; do NOT redraw/recolor.

OUTPUT BACKGROUND — CRITICAL: uniform light gray solid #B8B8B8 (184,184,184) filling the whole image. Do NOT use white/near-white (anti-aliased edges leave white halo after bg removal), off-gray, gradient, cream, paper, checkerboard or any pattern.

Output: a single high-res sprite sheet with ATOMIC, DEDUPLICATED assets on uniform #B8B8B8. No text, no numbers, no compound widgets, no duplicates.

F · 主题转换（最贴节日换皮）

这是唯一鼓励"放飞"的功能——明确写 "NOT a strict 1:1 reskin"，把界面当节日营销海报来重做。做节日换皮最该抄这段。布局可重排 / 大量加氛围元素 / 角色完全重塑 / 背景光影戏剧化，但文字逐字 + 区域语义角色必须保留。
buildThemePrompt 全文
This is a CREATIVE THEME REIMAGINATION task. Re-design a game UI screen as a themed event poster, taking creative liberty to maximize the seasonal vibe.

Reference image #1 = ORIGINAL UI (CONTENT SOURCE).
KEEP from image #1 (MUST survive):
- All text labels, button captions, numbers, level indicators verbatim — do not translate, do not change wording
- The general purpose of the screen and the rough roles of each region (a buy-button stays a buy-button, a reward grid stays a reward grid)
- The semantic meaning of currency/resource icons (visually re-style is OK)

CREATIVE FREEDOM — NOT a strict 1:1 reskin. Should feel like a fresh themed event poster, not a copy with stickers slapped on:
- Composition and layout can be REARRANGED to better showcase the theme (hero can move/pose differently, panels repositioned, props added on edges)
- Add abundant themed atmosphere elements that did NOT exist in image #1: snowflakes, falling petals, lanterns, ribbons, sparkles, fireworks, foliage — be generous
- Re-design the mascot/hero into a themed character (an explorer becomes Santa or a witch — fully reimagined, not just a costume swap)
- Re-design background scenery, lighting, mood dramatically
- Frame/border ornaments can be entirely new (pine garlands, spider webs, plum blossoms, golden dragons)
- Color palette completely overhauled to theme palette
- Treat the screen as a marketing poster for the themed event — high visual impact wins

TARGET THEME: [themeText  或  (no description provided — infer from reference image #2)]

[hasRef] Reference image #2 = THEME VISUAL ANCHOR. Use it for art style, color palette, decorative motifs, mascot vibe. Do NOT use image #2's layout — only its visual theme cues.

Output: rich, atmospheric, polished game UI mockup, sharp font and icon rendering, high resolution.

G · 自由画布 & 局部修改

自由画布：多图按用户在 UI 看到的"图1/图2"顺序进 reference_images，用户中文指令逐字当权威，明确禁止默认"把所有图融合"。
局部修改是短板：Morphix 没用 mask，而是让用户画多边形→把顶点坐标(0-100%)写进 prompt 靠模型自觉"框外像素与原图一致"。我们有真·二值 mask 的 image_mask_edit（服务端合成框外原图），这块能做得比 Morphix 更准。

H · 预制模版系统（buildReskinPayload · 5-pass）

⚠️ 资产缺口先说在前头：模版系统 = 分类参考图库 + stylePrompt。stylePrompt 逻辑我们逆向到了，但那套分类参考图库（X2/X16/X17/P2/X3 每个一组）在 Morphix 服务器上，我们从没下到本地，也不打算去拿（已决策走自建，见下）。所以本节的「编排逻辑」当方法论留存，「图库」按我方素材另起。

逻辑层（已具备，当方法论用）

5 个系统模版每个 = 一组按元素类型分类的参考图 + 一段 stylePrompt。selectRefs 按 5 个 pass 把元素槽映射进 reference_images：

Pass元素槽映射 / 规则
0character 角色（多张拼板）→ image #2；存在时 prompt 顶部强制首令：从角色库挑唯一最匹配的一个还原，禁止多角色融合/平均
0.5scene 场景→ image #3
1screen 整屏 (冲突解决点)有 userRef → 用 userRef 占 #4、模版自带 screen 让位；无 userRef → 用模版 screen
2panel / widget / icon / frame / other 各 1→ image #5..#8
3补充屏幕（仅当无 userRef 时）—
冲突规则一句话：整屏按用户参考图，其余元素按模版库。本质是"用户给了什么就尊重什么，没给的槽才从预置库兜底"——模版系统存在的唯一目的就是兜「用户没给风格参考图」这个场景。
stylePrompt 配方原则（templates.js 注释，自建时照搬）：参考图能说明的就别在 prompt 里说——stylePrompt 只写参考图传达不出的信息（量级/分布/数量约束），不描述具体颜色/场景。踩坑黑名单：X3 "painterly"/"笔触" 永久 BAN（涂抹感罪魁）；"L 形角包"单独点名会让所有装饰退化成 L 角包 → 要列举多种金色装饰让模型按 reference 灵活挑。

资产层（缺口 + 自建路径 · 已决策走 C）

为什么不去拿 Morphix 的库

  • 它在 Morphix 服务器上，本地无；逆向只拿到了 prompt 文本，没下图集。
  • Morphix 预置的 X16/X17 压根不是我们在做的项目，抄它的图对我方无用。
  • 模版库本就该按自己项目建——这不是"少了个现成资源"，是"本来就该自产"。
