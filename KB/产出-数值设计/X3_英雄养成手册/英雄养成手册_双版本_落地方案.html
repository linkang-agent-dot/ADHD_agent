英雄养成手册 · 双版本（基础 + 豪华） 落地方案

2026-06-30 · 范式：新开独立活动，老活动 2701 完全不变 · ActvType=27 二选一互斥 · 飞流程：配置 + 服务端 + 客户端

1. 一句话目标

给「英雄养成手册」加 $49.99 豪华版（现有 $29.99 单档之外）。不改老活动——新开一个独立活动，一上来就带「基础 / 豪华」二选一互斥购买：点购买弹二选一窗（去价格），选一档付费；豪华 = 基础每日奖励 ×2 + 满勤封顶大奖。

2. 三个关键设计决策（为什么这么做）

① 新独立活动，不碰老活动

老活动 ContentID 2701 一行不动。新活动 = 新 ContentID + 新 ActvOnline + 自己独立一套基础 Pack/Group/Reward（购买状态与老活动物理隔离，玩家在老活动买过不会让新活动误判已购）。
表级新增的 Pack2/FinalReward 两列对老活动默认空(=0) → 老活动仍走单档逻辑，向下兼容。

② 豪华日奖 = 基础 ×2「发奖翻倍」

不另建豪华奖励组(Group2)、不克隆 30 行 ×2 的 Reward。服务端发奖时，豪华买家把同一个 RewardID 再发一次 = 数量精确 ×2（种类不变）。
好处：零新增 Reward 行、无「向上取整」歧义（翻倍永远整）、对共享框架改动最小、策划改基础数值豪华自动跟随。

③ 二选一 = 双礼包 + 客户端互斥

活动挂两个礼包：基础 giftId = activityId（沿用旧逻辑）+ 豪华独立 giftId。互斥为客户端表现（买一档→隐藏另一档购买键），与世界杯竞猜(ActvType=71)同款成熟范式。
满勤大奖：新增 FinalReward 列，豪华买家满勤(最后一天)追加发放一次。

对比早期方案（Pack2 + Group2）

早期设计用 Pack2 + 独立 Group2（豪华奖励显式 ×2 配在表里）。本方案改用 Pack2 + FinalReward + 发奖翻倍：少一张奖励组、少 30+ 行配置、框架风险更低。
若策划/审核要「豪华数值显式可见」，可回退到 Group2，服务端 GetReward 改读 Group2 即可（已预留判断点）。

4. 服务端改动详情（已写入分支）

核心思路：单 chokepoint 注入，老活动零影响

ActvType=27 所有发奖路径（单天领 / 一键领 / 自动补发）都经 ActivityPaySignCondition.GetReward()。豪华翻倍 + 满勤大奖只在这一处注入，且全部用 #if _SERVERLOGIC_ + IsLuxuryPurchased() + Pack2>0 三重闸门 → 基础档 / 老活动完全不进新分支。 
// ActivityPaySignCondition.GetReward —— 发奖唯一入口
var rewards = new List<int>(){ cfg.Reward };
#if _SERVERLOGIC_
if (IsLuxuryPurchased(activityId)) {        // 豪华买家
    rewards.Add(cfg.Reward);                // ← 同 RewardID 再发一次 = 数量×2
    if (purchaseCfg.FinalReward > 0 && IsLastSignDay(activityId, cfg.Day))
        rewards.Add(purchaseCfg.FinalReward);// ← 满勤最后一天追加封顶大奖
}
#endif
return rewards;
买礼包→ OnBuyGiftForGiftId → PaySignOnBuyGift→ gift→activity 反查 → SetActivityGiftPurchased(activityId)→ isPurchased=true→ 领奖时 GetReward 判 IsLuxuryPurchased
  • 双礼包创建：OnAddActivity 先建基础(giftId=activityId)，Pack2>0 再建豪华(独立 giftId，IdGenerater)，都挂 activityId 索引；重登/重建按 activityId+cfgID 复用旧 giftId，不丢 purchaseNum。
  • 判档：IsLuxuryPurchased = 找到豪华 giftId 且其购买次数>0。不新增 proto 字段（购买态从 GiftMeta 现成数据推导）。
  • 退款：豪华退款也能清 isPurchased（已修反查），堵「退款后继续领付费奖励」。

⚠️ 二选一服务端硬拦截（买了一档禁止再买另一档）本期走客户端表现（同世界杯，Pack BuyCount=1 限每档各 1 次）。如需服务端强约束，在购买校验处加「另一档已购则拒」——已标为可选硬化项，非阻塞。
