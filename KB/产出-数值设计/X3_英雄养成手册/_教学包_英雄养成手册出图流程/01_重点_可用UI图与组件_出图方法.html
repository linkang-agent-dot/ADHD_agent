怎么产出「可用的 UI 图 / 可用的 UI 组件」

X3 活动界面出图最大卡点的方法教学 · 以养成手册双版本案为范例 · 2026-06-24
这页只讲一件事：流程谁都会列，真正卡人的是 ——「怎么搞出一张能落地、美术/程序能照着干的 UI 图？怎么拿到能直接用的真实 UI 组件？」

答案两句话：
① 可用组件 = 不要画，去游戏 prefab 里把真 sprite 抠出来（GUID 反查）。
② 可用 UI 图 = 真组件先拼个「位置全对但丑」的底，再用 AI reskin 换一层皮。布局靠拼装锁死，观感靠 AI 统一——分两层，别想一步到位。

〇、反面教材：没走这个流程，直接 AI 生图出的"垃圾图"

这几张是养成手册早期没走流程、直接喂 AI 整张生成的——同事先看这个，就懂了为什么必须走后面的方法。

v1_a · 纯 AI 直出
方向对，但底飘：格阵自己重排、主视觉乱画、真实细节全丢
v1_b · 纯 AI 直出
同样飘——AI 不知道游戏真实界面长啥样，纯靠脑补
v2 · 真底 img2img 加页签
半套流程：虽用真底但仍整张重画 → 还是飘 + 双页签方向也被推翻
这三张说明两件事：
① 纯 AI 整张生成（v1_a/v1_b）= 必飘。AI 没有游戏真实界面当锚，会自己重排格阵、换主视觉、丢真实细节，跟游戏里对不上，程序没法照着实现 = 废图。
② 就算喂了真底但还是"让 AI 整张重画"（v2）= 照样飘。关键不在"有没有真底"，而在真底是当"忠实复刻的结构锚"（只换皮）还是当"参考之一"（AI 重画）——后者就退化成 v1。正解见下面第三节第 4 步：图#1 视觉必须全 ERASE、只取布局。

一、为什么"直接 AI 生图"做不出可用 UI 图

路子结果为什么不行
纯 AI 整张生成风格对，但布局飘离真实界面AI 自己重排格阵/换主视觉/丢真实细节，跟游戏里对不上，程序没法照着实现
纯 CSS / 真素材拼装位置全对，但接缝/光影不统一、糙拼出来像缝合怪，不能当成品给美术看
✅ 真组件拼装 → AI reskin位置对 + 观感统一各取所长：拼装负责"位置全对"，AI reskin 负责"成品观感统一无缝"
核心认知（讲给同事一定要点透）：
真素材拼装截图 = 远胜线框的"结构稿（图#1）"。 线框要 AI 去猜内容/比例；真组件实摆的拼装，位置/文字/比例全是确定的，AI 只需要换一层皮，不用猜——所以成品又准又稳。

二、可用 UI 组件怎么来：从 prefab 反查真实 sprite（零 AI）

不要让 AI 画图标/按钮/边框——游戏里全有现成的。把目标界面 prefab 用到的真实 sprite 抠出来，每块都是独立真实组件，天然好拆、保真=prefab 本身。

STEP 1
从 prefab 抽 sprite 的 GUID
grep -oE "m_Sprite: \{fileID: [0-9]+, guid: [a-f0-9]{32}" <prefab> \
  | grep -oE "[a-f0-9]{32}" | sort -u
养成手册的 prefab = x3-project\client\Assets\Res\UI\Prefab\Activity\UIActvLogin.prefab
STEP 2
GUID → PNG 反查（扫 .meta）
# 扫 Assets/ 下每个 .meta 文件头部 guid:，命中即对应 PNG（路径去掉 .meta 后缀）
import os,re
for root,_,fs in os.walk(r"...\Assets"):
    for fn in fs:
        if fn.endswith(".meta"):
            head=open(os.path.join(root,fn),encoding="utf-8",errors="ignore").read(400)
            m=re.search(r"guid:\s*([a-f0-9]{32})",head)
            if m and m.group(1) in 目标GUID集: print(目标GUID, os.path.join(root,fn)[:-5])
STEP 3
拷进原型 _assets/，HTML 拼
真实 PNG 拷进原型同目录 _assets/（自包含、零绝对路径）；HTML 用 background:url(./_assets/x.png) center/100% 100% 拼，9-patch 框用 100% 100% 拉伸。

找组件的线索（省时间）

  • sprite 名常带活动语义：养成手册 = *_nourish*（nourish=养成）。一套活动美术通常同前缀邻近放：Activity/ 下 nourish / _1 / _2 / _3 = 主视觉 / 锁格 / 解锁格 / 大面板。
  • 原界面没有的新控件（如本案的"切换栏/页签"）：复用同风格真实件（金按钮 img_cm_anniu1_gold）拼，或去带该控件的 prefab（BP/Tab 类）里抠。
  • 养成手册实抠出的真实组件（13 件）已存：KB\产出-交互原型\X3_英雄养成手册\_assets\

三、可用 UI 图怎么来：真组件拼装 → AI reskin 五步法

图#1 = 真组件拼装
位置全对，但丑/有接缝（没关系！）
→
reskin 成品
AI 换皮后，游戏级观感（"就是这个味道"）
★ 最值钱的血泪教训（护航令 BP 案实证）： 拼装（图#1）做到第 2 步就当成品交 → 被连判几版"烂完了/痛苦"。根因 = 漏了第 4 步 AI reskin。
拼装看着糙（手拼/CSS 兜底木板/道具重复/接缝）完全没关系——它的视觉在第 4 步会被 AI 全部 ERASE，只取布局。 把结构锚当成品交 = 必烂。别在第 2 步死磕拼装精度，赶紧进第 4 步。
STEP 1
拆真组件
从目标 prefab 抠真实 sprite（=上面第二节）。
STEP 2
CSS 拼装定精确布局（= 图#1 结构稿）
HTML 用真组件按真实坐标拼出布局，新控件也摆到位。可顺带加交互（购买/二选一互斥/领取）做活原型供"定交互"。chrome headless 截图 = 图#1。布局对就行，别追求好看。
STEP 3
（选）出适配新控件的底图
原界面没地方放新控件时，以老底图 img2img 加槽位。活 DOM 叠"烤死的槽"会对不齐 → PIL 竖扫测槽真实像素坐标再贴（详见 GUI 知识 §3）。
STEP 4 ★关键
AI reskin 合成成品（Morphix buildReskinPrompt 职责切分）
喂什么作用
图#1 = 拼装截图只取 布局/文字/控件位置/图标语义，视觉全 ERASE
图#2 = 真实游戏界面截图design system 整套抄：背景/面板/边框/按钮/格子/配色/字体
  • 模型 gemini（保布局）；要更强风格保真可加跑 gpt 一版对比。aspect 9:16。
  • 新元素显式声明：图#1 有而图#2 没有的（切换栏/封顶宽格），prompt 明确"按图#2 视觉语言原生渲染"（如分段控件嵌进羊皮纸 header / 封顶 = 图#2 奖励格的加宽金紫发光版）。
  • X3 调优（必加）：禁 painterly/笔触；金饰列举多种让模型按 ref 灵活挑（别单点名）；中文文案逐字保留；参考图能说明的别写进 prompt。
  • 图#2 去哪找：C:\Users\linkang\Pictures\X3验收\ 验收截图库（真实活动界面）。挑与目标界面框架最接近的一张。
STEP 5
S6 拆槽交付
成品认可后 → 四层拆解（F固定/T主题/B背景/TXT文本）+ 槽位规格表，给美术/程序。已有真组件 PNG 的，拆槽 = 组织真组件清单 + 标"需新出"项，不必再 AI 切。

四、变体：参考图是竞品 / 方向不对 → 先转方向再 reskin

当图#1 来源是竞品截图且方向不对（养成手册二选一弹窗：竞品是横屏 BP，我们要竖屏弹窗）——别直接 reskin（硬塞会变形），两段式：

两段式产出：竞品(横)→竖屏转换→X3 reskin 成品（弹窗 alt1）
第一段
横竖转换（Morphix buildFlipPrompt）
reference=竞品截图，gemini，aspect=目标方向。只翻布局不换风格：关闭键/货币/标题按目标方向重排、网格列数 reflow（横 4-6 列→竖 2-3 列）、轴向翻转、保两选项并列、文字逐字留。产出竖屏结构稿。
第二段
X3 reskin
把竖屏结构稿当图#1、真实 X3 界面当图#2，走第三节 STEP 4。可在 prompt 里顺带：竞品档位名→我方版本名、删价格数字、加二选一描述行。
坑：竞品奖励图标会残留 → prompt 强调用图#2 格子风格，真道具配置时再换；竞品业务文案（"达到45级返还"）要改成我方口径。
