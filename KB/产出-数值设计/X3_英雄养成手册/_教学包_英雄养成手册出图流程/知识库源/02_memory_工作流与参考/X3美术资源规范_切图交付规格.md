X3 节日活动**需产出新美术资源**时的规范（纯复用不出资源的只标"复用XX现成✅"，不进此规范）。每个资源类型定 3 个维度。配合 [[reference_x3_client_resources]]（DK→GUID 注册）、[[workflow_x3_festival_design_doc]]（策划案模板）。

## ★出图后默认核验（用户定2026-06-22·所有换皮出图后必做，不等提醒）
拿**对应参考图(复用源)**核一遍，4 项都要对齐，不只是尺寸：
1. **尺寸** W×H（=参考；gpt默认出1024~2048大图，必须缩到目标）
2. **格式/透明** mode（图标/tile=RGBA真透明；背景=RGB不透明）。⚠️**gpt"透明底"常是假透明**(RGB画浅灰/棋盘底,无alpha)→走 **GRFal remove_background 抠图**(单次,非双底差分)扣真透明 [[reference_x3_hero_skin_video_production]];验真=`Image.getchannel('A').getextrema()[0]<255`。
3. **上下左右边距**（=参考；决定 sprite 在 prefab 格子/槽位的对齐）
4. **主体占比**（内容 bbox 宽/高/面积 % = 参考；**不能简单居中**——如iso岛底座要落格子同位、高物件向上溢出）。
核验脚本：PIL 读 `getbbox()` 算内容 bbox + 四边 margin + 占比，逐项对参考。差太多→trim 后按参考 bbox 位置缩放重贴，不是直接 resize。

## 维度 A · 出美需颗粒度（策划写美需必含要素）
| 资源类型 | 美需必含 |
|---|---|
| 活动背景图 | 场景定位 + 主体焦点 + **元素清单≥3** + 构图层次(前/中/远景) + 氛围配色 + 避坑 + 参考图 |
| 礼包banner/头图 | 主视觉 + 文案/价格位预留 + 风格 + 参考 |
| HUD入口icon | 主体识别物 + 小尺寸可辨 + 风格统一 + 参考 |
| 道具图标 | 形态/主体 + 材质质感 + 配色 + 角标/底座(如需) + 参考 |
| 外显(铭牌/头像框) | 主体设计 + **品质感配色**(传说/史诗) + 边框/装饰 + (3D则形态/动态) + 参考 |
| 玩法UI(转盘等) | **逐部件**(边框/中心/指针/格子分区) + 状态(选中/默认) + 参考 |
| 角色立绘/spine | 角色设定 + 姿态/动作 + 服饰配饰 + 表情气质 + 风格 + (spine:动作列表) + 参考 |
| 特效 | 元素 + 动态过程 + 触发时机 + 节点/路径 + **优先级** + 参考 |
| 视频 | 时长 + 脚本/分镜 + 风格 + 参考 |

通用范式(参 x2-art-requirement/p2-festival-art-brief)：特征陈列≥2点 / 底座支撑说明 / 动态描述(特效·LV4-6必填) / ≥1方向备选标"倾向" / 避坑清单。

## 维度 B · 落地交付
- 格式：png(UI/图标/背景) / spine(立绘/角色动画) / fbx+prefab(3D家具/模型) / mp4(视频)
- 命名：DK_前缀(配置引用名,DK→GUID注册) ；资源文件 img_/DK_icon_ 等
- 路径：活动背景/banner=`Assets/Res/UI/Spirits/ActivityImg`(及 ActivityImg_Download 下载版) ；转盘/通用活动图=`.../Activity` ；家具=`Assets/Res/Furniture/...`
- DK→GUID 注册见 [[reference_x3_client_resources]]

## 维度 C · 格式现状（★核心：尺寸不统一，跟复用源对齐，逐个查）
**X3 同一需求跨模块/节日尺寸完全不一致**（实测，2026-06）：
- **活动背景**：无统一标准——尼罗大转盘 `queen_bg_3`=1080×1344 / 情人节VD=540×960 / 愚人节AFD=800×800 / 烧烤bbq=540×500 / 春节·圣诞·亡灵 顶部条=1080×140 / 其他1080×944…
- **HUD 活动 icon**：124×136（统一✓）
- **头衔/铭牌(title)**：主流 752×192，有变体 792×76
- **头像框(AvatarFrame)**：256×256（GVG/玩家框）
- **转盘格子 icon**：不统一 112×116 / 92×92
- **转盘元素 bg**：小元素 112×188 等

**规则**：
1. **复用换皮型** → 新美术尺寸 = **与复用源活动的对应资源同尺寸**（美术直接取原资源尺寸替换，最准；不要假设统一值）。
2. **新出资源** → 查**目标 UI 槽位**或同类现有资源尺寸（铭牌~752×192 / 头像框256×256 / HUD icon124×136）。
3. 出美需 C 维度必写："尺寸=与复用源XX同尺寸" 或 具体查到的px。

## 维度 D · ★换皮活动必须双参考（老图 + 新图，缺一不能出图）
换皮活动的「参考」只给复用源老图**不够**——老图只能参考**格式/结构/尺寸**，美术不知道换皮后的新元素长啥样，画不出来。每个美需项必须给两类参考：
1. **老图参考·格式结构** = 复用源活动的旧界面图（KB 包装稿截图 + 复用源活动客户端资源目录）→ 告诉美术版式/尺寸怎么对齐。
2. **新图参考·投放物元素** = 本期**实际投放的新东西**的资源图 → 告诉美术新主题/元素该画成啥样，使背景与所售内容一致。
   - **关键认知**：换皮活动投放的「新东西」（皮肤/门头/地板/家具/船皮肤等）**绝大多数是游戏里已存在的现成资源**——要去**配置表(FurnitureSkin/HeroSkin/Item)+客户端**里**搜出来**，不是 GRFal 生成。搜不到（如某主题的铭牌/头像框深海版无现成）才标「🔧需新出」。
   - **背景类无独立投放物**（许愿池/酒馆等只换背景）→ 新图参考挂**节日主视觉锚**（主推皮肤），保证全节日视觉一致。
   - 投放物→资源路径解析：配置表字段是 DK 名（如 `DK_Role_Spine_17_Skin01`/`DK_FurnitureFloor_Actv_Summer`），用 `find` 在 client 里搜实际文件；HeroSkin 立绘=`Spine/Prefabs_Download/Role_Spine_<heroid>_Skin<n>.prefab` + 头图 `UI/Spirits/Role/Character Portraits/Img_C_H_<heroid>_Skin<n>.png`；家具 icon=`UI/Spirits/Furniture/Actv/icon_jiaju_*.png`。
   - **写完务必逐路径验存在**（KB 包装稿 + 客户端投放物都要 `[ -e ]` 验过）。

## 落地用法
出美需时每个资源项 = A(颗粒度) + B(格式/命名/路径) + C(尺寸=跟复用源对齐或查到px) + **D(老图参考+新图参考双栏)**。深海节实例见 GSheet「深海节-活动美术需求」(9列:#/模块/类型/要求/尺寸格式/老图参考/新图参考/状态/完成)。
