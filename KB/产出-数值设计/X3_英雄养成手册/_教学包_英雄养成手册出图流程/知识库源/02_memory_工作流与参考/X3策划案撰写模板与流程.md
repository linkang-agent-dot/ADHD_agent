X3 活动/节日策划案的**权威模板**。触发"写X3策划案/活动方案/节日策划案/玩法案"先读本条 + [[feedback_gsheet_design_doc_format]]。

**Why:** 早期凭临场发挥写，返工严重（深海节周卡改4版/文案被指粗糙/模块拆分反复）。后用 6 个好范本（推币机/挖孔/海妖/深海/航海之路/拜访礼包）提炼出能"概括大部分情况"的模板——**一切策划案 = 「活动模块」的组合**，骨架固定、内容可变。

## ★内容来源与分工（重要）
- **内容优先自己搜，搜不到再找用户**：数值/奖池/ContentID/配置/资源名等——agent **先搜 X3 配置表(tsv `C:\x3\gdconfig\tsv`)/知识库/数仓/代码** 自己搞出来；只有真拿不到的（界面截图、尚未确定的换包装方案/复用图标注）才找用户要。**不编造，搜不到先问，不瞎填**。
- **agent 的职责 = 知识库层 + 搜证 + 落地**：自己搜出可得内容；把用户给的(或搜出的)策划内容**沉淀进知识库**(按节日/活动归档)，后续被提到时**调用**；按模板**结构化落地到 sheet**。
- **复用换皮型活动**：交付重点是「包装层 + 复用源(ContentID) + 美术换皮清单 + 复用图片」，规则/配置/界面写"复用不变"即可，不深写（范本那种深度只针对**新玩法/需设计**的模块）。

## ★验收闭环（建标记 → 收工自动拦 → 验收 → 清标记）
策划案走通用验收员 `task-checker`(type=design-doc) + 收工拦门，主 Claude 负责标记生命周期：
1. **开写时建标记**：`~/.claude/.pending_verify/<任务名>.json`，内容 `{"task":"<任务名>","type":"design-doc","sheet_id":"<目标SheetID>","status":"pending"}`。
2. **写完准备收工**：派子 agent `task-checker`（传 type=design-doc + 产物位置 + 任务名）跑验收。
3. **按结果处理**：
   - 全过 → **删**该 marker → 收工。
   - 有 blocker → 把 marker `status` 改 `reviewed`，把 blocker 列给用户定『修 / 跳过』；修完重派；用户跳过则记 done_with_concerns 后删 marker。
   - 验收员读不到产物/跑不了 → 同样改 `reviewed` 并明确告诉用户「验收没跑成，需人工确认」，**不假装通过**。
4. **回流（收工前必做）**：本轮验收或评审暴露「清单没覆盖的新问题类」(如删除链残留、机制漏客户端表现) → **当场 codify 进 `C:\ADHD_agent\.claude\quality-gate\design-doc-checklist.md`**（加 block/warn 规则 + 记「清单进化日志」来源案例+日期），再收工。让 checker 越用越全、不靠人记性——这是把「被坑→改产物」升级成「被坑→改清单(下次自动拦)」的元迭代。
> 没派验收就想收工时，Stop hook 会拦住并提示。整套机制（脚本+清单+说明）在 `C:\ADHD_agent\.claude\quality-gate\`；验收员 agent 在全局 `~/.claude/agents/task-checker.md`。这步是为了根治"写完不检查就收工"。

## 〇、写前必做（动手前，别跳）
1. **读目标 sheet 底稿全文 + 拉页签列表**（用户常已铺格式/改内容/留图位；读到空先查页签列表，可能被重命名）。
2. **读同类范本**（下方范本库），按其颗粒度对齐——一读范本就到位。
3. **查准数据**（拥有率/付费/R级口径见 [[reference_x3_datain_asset_query]]；机制见 [[reference_x3_monetization_mechanics]]、[[reference_x3_voyage_art_chain]]）。

## ★落地前确认门（两步 gate，不要跳过直接写 sheet）
读完参考、查完数据后，**先跟用户确认两件事，确认完再批量落地写 sheet**：
1. **确认活动清单（checklist）**：列出本期所有活动（哪些上、哪些复用、哪些归别的 agent/排除），用户确认增减。
2. **确认各模块详细内容**：逐模块（或先骨架后细节）过一遍内容，用户拍板/改，再写。
> Why：直接把全部模块写进 sheet 易返工（深海节就因没先确认清单+内容反复重写）。先确认清单与内容，落地只是把确认稿誊进 sheet。

## 一、★活动模块（最小不变单元，所有案子都用这个）
每个活动/玩法都按此结构写。槽位标★固定/◇可选：
```
[活动名]
〇、包装层 ★            (换皮的"皮"，独立成层：活动名称[本期主题名] / 活动描述[界面主标题·副标·按钮文案，出成品不留"换XX主题"占位] / 主题视觉关键词；与设计目的[why]、规则说明[how]分开)
一、设计目的 ★          (1.x 核心目的=why；◇付费点 ◇玩家追求/目标人群 ◇最终大奖)
二、玩法说明/概述 ★      (一段话讲清核心循环；◇竞品示意+界面参考)
三、规则说明 ★          (按功能点逐条编号，统一缩进 1)/●/◆/◇ + "(配置)"标注)
   ·基础设定 ★槽        : 新增活动类型(=NN) / 功能解锁 / 活动开启(排期) / 功能入口
   ·各玩法机制 ★可变     : 本玩法的子机制逐条(深海之路如:格子/升级/各岛事件/付费道具/神秘宝藏)
   ·阶段奖励 ◇          / 红点提示 ★槽(入口/页签/奖励各红点条件逐条) / 道具回收◇(带邮件id) / 邀请函◇ / 大奖跑马灯◇(MarqueeRuntips id)
   ·#其他: 需支持活动通用功能(名称/描述/时间/背景/英雄spine/链式礼包/活动buff)
配置表(内嵌) ★         : 每张表 4行表头schema = 类型行(cs/int) + 引用行(Item/Reward/外表.字段) + 中文字段名 + 英文字段名；列出本活动相关全部表+全局变量(ConstConfig)
界面交互 ★            : 逐界面(UIxxx.prefab) → 逐元素编号 → 每元素[显示内容/状态机/点击反馈/读哪个配置/Text key/跳转]
                       ★有交互的子功能(商店/弹窗/排行/结算等)：先走《X3交互模块工作流》(`C:\ADHD_agent\KB\方法论\X3交互模块工作流.md` + 同目录范本)——**原型先行**：产「活原型+实时说明」一体HTML、点哪讲哪、确认交互(含状态机/动线/边界/待定)后，再把内容回填到这里的元素表。纯数值/无交互子功能不用，直接逐元素填。
                       ★新界面要出效果图：走《X3_AI出图工作流》§8 S1-S5(`C:\ADHD_agent\KB\方法论\X3_AI出图工作流_角色皮肤换装+活动UI换皮_世界杯案.md`)——S1参考图拆基因+ASCII对结构→S2通用风格结构稿→S3 Morphix职责切分换本游戏UI风格(图#1=结构稿ERASE/图#2=真实UI截图)→S4每轮只改一处→S5效果图回填本段+美术需求页。Morphix prompt库挂点表也在§8。
[后续迭代] ◇          : 二期优化点(各带全局变量)
[美术·特效需求] ◇      : 主界面设计+关键词/活动图标/道具(名字-需求-描述)/特效表(描述+参考图+优先级)
[自测BUG] ◇           : 问题描述+参考图
本地化 ★              : Text key 清单(key|中文|英文|说明)
```
**填写指引**：①规则说明按"功能点"逐条，不写流水账 ②每个UI元素都要落"读哪个配置/Text key"，不留"换主题"占位 ③配置表内嵌schema让程序直接对 ④红点逻辑每条单列。

## 二、顶层组合（决定是哪类案子 = 几个模块 + 要不要总览）
| 案子类型 | 顶层 + 模块 | 范本 |
|---|---|---|
| **单玩法案** | 1 个活动模块（规则说明深化） | 推币机/挖孔/拜访礼包/航海之路 |
| **节日包装/优化案** | **节日主题包装(节日名/开场白/视觉主色/题材方向+避坑)** + 节日级总览(0节日开启时间/付费深度思路/增量预估[模块\|现状ARPU\|预测\|增量\|依据]) + 活动排期总览(活动\|排期\|开发量[颜色]\|复用/新增) + 各模块解锁条件 + **N个活动模块(各带包装层)** | 深海/夏日 |
| **系统rollout案** | 活动排期(甘特) + 目录(跳转表) + 框架模块 + **N个子活动模块** | 海妖 |
> 从属功能并入主模块（如"跑X圈成就礼包"并进大富翁作 2.x，不独立）。
