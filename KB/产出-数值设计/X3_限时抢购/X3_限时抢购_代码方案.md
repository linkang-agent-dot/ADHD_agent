## 结论先行

1. **走哪种活动模型**：**新增 ActvType（主玩法 + 专属 proto + 服务端 handler + 配置表 + 客户端 UI），不走纯 customParam。** 依据：限时抢购是 X3 没有的**整套新玩法**（全服限量库存计数 / 单人限次 / 分享→助力→抽奖破冰链 / 三场时段 / 服务端裁决抽奖），有大量本玩法专属的服务端状态和协议——这与扭蛋机走新 ActvType 同理，跟世界杯竞猜"只换两个礼包 id"的 customParam 场景不是一类。**customParam 只作为可选叠加层**：把"每期 8 个礼包 id / 抽奖池 id / 三场时间偏移"这些高频运营参数用 customParam 传入，避免每期改配表——但**货架包/抽奖池仍走配置表并按节日 fork**（对齐 X2 与扭蛋机，也是"抽奖池必 fork"这条雷的落地要求）。见 §2。

2. **方案覆盖**：① 玩法规格 formalize（§1）② X3 活动模型选型 + 协议 + 配置表 schema 草案（§2、§3）③ 客户端 4 件套迁移改造点（§4）④ 复用扭蛋机 vs 限时抢购独有（§5）⑤ 已知雷→程序硬约束（§6）⑥ 待拍板/待确认清单（§7）。

3. **待拍板点：11 个**（真源 §五 的 4 个 + 程序侧识别的 7 个），其中**头号硬点 = "全服限量"是否真跨全服共享库存**（决定要不要动 CenterServer，成本天差地别）。见 §7。

## 一、玩法规格 formalize（程序可实现口径）

从真源 §1.2 提炼。所有数字（三场时间、时长、限量、限次）**均配置读取，不写死**（对齐 X2：schedule 在 `flash_sale_buy_opentime`/`flash_sale_buy_duration`，单位秒）。

### 1.1 时段模型（预告 + 抢购三场）
- **预告期**：活动开启 → 首场抢购开始之间。可看礼包、可分享，**无购买按钮**。主界面倒计时 = 倒计时到"首场开始"。
- **抢购期**：每日 3 场，相对活动开启的偏移 = `[7200, 36000, 64800]` 秒（=UTC 2:00 / 10:00 / 18:00，兼容时区）；每场持续时长 = `[3600, 3600, 3600]` 秒（默认各 60 分钟，可配）。抢购期倒计时 = 倒计时到"本场结束"。
- **"期"的定义**：三场算**一期**（单人限次以"期"为单位，见下）。X3 需明确"期"边界如何切（按天？按活动生命周期？）——**待确认（§7-P8）**。

### 1.2 抢购约束（两把锁）
- **① 全服限量**：活动范围内所有服**共用一个抢购数量**（每个礼包一个全局余量计数器，抢完即无）。→ **跨服共享库存，X3 落地的头号架构点，见 §7-P1。**
- **② 单人限次**：每期每礼包限购 1 次。→ 服务端按 (玩家, 礼包id, 期) 维度记购买数。
- **8 礼包/期**：一期货架 8 个礼包，玩家可全买。

### 1.3 购买
- 支付方式：X2 用"游戏代币 / 礼包支付(IAP) / 唱片"。X3 货币映射**待拍板（§7-P4）**。
- 直购包（vm/代币）与 IAP 包并存（对齐 X2：vm 包内嵌 Cost/CostLimit 渲染正常，IAP 包价格/限购跨表查 iap_config/iap_template——这也是 X2 占位数据雷的根源，见 §6）。

### 1.4 社交破冰链（限时抢购独有，扭蛋机没有）
```
分享（首次分享 → 首次分享奖）
   → 被助力（他人助力，累计被助力次数）
      → 助力达标（3 档：第1/2/3次被助力，各发助力奖，含高级抽奖券）
         → 抽奖资格（被助力次数达标 且 买过至少1个礼包）
            → 抽奖（抽 8 礼包之一，抽中免费得；抽奖内限购不生效）
```
- **破冰逻辑**：链条设计成"分享→助力达标→差一次购买就能抽"，用抽奖诱导首次付费。
- **抽奖 ≠ 扭蛋机的付费抽**：这里抽奖是"达标后免费抽一次"，服务端从抽奖池裁决奖励并下发（客户端不读奖池）。抽中的礼包**免费到手，且不占全服限量/单人限次**。

### 1.5 主界面
- 标题栏：标题 / 倒计时 / FAQ / 剩余抽奖次数 / 退出。
- 抢购倒计时：预告期→到首场开始；抢购期→到本场结束。
- 开启时间点介绍：时间和时长均配置读取。

## 二、X3 活动模型选型（新 ActvType vs customParam）

### 2.1 判断依据（为什么是新 ActvType）

| 维度 | 世界杯竞猜（customParam 就够） | 限时抢购（必须新 ActvType） |
|---|---|---|
| 服务端新增状态 | 无（就 2 个礼包 id，复用 gift 原语） | **多**：全服限量余量、单人限次计数、被助力次数、抽奖资格/次数、分享标记 |
| 新玩法逻辑 | 无（买包=现成 Pack.BuyCount 限购） | **有**：全服库存扣减、抽奖裁决、破冰链状态机 |
| 专属协议 | 1 个 data message（复用现成请求） | **需新增 buy/raffle/share 多组请求 + 专属 data** |
| 客户端 | 复用现成 UI | **4 件套新 UI（横→竖 + DK + 货币显示）** |
| 可变的只是"哪几个包" | ✅ 是 | ❌ 不是，玩法本体就是新的 |

→ 限时抢购 = 扭蛋机同一类（整套新玩法搬运）→ **新 ActvType**。这跟扭蛋机决策一致（扭蛋机选 82）。

### 2.2 customParam 作为叠加层（可选，推荐）
把**每期变的运营参数**用 iGame customParam(JSON) 传入，服务端在专属 Condition 层反序列化 + 校验后建活动数据：
```json
{
  "packIds": [pkg1..pkg8],        // 本期 8 个货架包 cfg id
  "rafflePoolId": 21870004,       // 本期抽奖池 id（已 fork）
  "scheduleOffsets": [7200,36000,64800],
  "durations": [3600,3600,3600]
}
```
- **好处**：换节日/换期次 = 改 iGame 后台重部署，货架包组合/抽奖池指向/时段免改配表（对齐 customParam 九决策 D1/D2/D6/D7）。
- **边界**：货架包内容、抽奖池奖励项、机制道具**仍在配置表定义并按节日 fork**（customParam 只传 id 引用，不传内容）——这是"抽奖池必 fork"雷的硬要求（§6-雷1）。
- **若程序评估叠加 customParam 成本 > 收益**（8 包 + 破冰链参数比"2 个 packId"复杂得多），可退化为**纯配置表 + 每期 fork**（对齐扭蛋机 ActvCircusGacha 的做法）。**这是 §7-P9 的待拍板点**。

### 2.3 X3 框架五件套（新 ActvType 注册清单，照扭蛋机套）
1. `client\Assets\Scripts\CSShared\Common\Const\ActivityConst.cs`：加 `TRIGGER_TYPE_FLASH_SALE`（编号见 §7-P2）+ 类型集合登记。（ActvType 权威枚举就在此文件，别信 server handler 注释——见 memory `reference_x3_actvtype_enum`）
2. `client\Assets\Scripts\CSSharedHotfix\Game\Activity\ActivityTriggerConditions\ActivityFlashSaleCondition.cs`【新】：`[ActivityTriggerCondition(N)]` 特性自动注册；`_SERVERLOGIC_` 段 OnAddActivity（读 customParam→反序列化→校验→建数据）+ OnRemoveActivity；`_CLIENTLOGIC_` 段 `GetActivityUIType()` 绑 UI 类。
3. 服务端 `ActivityMeta.cs` OnRemove switch + `ActivityMeta.Msg.cs` 路由 switch 各 +1 case。
4. 导表 `gdconfig\Tools\table_exporter\def\actvonline_def.py`：常量 + 类型集合（`SINGLE_SERVER_ACTIVITY_TYPES` / `GIFT_TRIGGER_ACTIVITY_TYPES` 视需要）+ 新表 def。
5. proto：`client\Assets\TFWConfig\Protobuf\activity.proto` 加 message + `ActivityData` 追加新 tag（**先查当前最大 tag**，见 §7-P2），生成 C# 进 `Protos\`。

## 三、协议 + 配置表 schema 草案

### 3.1 协议（proto，cspb/命名空间随 X3 现状）

> X3 走类型名路由、无手写消息号（对齐 X2 A.8 与扭蛋机 proto 生成惯例）。所有 tag **先 grep 当前最大值再追加**（customParam 模板 D4/陷阱1：竞猜踩过"以为 36 空闲实占 62"）。

**主数据 message `ActivityFlashSaleData`**（下发客户端渲染，挂 `ActivityData.flashSaleData = <新tag>`）：
| 字段 | 类型 | 含义 |
|---|---|---|
| packages | repeated FlashSalePackInfo | 本期 8 包状态 |
| curWindowIndex | int | 当前第几场（-1=预告期） |
| windowStartTime / windowEndTime | long | 本场起止（客户端倒计时） |
| scheduleOffsets / durations | repeated int | 三场偏移/时长（客户端"开启时间介绍"） |
| shareDone | bool | 首次分享奖是否已领 |
| assistedCount | int | 被助力次数（破冰链进度） |
| assistRewardState | repeated int | 3 档助力奖领取状态 |
| raffleTicketCount | int | 剩余抽奖次数（=标题栏"剩余抽奖次数"） |
| raffleUsedCount | int | 已抽次数 |

**FlashSalePackInfo**（货架格）：`packCfgId:int, globalRemain:int(全服余量), personalBought:int(本人本期已购), price/costDisplay(引用Pack), soldOut:bool`。

**请求/应答**（每个走类型名注册路由）：
| 消息 | 字段 | 语义 |
|---|---|---|
| `FlashSaleBuyReq` | actvId, packCfgId | 买某货架包 |
| `FlashSaleBuyAck` | errCode, actvId, packCfgId, reward[], data(刷新后主数据) | — |
| `FlashSaleRaffleReq` | actvId | 达标后抽奖（=X2 的开箱/PrizeDraw） |
| `FlashSaleRaffleAck` | errCode, actvId, hitPackCfgId, reward[], data | 服务端裁决，回中奖礼包 id + 奖励（对齐 X2 `PrizeDrawAck{PackageId, rewards}`） |
| `FlashSaleShareReq` / `Ack` | actvId | 首次分享奖 |
| `FlashSaleAssistReq` / `Ack` | actvId, targetUserId | 助力他人（若走活动自有助力；也可能复用 X3 现成社交/助力系统，见 §7-P6） |

### 3.2 配置表结构

X3 活动落地遵循「`ActvOnline` 主表挂活动实例 + 各 ActvType 有专属内容表；ActvOnline ID = 100000 + ContentID」（memory `reference_x3_config` §活动 ID 分配规律）。

**(a) ActvOnline 主行**（`ActvOnline__ActvOnline.tsv`）
- 新分配 **ContentID**（→ AO_ID = 100000+CID 自动派生），`ActvType` = 新号（§7-P2），`TimeController` 定档后配（当前填 0），`MailID` 必填（memory `feedback_x3_actv_mailid_check`）。单服 vs 跨服由 `CrossServerRank` 决定——**与全服限量强相关，见 §7-P1**。

**(b) 主内容表 `ActvFlashSale`（新表）**——照扭蛋机 `ActvCircusGacha` 的建表流程（新 xlsx→def→proto），字段草案：
| 列 | 含义 |
|---|---|
| Id | = ContentID |
| ScheduleOffsets | 三场偏移 `[7200,36000,64800]` 秒 |
| Durations | 三场时长 `[3600,3600,3600]` 秒 |
| PackList | 本期 8 货架包 cfg id（`|` 分隔或 JSON）——**若走 customParam 则此列可空，由后台传** |
| RafflePoolId | 抽奖池 id（指向 (c)，**必 fork**） |
| GlobalLimits | 每包全服限量（数组，与 PackList 对齐） |
| PersonalLimit | 单人每期每包限购（=1） |
| ShareReward | 首次分享奖（RwdWithSetting/掉落包 id） |
| AssistThresholds | 3 档助力门槛（被助力次数）|
| AssistRewards | 3 档助力奖（含高级抽奖券） |
| RaffleTicketItemId | 抽奖机制道具 id（=累计道具，§7-P3） |
| SkinBoxItemId | 皮肤随机宝箱道具 id（§7-P3） |
| DefaultDisplaykey / IconDisplaykey / BannerUrl | DK/美术（§4） |

**(c) 抽奖池表 `ActvFlashSaleReward`（新表，或复用 X3 Drop/Reward 机制）**——对齐 X2 "2187 抽奖池"，**每节日 fork**：
- 每行 = 一个抽奖池 id + 权重奖励项列表：`{Id, Drop:[{typ,id,num,wgt,serial_number,is_highlight}...]}`
- 抽奖池 6 项（真源 §1.4）：占星皮肤 0.02% 保底钩 / 万能英雄碎片 15.62% / 高级抽奖券 29.68% / 成长线自选宝箱 23.43% / 2h 加速 31.25%（X3 需换对应道具，大奖外显 §7-P5）。
- **落地可复用 X3 现成 `Reward` 表掉落机制**（memory `reference_x3_config`：Pack.Content ↔ Reward.col1 同值掉落包；clone 必同步 clone Reward）。程序定：抽奖池是新建独立表，还是复用 Reward 表的权重掉落——**§7-P10**。

**(d) 货架包**：复用 X3 `Pack` / `Reward` 表机制（8 个 Pack，各自 Reward 掉落包）。IAP 包另配 `iap` 相关表（价格/限购）。**clone 铁律**：复制 Pack 行必同步 clone Reward 掉落包（否则礼包内容空）。

**(e) 机制道具（累计道具）**：新建 2 个 Item（对齐 X2 的抽奖道具 11119549 / 皮肤随机宝箱 11119550 class=flash_sale_reward_box）：
- **抽奖券/累计货币**：HUD 累计货币，助力奖/礼包产出，攒够抽奖。
- **皮肤随机宝箱**：独立 Item，`category_param.drop` 定开出内容，走 UseItem 派发（X2 是 class=flash_sale_reward_box）。X3 建 Item 照 memory `reference_x3_config`「新建可发放道具 + DK 机制」（克隆现成 item、清 col10 ItemObtain、配图标 DK 走 Path_Item）。
- **⚠️ 发累计道具的 IAP 礼包必须一起换机制道具 id**（§6-雷2）。

**(f) 任务/助力解锁**（对齐 X2 flash_sale_gacha 的 `arg1` 助力 task，Displaykey==2 门）：X3 用 ActvTask 或活动自有破冰状态机——**§7-P6**。

## 五、复用扭蛋机 vs 限时抢购独有

### 5.1 可直接套扭蛋机已落地的结构
- **新 ActvType 注册五件套**（§2.3）——一字不差同套流程。
- **proto 追加纪律**（先查 max tag、顺序追加、绝不改旧 tag）。
- **服务端 handler 骨架**：校验→扣费/扣库存→发奖→触发/更新→BI 打点，克隆 `ActivityMeta.XXX.cs` partial 模式。
- **GiftMeta 底层原语**：发货架包复用 `CreateActivityPack(giftId, packId, endTime)`（竞猜/扭蛋机都复用它），数据挂专属结构。
- **配置表按节日 fork**：奖池/货架包/机制道具每节日新建，不原位改旧节日（memory `reference_x3_config`：复用源=拷贝模板分配新 ID）。
- **落地纪律**：dev 受保护、sparse worktree 只铺 server/、ExportTable exit0 硬门、jolt_verify 验证构建（memory `workflow_x3_auto_jolt_export`）。
- **GM 命令本地复现 deploy**（customParam D9）——若走 customParam 叠加层，必配 `GMDeployFlashSale`。

### 5.2 限时抢购独有（扭蛋机没有，需新写）
| 独有点 | 说明 | 落地难度 |
|---|---|---|
| **三场时段调度** | 每日 3 场 utc2/10/18，预告期无购买按钮，倒计时两态 | 中（时间窗判定 + 客户端倒计时两态） |
| **全服限量库存** | 所有服共用一个余量计数器 | **高（可能要 CenterServer，§7-P1）** |
| **社交破冰链** | 分享→被助力→助力达标→抽奖资格；抽奖=达标免费抽（不是扭蛋机的付费抽） | 高（破冰状态机 + 助力系统接入） |
| **抽奖免费 + 不占限量** | 抽中的礼包免费到手、限购/限量在抽奖中不生效 | 中（发奖分支绕过限购/库存扣减） |

→ 扭蛋机是"加权双池付费抽 + 排行 + 积分任务"；限时抢购是"限量货架 + 破冰抽奖"，**抽奖核语义不同**（扭蛋机付费抽 vs 限时抢购达标免费抽），不能直接抄扭蛋机抽奖核，但可复用其"服务端权重随机 + 服务端裁决下发"的骨架。

## 六、已知雷 → 程序实现硬约束（真源 §四，内联为约束）

> 来源 memory `feedback_x2_flashsale_placeholder_data`（X2-43094 血泪）。搬 X3 时**必须在实现里落成硬约束**，否则重演"抽不了奖/开不了箱/占位数据"。

- **【硬约束 1】抽奖池必 fork。** X2 抽奖池是藏在 `flash_sale_raffle.status` 里的**间接引用**（不在活动 components 表面），全表只有 3 行且各节日共用、drop 写死指旧节日货架包。X3 落地：**每节日 fork 独立抽奖池 id**（§3.2-c），customParam/主表 `RafflePoolId` 指向本期 fork 的池；**绝不指旧节日的池**（旧节日下线后取不到 → "开不了箱"）。程序在 OnAddActivity 校验 RafflePoolId 存在且属本期。

- **【硬约束 2】累计道具连 IAP 一起换。** 只改活动组件里的机制道具没用——**发累计道具（抽奖券/皮肤宝箱）的 IAP/货架礼包的产出内容也必须换成本节日的机制道具 id**，否则产出端发旧道具、玩家身上没新道具 → "抽不了奖"。X3 落地：配置校验时端到端对齐——货架包/助力奖的 Reward 掉落包里发的抽奖券 id == 主表 `RaffleTicketItemId`。

- **【硬约束 3】push ≠ 生效，要重开活动。** 配置改完必须**构建 + 部署到测试服 + 重开活动**才会重读抽奖池/库存配置。占位数据（限购 2222 / 价 555）多是"服跑旧配置"。X3 落地：测试与上线 SOP 明确"改配置后必 jolt_verify 导表成功 + 部署 + iGame 重开活动"；开箱"卡住没奖窗"先查"服务端奖池产出的包 ⊆ 客户端货架集"+"是否重开"，别先动客户端。

- **【硬约束 4】换皮残留清零。** iap 描述残留旧节日名 / recharge_actv 指旧底座 / raffle 复用底座奖池 → 全部按本节日 fork/改名。
