01

定位：这个节日在职业坐标里的位置

来自 2026 H1 绩效自评的定位结论：交付广度已稳定（一人并行三项目全链路、AI 用量部门第 1 / 公司第 8、20+ 自建 skill），差距集中在三个更高阶维度。拓荒节搬运不是又一次换皮，而是三个差距的正面补课场。

H1 · 已经站稳的

做得出 · 做得全

  • 一人三项目：X3/X2/P2 节日全链路无延期开天窗
  • AI 杠杆：单人产能放大到团队量级，重复劳动全 skill 化
  • 知识库：踩坑资产化，换皮多为 copy-and-swap
➜
H2 · 要补的三条线

做得深 · 做得稳 · 做得可接管

  • 深：depth/ARPPU 没拉起来——breadth 起了、高价档没跟上
  • 稳：上线前质量门不牢，H1 有 3 起补发级事故
  • 可接管：流程标准化偏事后补救，规范多在事故后才补
本次新增 · 第四条线
从「一人团队」到「产能 ×2」。组员转全栈是这套 AI 工作流第一次复制给第二个人——H1 的定位是「部门里把 AI 用得最深的人」，H2 的自然升级是「把这套方法论输出给别人的人」。带出一个能独立跑模块的组员，本身就是比多做一个活动更硬的业绩。
02

团队盘点：两人的卡点互为镜像

分工策略的出发点不是「谁擅长什么就做什么」，而是先看清：你的短板恰好是他的长板，他的短板恰好是你的长板。

林康数值策划 / 活动运营 → 全栈
长板

产品意图、数值 ROI、投放定位、数据复盘——「为什么做、做成什么样」的判断力

卡点

AI 写的代码能实现功能，但看不出合不合理、有没有隐藏坑——缺代码判断力

解法

不自己补课看代码——把代码审查权交给组员 + AI 审 + 端到端验证兜底（§05 三道闸）

组员X3 客户端程序 → 全栈
长板

代码判断力、客户端实现、工程直觉——「这么写行不行、坑在哪」一眼能看出

卡点

不懂搬运的目的——投放、ROI、定位这些「搬过来其实要调」的产品判断没概念

解法

不让他猜——每个模块开工前拿到一页「目的卡」，配 x3-numerical-design（专为 0 基础程序员写的 ROI 教程）自查

结论：最优策略不是各自补课（太慢），而是把审查权交叉——他当你的代码守门员，你当他的意图守门员。
03

分工三原则

按「策划活 / 程序活」切是最差解：两人永远待在舒适区转不了全栈，而且互相等、串行最慢。

壹

按模块切，端到端 owner

每个模块一个 owner，从方案→配置→代码→测试→上线全程负责。这才是真的转全栈；也消灭了「我等你方案、你等我代码」的串行。

贰

交叉把关，不等补课

林康的代码 MR 必经组员 review 才能合；组员的模块必先拿到目的卡再动工、上线前由林康验收数值面。各守对方的短板。

叁

先总表，后动手

「不知道从哪搬、搬什么」的病根是逐个模块摸着走。先花 1–2 天把 8 个模块一次性盘完（§04），总表即本节日唯一入口文档。

05

两个把关机制的落地细节

交叉把关不能停在口号，落成两个具体工件：代码三道闸（守林康的短板）和目的卡（守组员的短板）。

代码三道闸

林康 AI 写的每一段代码，缺一不可 —— 「能实现功能」≠「合理没坑」，差距全靠这三道拦

1

组员人审 MR（主力闸）：合入前必经组员 review，工程合理性、隐藏坑由他判

2

AI 独立审一遍：sub-agent code review。储蓄罐 MR!525 就是「玩法测过 + AI 审过」才合的，实测有效

3

测试环境端到端：走真实入口触发、验到道具真实入账——不是「编译过 + 界面弹出来」就算完。H1 三起事故的共因就是验证停在中途

目的卡 · 一页五行

组员每个模块开工前拿到，动手前对齐 5 分钟，替代「做完才发现方向不对」的返工

1

定位：这模块在节日里拉付费率还是拉深度

2

投放红线：ROI 上限；变现侧数值绝对不动、流量侧可动

3

搬过来要改什么：附 X2 回归结论作依据（≠ 照抄）

4

参考往期：对标 X3 哪个已跑过的模块

5

验收标准：上线后看什么数据算成功

第一张目的卡的现成素材：X2 拓荒 D10 复盘——双付费点拆成 7+7（前半 gacha / 后半大富翁）导致首周付费率轴丢失、深度未达预期。X3 版必须修正节奏。这是「搬 ≠ 照抄」的头号实例，写进卡里组员立刻懂为什么要调。
06

四条线在这个节日里怎么兑现

把 §01 的定位逐条映射到 §03–§05 的具体动作——这样月底复盘时，每条线都有可检查的落点，而不是空口号。

Ⅰ做得深

H1 差距：breadth 起了、depth/ARPPU 没跟上，增量堆在 $4.99–$9.99
  • X2 D10 结论喂进设计：修正 7+7 双付费点节奏，别把付费率轴拆丢
  • 变现侧模块（gacha/强消耗/抢购/基金）数值统一由林康盘，深度目标进方案
  • BP 基金直接套用两档 BP 锚点经验（至尊选择率 90% 的显性价值锚打法）

Ⅱ做得稳

H1 差距：3 起补发级事故，验证停在「配置写对了」没到「玩家拿到的对不对」
  • 三道闸前置为上线 gate，不过不收工——不是事后补救
  • H1 三类漏点进强制 checklist：中转型产出终点入账 / 换皮关联模块 / 边界场景
  • gacha·宝箱类 = 中转型产出重灾区，入账核对列为验收硬项

Ⅲ做得可接管

H1 差距：规范多是事故后才补的，知识偏流水账
  • 搬运总表 = 唯一入口文档，冷启动即可接管
  • 每模块沉淀「X3 载体对照 + 换皮清单 + 决策记录」三件套
  • X2→X3 换皮 skill 目前空白，这次的 SOP 就是它的地基——下个节日搬运变 copy-and-swap

Ⅳ产能 ×2（新增）

H1 定位：部门里把 AI 用得最深的人 → H2：把方法论复制给第二个人
  • 组员从「纯配置低风险模块」起步，阶梯式上难度（配置→改造→数值参与）
  • 目的卡 + x3-numerical-design skill = 可复用的带人工件，不靠口头传帮带
  • 验收标准：节日结束时组员能独立端到端跑一个模块
07

节奏：按月度周期倒排

月度节日固定节奏 = 第 2 周周三部署 / 周四灰度 D0 / 周五全服。同期深海节（7/3–7/12）运营与世界杯收口（至 7/19）仍在跑，全程碎片化并行，靠总表和 owner 制降低切换成本。

现在 · 7 月上旬

搬运总表 + 消 ❓（1–2 天集中盘）

8 模块四问盘完；限时抢购程序量评估（组员，半天）与「掉落主活动玩法确认」是第一批要消掉的 ❓。总表定稿即出美需与文案需求——美术出图和 18 语言翻译是最长外部线，当天就发出去。

7 月中

结对首模块：掉落活动

两人一起走完整链路（方案→配置→验证→checker），把 X2→X3 搬运 SOP 跑通并写成 checklist。别一上来就分头——两人各踩一遍同样的坑最亏。

7 月中下旬

拆开并行 + 数值盘

按 §04 的 owner 分配并行开发；林康集中出变现侧数值盘 + 逐模块发目的卡；组员按「七日任务 → gacha → 强消耗」阶梯上难度。每模块完成即过三道闸。

8 月初

提测 + 上线前 gate

提测→改 BUG（3 天）→验收（1 天）；换皮清单/i18n 扫描/端到端三件套全过，入账核对硬项打勾才算完。

8 月第 2 周

周三部署 → 周四灰度 D0 → 周五全服

上线即启动日报监控（x3-festival-monitor 已自动化）；D10/D14 回归按 breadth vs depth 两条线拆，depth 不达标模块进下期 must-fix。

08

会撞上的四件事（提前接住）

两人并行的仓库冲突同时改配置仓/代码仓会撞分支和 xlsx 锁。用 git worktree 隔离（坑已记录：worktree 刚建完别马上拆）；分支规范照旧——feature 分支 + MR，改前 git branch --show-current。
测试验收没人认领全栈分工最容易漏的就是「谁来验」。规则定死：owner 自测端到端 + 对方交叉验收自己擅长的那一面（他验技术面，你验数值和投放面）。
外部依赖启动太晚美术出图和翻译周期最长，总表盘完当天就发需求，别等模块做完才想起来。
SOP 沉淀被赶工挤掉这次每模块的换皮清单+决策记录不是额外负担，是下个节日的 copy-and-swap 地基和 X2→X3 skill 的原料——被挤掉就等于下次重付一遍学费。
